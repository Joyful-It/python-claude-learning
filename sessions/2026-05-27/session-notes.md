## 二、秒懂公式（背！）

```
所需显存 ≈ (模型参数量 B × 每参数字节数) × 1.2
                                    ↑
                       20% 开销 = KV Cache + CUDA 算子
```

### 三精度字节数

| 精度 | 字节/参数 | 7B 算例 | 场景 |
|------|----------|--------|------|
| **FP16**（原生） | **2 Bytes** | 7×2×1.2 = 16.8 GB | 原生推理，精度最高 |
| **INT8**（8位量化） | **1 Byte** | 7×1×1.2 = 8.4 GB | 平衡精度与显存 |
| **INT4**（4位量化） | **0.5 Bytes** | 7×0.5×1.2 = 4.2 GB | 极限压缩，家用卡首选 |

> INT4 4.2G 只是最低垫底，长文本 KV Cache 会暴涨 → 实际建议 6-8 GB

---

## 三、推理 vs 训练——显存差距巨大

### 推理只往前走

```
推理：输入 → 前向传播 → 输出
显存 ≈ 模型参数 × 精度
```

### 训练往回走，多存三样

```
训练：输入 → 前向传播 → 反向传播 → 更新参数
显存 = 模型 + 梯度 + 优化器状态 + 中间激活值
       ↑      ↑        ↑            ↑
     同推理  同模型大  AdamW状态    每层结果缓存
                     ≈模型×2
```

### 行业黑话

| | 推理 | 全量微调 | LoRA | QLoRA |
|------|------|------|------|------|
| 每1B显存 | ~0.6 GB (INT4) | **~16 GB** | ~2.5 GB | **~1 GB** |
| 7B 需要 | ~6 GB | **112 GB** ❌ | ~18 GB | **~8 GB** ✅ |
| 32B 需要 | ~24 GB | 512 GB ❌ | ~80 GB | **~32 GB** |

### 为什么全量训练这么贵？

```
全量微调 7B：7 × 16 = 112 GB → 单张 4090（24G）直接报废
LoRA 微调 7B：只需 ~15-20 GB  → 单张 4090 刚好能跑
QLoRA 微调 7B：只需 ~7-9 GB   → 连 3060（12G）都能跑
```

## 四、一句话结论

```
推理 → FP16/INT4 装模型就行，家用卡完全够
训练 → 必须 LoRA/QLoRA，个人开发者绝不可能全量微调大模型

口决：推理装得下就行，训练还要存梯度+优化器+中间结果，贵 4 倍起步
```

# 下午：LoRA 低秩微调深入

## 今日理解跃迁（下午新增）

> 从"LoRA就是冻结原参数只训小参数" → "LoRA本质是把 ΔW(1677万参数) 拆成 B×A (6.5万参数)，用两个低秩矩阵的乘积逼近全量更新，省了256倍"

---

## 一、什么算模型参数？

| 是参数 | 不是参数 |
|--------|---------|
| Linear 的 weight/bias | 训练数据 |
| CNN 卷积核权重 | batch size / epoch |
| Embedding 词向量表 | 学习率（超参数） |
| Attention Q/K/V/O | loss（损失值） |
| LayerNorm 参数 | 梯度（更新方向） |

> 7B 模型 = 70 亿参数，B = Billion（十亿）

---

## 二、全参数微调 vs LoRA

### 全参数微调

```
W' = W + ΔW      W 和 ΔW 都是 4096×4096（1677万参数一起训）

问题：显存大、慢、每个任务存一份完整模型、数据少时灾难性遗忘
```

### LoRA 微调

```
核心：冻结 W，ΔW 不直接训，而是拆成 B×A

  ΔW = B × A
  B: 4096 × r（竖条）
  A: r × 4096（横条）
  
  输出 = Wx + (α/r) × BAx
         ↑原模型      ↑LoRA分支
```

| | 全参数微调 | LoRA (r=8) |
|------|:--:|:--:|
| 可训练参数 | 16,777,216 | **65,536** |
| 倍数 | 1× | **省 256 倍** |

---

## 三、为什么叫"低秩"？

```
ΔW = B × A
  
  rank(ΔW) ≤ r（r 就是秩的上限）

  r=8 → ΔW 最多是秩 8 → 低秩
```

**A 和 B 的尺寸由什么决定？**
- 原 W 形状决定 B 的列数和 A 的行数
- r（秩）决定中间维度
- 具体数值由训练自动学习，不是人手指定

---

## 四、初始化和训练过程

```
初始化：A 随机，B 全零
  → 刚开始 BA = 0 → 模型 = 原模型，不会突然发疯

训练：
  数据 → W(冻结)算出基础输出
       → A,B 算出修正输出
       → 相加 → 算 loss → 反向传播
       → 只更新 A 和 B，W 不动
```

## 五、代码核心（三行）

```python
base_output = x @ W.T                          # 原模型，冻结
lora_output = x @ A.T @ B.T * (alpha / r)      # LoRA 分支，训练
output = base_output + lora_output             # 相加
```

## 六、LoRA 优缺点

| 优点 | 缺点 |
|------|------|
| 省显存（不用存原参数梯度） | 基础模型弱，LoRA 也救不了 |
| 训练快（只训少量参数） | 新任务差异太大可能不够 |
| 保存小（每个任务只存 A,B） | r 需要实验调 |
| 任务切换方便（换 LoRA 权重即可） | 复杂任务效果不如全参数微调 |
| 保留原模型能力（W 不动） | 加在哪些层需要调整 |

---

## 七、学员理解纠正

| 学员原话 | 判断 | 补充 |
|---------|------|------|
| "全参数调太费资源" | ✅ 对 | 7B 全量 ≈ 112GB 显存 |
| "冻结一些参数，只调节一些小参数" | ✅ 核心对 | 但"小参数"不是原参数里挑几个，而是**新增** A,B 两个小矩阵 |

> LoRA 不是"从 W 里挑几个参数训"，而是**在旁边挂两个新的小矩阵 A,B，只训它们**。

---

## 阶段一：破除名词迷雾——不是卷积

| 学员追问 | 澄清 |
|---------|------|
| "LoRA 跟卷积有关系吗？" | **没有。** 卷积=滑动窗口扫图，LoRA=大矩阵乘法 |
| 类比 | 全参数=把 1677 万像素重画，LoRA(r=8)=只给 8 种颜色润色 |

**学员第一步突破：**
> "意思是原本数据进来要乘 4096×4096 的矩阵，现在只需要用两个小矩阵，先进行降维再升维，再将他们相加是吗？"

→ 自己点出了 LoRA 的**瓶颈网络结构**

---

## 阶段二：核心卡点——"乘"与"训"的分离

| 学员深度追问 | "既然微调，为什么还要乘原矩阵？乘原矩阵不是变全调了吗？" |
|-------------|----------------------------------------------------------|

**类比：游戏 DLC**

```
原模型 W = 《黑神话：悟空》100GB 游戏本体（只读锁死）
LoRA A,B = 1GB DLC 扩展包（可写可改）

前向传播：必须加载 100GB 本体 + 1GB DLC → 游戏能跑
反向传播：只改 DLC 文件夹，本体设为 requires_grad=False
```

| | 前向传播（读） | 反向传播（改） |
|------|:--:|:--:|
| W (原模型) | ✅ 乘了，用它的知识 | ❌ 不算梯度，不更新 |
| A,B (LoRA) | ✅ 乘了 | ✅ 算梯度，更新 |

> **乘 W = 读课本，训 W = 改课本。LoRA 读了但不改。**

**学员第二步突破：** 彻底分清"前向乘（读知识）"与"反向改（调参数）"在显存上的天壤之别。

---

## 阶段三：主动推导完整流水线

**学员用自己的话串 LoRA 全流程：**

> "数据进来和每个神经元相乘提取特征……等到反向传播时，我们随机生成两个矩阵挂载到神经网络，与前向传播的结果降维升维，与主线路相加，但是反向传播只更改 A、B 矩阵其他不动。"

| 学员推导 | 纠正（唯一） |
|---------|------------|
| 整个流程和结论 | ✅ 90% 正确 |
| "反向传播时随机生成 A,B" | ⚠️ A,B 在**训练开始前**就挂好了，不是反向时才挂 |
| | A 随机初始化，B **全填 0**（保证 BA=0，DLC 开局不干扰本体） |

---

## 阶段四：参数"出身"的最终闭环

**学员推导：**

> "如果从头训模型，每一个 W 确实都是随机的；但在 LoRA 场景下，这个 W 已经不是随机的了，它是通用的（大厂烧钱训好的底座），所以它不动。而挂在旁边的 A 和 B 才是嗷嗷待哺、等着被反向传播修改的随机打工人。"

---

## 终极公式

```
y = x @ W₀  +  x @ A @ B
    └─只读本体─┘  └─动态DLC─┘

主路线（x@W₀）：取通用特征，锁死，不吞显存
辅路线（x@A@B）：降维→升维瓶颈结构，反向传播只盯它们
```

## 矩阵维度顿悟

```
x 的形状：(batch, d_in)         ← 输入数据的列数 = d_in
W 的形状：(d_in, d_out)         ← 行必须 = d_in（内维对齐！）
A 的形状：(r, d_in)             ← 另一面也要对齐 x
B 的形状：(d_out, r)            ← 输出维度和 W 一致

→ W 的行向量 是由 输入数据的列向量 决定的
   不是魔法数字，是矩阵乘法规则！
```

# 晚间：DeepSeek GRPO 深入

## 今日理解跃迁（晚间新增）

> 从"GRPO 就是省掉评论家模型" → "GRPO 本质是同模型生成 16 条回答，用规则打分 + 组内平均做基线，一条回答优势=0 什么也学不到"

---

## 一、核心误解纠正

| 误解 | 纠正 |
|------|------|
| 16 个模型互相对抗 | **同 1 个模型**，生成 16 条不同回答 |
| "淘汰"差的 | 不是淘汰，是**统计比较**：高于平均→强化，低于平均→弱化 |
| 没评分标准 | 用**规则打分**（数学题对答案、格式检查），不需要学习 |

---

## 二、GRPO 评分机制

### 两条规则奖励

| 奖励类型 | 怎么评 | 例子 |
|---------|--------|------|
| **准确性奖励** | 答案对不对 | 数学题：最终答案和标准答案比对 |
| **格式奖励** | 有没有按要求写 | 有没有用 `<think>` `</think>` 标签 |

### 统计比较（核心）

```
Group平均分 = (回答1分 + ... + 回答16分) / 16
优势(Advantage) = 自己的分 - Group平均分

优势 > 0 → 好于平均，加大这条路的概率
优势 < 0 → 差于平均，降低这条路的概率
```

**只生成 1 条？** → 组平均 = 自身 → 优势永远 = 0 → 什么也学不到

---

## 三、PPO vs GRPO

```
PPO:  Advantage = 实际奖励 - Critic预估分    ← Critic=另一个同等规模模型
GRPO: Advantage = 实际奖励 - 16条回答平均分   ← 不需要额外模型
```

| | PPO | GRPO |
|------|------|------|
| 架构 | Actor + Critic（两个模型） | 只有 Actor |
| 显存 | Actor×2（Critic 同等大） | 只需要 Actor |
| 奖励 | Critic 学习预估分 | 规则打分 + 组平均 |
| 省了什么 | - | **省人工标注 + 省 Critic 模型** |

---

## 四、非数学题怎么"思考"？

DeepSeek 两步走：

**第一步：数学/代码上练出"打草稿"习惯**
→ GRPO + 规则奖励（有标准答案）

**第二步：用少量人工示范迁移到通用领域**
→ 收集几千条高质量思考过程 → SFT → 模型学会"任何问题都先 `<think>`"
→ 再做一轮 RL，加上有用性/无害性奖励

# 学员自主总结：DeepSeek-R1 完整训练流程

> 以下为学员自主整理，原文收录。

## 一、起点：R1-Zero 实验（纯 RL 探索）

**目的**：验证核心假设——不用任何人工标注的思维链数据，仅靠强化学习，能否让模型自主学会推理？

**做法**：
- 直接拿 DeepSeek-V3-Base 作为基座模型
- 不进行任何监督微调（SFT），直接投入纯强化学习训练
- 使用 GRPO 算法

**GRPO 工作机制**：
1. 对同一个问题，旧模型生成一组答案（如 16 个）
2. 用规则奖励给每个答案打分：答案是否正确、推理过程是否在 `<think>` 标签里
3. 计算这组答案的平均分作为"基准线"
4. 每个答案的奖励减平均值 = 优势值（高者为正，低者为负）
5. 用优势值驱动反向传播，增加高优势答案概率，压制低优势答案

**成果**：模型自主涌现"啊哈时刻"——会自己打断推理，进行"等等，我可能搞错了，让我重新检查一下"的自我纠错

**问题**：推理能力强但可读性极差——中英混杂、逻辑跳跃、格式混乱，只会做题无法发布

---

## 二、正式产品 R1 的四阶段训练

R1 并非在 R1-Zero 上继续改，而是另起一条管线，同样从 V3-Base 出发，吸取 R1-Zero 教训。

### 第一阶段：冷启动微调（SFT）——让他"可读"

- 收集几千条高质量、人工撰写或修正的思维链范例
- 对 V3-Base 进行监督微调
- SFT 原理：给模型"问题+标准答案"，模型生成的每个字和标准答案逐字比对，差距越大损失越高
- **成果**：学会了"先思考再回答"的基本格式，输出规范可读

### 第二阶段：推理专项强化学习（RL）——让他"推理"

- 将第一阶段种子模型投入推理专项 RL
- 使用数学、编程等有标准答案的任务
- 继续 GRPO + 规则奖励
- **与 R1-Zero 区别**：起点不同——R1-Zero 从白纸 V3-Base 开始，R1 从已会规范输出的冷启动模型开始
- **成果**：推理能力大幅提升，自我反思再次涌现，但变成"偏科生"

### 第三阶段：拒绝采样微调（SFT）——让他"平衡"

- **拒绝采样** = 生成一批，挑出好的，扔掉差的
- 推理题：直接比对标准答案，错的扔掉
- 开放题（写作等）：用奖励模型打分，只保留高分回答
- 混合原始通用训练数据，做第二次 SFT
- **成果**：不再偏科，能推理也能正常交流

### 第四阶段：全场景强化学习（RL）——让他"精通"

- 奖励信号升级为混合奖励：
  - 规则奖励（推理题：答案正确性+格式规范性）
  - 奖励模型打分（开放任务：有用性/连贯性/安全性）
- **成果**：推理强 + 回答有用/安全/符合人类喜好 → 最终可发布的 DeepSeek-R1

## 四、四种训练方式小结

| 阶段 | 训练方式 | 数据/信号来源 | 核心目的 |
|------|------|------|------|
| 第一阶段 | SFT | 人工撰写的思维链范例 | 学规范、学格式、保证可读 |
| 第二阶段 | RL | 规则奖励（答案+格式） | 练推理、长核心能力 |
| 第三阶段 | SFT | 拒绝采样筛选的高质量数据 | 补通用、找能力平衡 |
| 第四阶段 | RL | 规则奖励 + 奖励模型 | 全精通、最终打磨抛光 |
