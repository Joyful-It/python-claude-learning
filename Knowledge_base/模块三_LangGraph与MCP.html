1.1 LangGraph 是什么

你们之前学过的 LangChain，是个agent框架，核心是流水线（Pipeline）——把提示词、工具、大模型串成一条线， 按固定顺序执行。就像品牌成品电脑，买来就能用，但不能随便改内部结构。

今天学的 LangGraph，是个agent框架，核心是图（Graph）+ 状态（State）。 图中的边（箭头）决定了数据走哪条路（分支、循环、并行），状态记录了数据走到哪了、变成了什么。 就像DIY组装电脑，不同型号主板、CPU、显卡让你自由搭配， 状态就像数据总线在组件间传递信息。

【注解】这里说的「图(Graph)」不是图片，是「节点+连线」的结构。 就像地铁线路图：每个站是一个「节点」，轨道是「边」。 数据就是乘客，从起点站上车，经过一个个站（节点）处理，最后到终点站下车。

下面这张图就是一个最简单的「有向图」——箭头表示方向，数据只能沿着箭头方向流动：

          (节点)        (节点)        (节点)
      ┌────────┐   ┌────────┐   ┌────────┐
      │ START  │──→│ judge  │──→│  END   │
      │ (入口) │   │ (判断)  │   │ (出口) │
      └────────┘   └────────┘   └────────┘
          ↑             ↑             ↑
        起点          处理站        终点

数据从START出发 → 进入judge处理 → 到END结束

这就是LangGraph的核心思想：用画流程图的方式写AI程序。

安装依赖：pip install langgraph

LangGraph的4个固定步骤

所有LangGraph程序都遵循同样的套路（记住它！以后每个demo基本都是这个结构）：

第①步：定义状态(State)  → 确定数据包长什么样
第②步：定义节点(Node)  → 写每个处理函数（每个函数做一件事）
第③步：构建图(Graph)   → 添加节点，通过边（箭头）把它们连起来
第④步：编译并运行      → 把设计图变成可执行的程序，启动它

下面用一个最简单的例子来演示这4步。

【注解】下面这段代码展示了 LangGraph 的完整工作流程：

  1. 导入工具：引入 TypedDict（定义数据类型）和 StateGraph（构建流程图）
  2. 定义数据格式：成绩信息包含 name（姓名）、score（分数）、grade（等级）
  3. 定义处理函数：judge 函数接收数据包，判断分数是否及格
  4. 构建流程图：创建一个流程，只有一个 judge 节点
  5. 编译运行：把设计图转成程序，输入学生数据，启动！

# ========== LangGraph 第一个程序 ==========

# 【注解】TypedDict：Python的类型注解工具。
# 用来规定字典里每个字段的名字和类型，写错了IDE会提醒你。
from typing import TypedDict

# 【注解】StateGraph：LangGraph的核心类，用来构建"状态图"（状态=数据，图=节点+连线）
# START：内置的"起点"标记，表示流程从哪里开始
from langgraph.graph import StateGraph, START

# ---------- 第①步：定义状态(State) ----------
# 规定数据包长什么样：有3个字段
class Chengji(TypedDict):
    name: str      # 学生姓名（str=字符串/文字）
    score: int     # 考试分数（int=整数）
    grade: str     # 成绩等级（初始传入时为空，由judge函数来填）

# ---------- 第②步：定义节点(Node) ----------
# 【注解】节点 = 处理函数。参数state接收当前数据包，返回要更新的数据
# -> dict 表示这个函数返回一个字典（键值对结构）
def judge(state: Chengji) -> dict:
    """判断成绩"""
    # state['score']：从数据包里取出score字段的值
    if state['score'] >= 60:
        # return 只返回修改的字段，没改的字段自动保留
        return {"grade": "及格"}
    else:
        return {"grade": "不及格"}

# ---------- 第③步：构建图(Graph) ----------
# 创建一张成绩处理流程图，括号里指定数据格式为"Chengji"
flow = StateGraph(Chengji)
# add_node：添加一个叫"judge"的节点，关联上面定义的judge函数
flow.add_node("judge", judge)
# add_edge：画箭头，从START起点指向judge节点
flow.add_edge(START, "judge")

# ---------- 第④步：编译并运行 ----------
# compile()：把"设计图"转成"可以执行的程序"
app = flow.compile()

# invoke：启动程序！传入初始数据{姓名:张三, 分数:80}
result = app.invoke({"name": "张三", "score": 80})
print(result)  # 输出：{'name': '张三', 'score': 80, 'grade': '及格'}
{'name': '张三', 'score': 80, 'grade': '及格'}

1.2 LangGraph 核心概念

刚才的 demo 背后就是这四个概念：

概念 代码里怎么写 一句话理解
State（状态） class State(TypedDict) 定义"数据包"长什么样，有哪些字段
Node（节点） def my_func(state): 一个函数，做一件事，更新数据包
Edge（边） add_edge(A, B) 固定路线：A做完直接去B
Conditional Edge（条件边） add_conditional_edges(A, router_fn) 岔路：根据当前数据判断下一步去哪

【注解】这4个概念的关系： - State 是"数据包"，在节点之间传递，每个节点都能读和改 - Node 是"处理站"，每个站做一件事，处理完后把结果写回数据包 - Edge 是"传送带"，从一个站到另一个站的固定路线 - Conditional Edge 是"岔路"，不是固定路线，而是根据当前数据选择走哪条路

LangGraph 的本质：用画流程图的方式写 AI 程序。定义函数（节点）、画箭头（边）、箭头决定State走向

核心公式：LangGraph = 节点 + 边 + 状态

以后会经常用的导入

from typing import Annotated, TypedDict
from langgraph.graph import MessagesState
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages

【注解】这些是 LangGraph 开发最常用的导入： - TypedDict：定义数据包格式 - Annotated + add_messages：让消息可以"追加"而不是"覆盖"（后面会讲） - MessagesState：LangGraph内置的标准数据格式，自带messages字段管理聊天记录 - StateGraph：创建流程图 - START / END：流程图的起点和终点

1.3 为什么学 LangGraph？

LangChain vs LangGraph

【注解】LangChain底层其实就是用LangGraph搭好的一个模板。 所以学LangGraph并不是抛弃LangChain，而是当模板不够用时，你自己有能力定制。

LangChain LangGraph
一句话 品牌电脑套装，买来直接用 DIY 组装电脑，按需自己配
能做循环吗？ ✅ 能（自动 ReAct 循环） ✅ 能
能插自定义步骤吗？ ⚠️ 有限，1.0+ 可通过 Middleware（@before_model 等）在模型调用前后插入逻辑，但受限于固定管线顺序 ✅ 完全自由，想加几步加几步，任意编排节点顺序
能让人中途插手吗？ ✅ 能（设暂停点） ✅ 能

核心区别就一点： langchain 是封装好的样品，你只能按它的固定管线来用（1.0+ 虽支持 Middleware，但仍是固定管线内插入）。LangGraph 让你彻底自由搭

- 以前大家花几年在混乱里学 Agent，现在你用 LangGraph，一次就把 Agent 学明白 - 等你真正掌握了 LangGraph，Agent 对你来说就不再是黑盒了。以后不管出现什么新的智能体框架，你都能一眼看懂它在干什么

1.4 StateGraph：亲手搭一个

流程概览

核心流程：

            ┌── 需要调用工具  →  tools → 回到 agent（再次判断）
START →  agent                     
            └── 无需调用工具  → END（返回最终回答）

关键概念： - MessagesState：LangGraph 内置的标准状态，自动管理消息列表 - ToolNode：内置节点，自动执行 LLM 发出的工具调用 - 条件边：根据 LLM 的输出决定下一步走哪条路

【第①步】定义状态：直接使用内置 MessagesState

这个例子中我们的"数据包"直接用 LangGraph 内置的 MessagesState， 它自带了 messages 字段来管理聊天记录，我们不需要自己定义状态类。

【第②步】定义节点：准备 Agent 组件

先准备好 Agent 需要的"工作站"：工具函数（@tool）、工具管理器（ToolNode）、绑定了工具的模型。 工具函数和call_model函数就是图中的"节点"——每个函数是一个工作站，做一件事。

from langchain_core.tools import tool
# 【注解】@tool 装饰器：把下面的函数标记为"一个工具"，LLM可以调用它
# @tool 是 Python 装饰器语法，本质就是把函数包装一下
from langgraph.prebuilt import ToolNode
# 【注解】ToolNode：LangGraph内置的"工具管理器"
# 它会自动检查LLM的回复，如果要调工具就自动执行
from langgraph.graph import MessagesState,StateGraph,START
# 【注解】MessagesState：LangGraph内置的数据格式，自带messages字段管理聊天记录
# 你不需要自己定义messages字段，直接用就行了

# ========== 定义节点（工作站） ==========

# ----- 节点函数1：search（天气查询工具）-----
# 【注解】@tool 让下面的函数成为一个"工具"，LLM可以调用它
# 这个工具函数会被 ToolNode 包装成一个叫"tools"的节点
@tool
def search(query: str) -> str:
    '''查询天气的工具函数'''
    if "sf" in query.lower() or "san francisco" in query.lower():
        return "温度是 60 华氏度，天气阴霾。"
    return "温度是 90 华氏度，天气晴朗。"

# 把工具打包成节点：search函数 → tools列表 → ToolNode节点
tools = [search]                       # 工具列表
model = llm.bind_tools(tools)          # 把工具绑定给LLM，告诉LLM可以调用它


tool_node = ToolNode(tools)            # 【节点】创建"tools"节点（自动执行工具）




# ----- 节点函数2：call_model（AI思考节点）-----
# 【注解】这个函数就是图中的"agent"节点
# 它的工作：读取聊天记录 → 发给LLM → LLM决定直接回答还是调工具
def call_model(state: MessagesState) -> dict:
    '''【节点函数】Agent 思考节点'''
    res = model.invoke(state["messages"])
    return {"messages": [res]}
    # 注意 res外部的中括号不能少，对应messages: Annotated[list, add_messages]中的list参数

【第②步 续】先定义条件路由函数（为第③步做准备）

条件路由函数决定了LLM回复后下一步去哪：要调工具就去tools节点，直接回答就结束。 这个函数是第③步add_conditional_edges要用到的判断逻辑，所以先在节点定义好之后再准备它。

def should_continue(state: MessagesState):
    '''【注解】条件路由函数 = "岔路口的指示牌"
    检查LLM最后一条回复：
    - 如果LLM说要调用工具 → 返回"tools"，去执行工具
    - 如果LLM直接回答了 → 返回"__end__"，结束流程'''
    last_message = state["messages"][-1]
    # 【注解】state["messages"][-1]：取messages列表的最后一条消息
    # [-1] 是Python语法，表示"列表的最后一个元素"
    # tool_calls：LLM消息的一个属性，如果LLM想调用工具，这个属性就有值
    if last_message.tool_calls:
        print(">> [条件边] 检测到工具调用 -> 路由到 tools 节点\n")
        return "tools"         # → 路由到tools节点执行工具
    else:
        print(">> [条件边] 无需工具调用 -> 路由到 END\n")
        return "__end__"       # → 路由到__end__结束

第2章：高级状态管理与条件分支实战

MessagesState 内部机制：messages 为什么必须是列表？

第1章里我们直接用 MessagesState 作为状态，但没有细讲它内部是怎么工作的。 现在我们要深入看穿它——因为第2章要自己定义状态了，理解 MessagesState 的机制才能用好它。

MessagesState 长什么样？

MessagesState 的底层定义，其实就是这样一个类（Class）——你可以理解为它就是个「提前做好的自定义状态模板」：

class MessagesState(TypedDict):
    messages: Annotated[list, add_messages]  # 注意：list 是列表，Annotated 是固定写法

它本质上就是 TypedDict，只不过 LangGraph 帮你写好了。所以你完全可以用自己的类替代它，它的核心就一句话：

messages 是一个列表，用来装聊天记录

为什么必须是列表？

因为聊天记录是一堆消息的集合：用户说一句、AI 回一句、工具返回结果……这些消息需要按顺序存起来。 在 Python 里，列表（list） 就是干这个事的——它可以装多个东西，而且有顺序：

messages = [
    HumanMessage("你好"),       # 第一条：用户说的
    AIMessage("有什么能帮您？"), # 第二条：AI 回的
    HumanMessage("查天气"),     # 第三条：用户又问
    ...                           # 继续追加……
]

列表是怎么更新的？—— 追加 vs 覆盖

这是初学者最容易搞混的地方，记住一个对比就好：

写法 更新行为 比喻
messages: list（普通列表） 覆盖：新消息替换旧消息 像黑板，每次写新的就擦掉旧的
messages: Annotated[list, add_messages] 追加：新消息添加到末尾 像日记本，每天新内容写在后面一页

MessagesState 默认就是追加模式（因为它内部用的是 Annotated[list, add_messages]）。

【图解】追加模式的更新过程：

初始状态：                                    messages = []               ← 空列表
第一步 invoke({messages: [用户消息]})：         messages = [用户消息]       ← 初始传入
    ↓ call_model 返回 {"messages": [AI回复]}   
    ↓ LangGraph 内部自动合并                    messages = [用户消息, AI回复]
    ↓ 
第二步 invoke（带着历史继续）：                   messages = [用户消息, AI回复, 新问题]
    ↓ call_model 返回 {"messages": [新AI回复]}  
    ↓ LangGraph 内部自动合并                    messages = [用户消息, AI回复, 新问题, 新AI回复]

那为什么节点 return 时还要包成 [res]？

因为 MessagesState 的 messages 字段始终是列表，所以 return 的时候， 你新加的消息也必须包在 列表中 返回，即 {"messages": [res]}——即使你只加了一条：

# ❌ 错误写法：返回单条消息
return {"messages": res}   # 报错！messages 必须是列表

# ✅ 正确写法：包成列表
return {"messages": [res]} # 把单条消息 res 装进列表

【类比】就像你去快递站寄东西： - 你寄的是一个包裹（单条消息 res） - 但快递站只接收整箱的（列表 [res]） - 所以你把这个包裹装进箱子里再交给快递站（[res]） - 快递站自动把你这箱东西合并到已有货架上（追加到 messages 列表）

2.1 消息追加机制：覆盖 vs 追加

2.3 条件边原理

条件边（Conditional Edge）让图不再是固定路线，而是根据当前状态决定下一步走向。

普通边 (Edge):
  A → B  （固定路线，不可变）

条件边 (Conditional Edge):
  A → (if 条件B  then  B  if 条件C  then  C...)  
  判断函数返回字符串 → 决定走哪条路

条件边不可替代的两个能力：循环 + 分流。

2.4 实战：条件边路由——AI客服品类分流

业务场景：客户说出需求 → LLM 判断信息是否充分 → 条件边决定下一步： 1. 信息不够 → 继续收集 2. 信息够了 → 分类后分流给对应品类客服（手机 / 家电 / 服装）

同一个条件边，能走向不同方向（继续收集 or 去分类），这就是条件边的核心价值。

【注解】add_conditional_edges(source, router_fn)：LangGraph 的条件边方法。第一个参数是源节点名，第二个参数是路由函数。路由函数接收当前 state，返回一个字符串（目标节点名）。

【注解】这里用了两组条件边来展示条件边的灵活性： - 第 1 组（judge → route）：判断信息是否足够。不够就回 ask 继续收集，够了就去 classify 分类。 - 第 2 组（classify → route2）：根据分类结果分流到手机/家电/服装三个品类客服。

两组条件边本质都是判断+分流——根据 state 中的某个值，走向不同的节点。

3.2 人工干预 (Human-in-the-Loop)

先理解三个概念

1. interrupt()
LangGraph 的暂停函数。节点里调用它，图就挂起等外部唤醒。类似 input() 效果，但不阻塞。

2. MemorySaver()
检查点。必须配，用来保存暂停时的状态（跑到哪了、数据是什么）。没有它，interrupt 没法用。

3. thread_id
会话 ID。图编译时指定 checkpointer 后，每次 invoke 要带 thread_id，用来区分不同的对话，互不干扰。

关系：MemorySaver(存状态) + thread_id(区分会话) + interrupt(暂停点) = 可恢复的工作流

【注解】interrupt()：LangGraph 的暂停函数。在节点中调用后，图会挂起并返回一个暂停状态给外部调用者，等待 Command(resume=...) 唤醒。

【注解】MemorySaver()：LangGraph 的内存检查点。编译时传入 checkpointer=memory 后，图每次运行到 interrupt() 都会保存当前状态，后续可以通过 get_state() 查看。

4.1 AutoGPT 核心循环

AutoGPT 是一个单 Agent 自主循环系统，核心是 拆解 → 执行 → 观察 → 再拆解 的循环：

  1. 拆解任务：用 LLM 生成当前该做的任务列表
  2. 执行任务：取第一个任务，模拟执行
  3. 观察决策：还有任务 → 回到第 1 步重新拆解；没了 → 结束

和 BabyAGI 不同：AutoGPT 每次回到 think（重新生成）， 可以根据执行结果增删改任务列表，而 BabyAGI 只对已有任务重排顺序。

简单记：AutoGPT = 重做计划，BabyAGI = 重排顺序。

4.2 BabyAGI — 任务驱动的循环执行

BabyAGI 的核心是一个 任务队列 + 优先级重排 + 循环执行 模式： 1. 任务队列：按优先级从高到低排列的任务列表 2. 逐个执行：从队列头部取出最高优任务执行 3. LLM 动态重排：根据执行结果，让 LLM 重新评估剩余任务的优先级 4. 自动结束：队列为空时循环停止

BabyAGI 的灵魂是 「取最高优 → 执行 → 根据结果重排剩余任务 → 继续」 的持续循环。

这里的 重排 不是写死的，而是用 LLM 动态决策，这才符合真实的 BabyAGI。

为了讲课效率，本示例使用硬编码任务（跳过 LLM 拆解，聚焦于 循环 + 动态重排 机制）。

5.1 层级主管模式原理

主管模式的核心是分而治之——一个主管 Agent 负责拆解任务、分发给多个专家 Agent，然后汇总结果。

                 ┌───── 主管Agent ─────┐
                 │  分解任务 | 汇总结果 │
                 └──┬────────┬────────┬─┘
                    ↓        ↓        ↓
                ┌──────┐ ┌──────┐ ┌──────┐
                │技术  │ │市场  │ │财务  │
                │Agent │ │Agent │ │Agent │
                └──────┘ └──────┘ └──────┘

关键概念： - Fan-out（分发）：一个节点指向多个节点，任务并行分发 - Fan-in（汇聚）：多个节点指向同一个节点，等待所有完成后汇总

本节的实现采用 并行 Fan-out/Fan-in 模式，三个专家同时执行。

6.1 MCP 是什么？

MCP = Model Context Protocol（模型上下文协议），由 Anthropic 提出的 Agent 间通信标准。

类比：就像 USB-C 统一了充电接口，MCP 统一了 Agent 与外部工具的通信接口。

【注解】MCP 的核心价值：不管底层使用什么传输方式（本地进程、HTTP、WebSocket），上层代码的调用方式（ClientSession）完全一致。这样开发 Agent 工具时不需要关心通信细节。

6.2 工作模式对比

MCP 采用 C/S（客户端/服务端）架构： - 服务端（Server） — 定义工具，等待客户端调用（server.py） - 客户端（Client） — 主动发起请求，调用服务端的工具

三种传输模式的区别仅在于通信方式不同，上层 API（ClientSession）完全一致：

模式 传输方式 服务端启动 客户端在哪 适用场景
stdio 子进程 stdin/stdout 由客户端自动拉起，无需手动启动 mcp_stdio_client.py（终端运行） 本地开发，最常用
sse HTTP SSE（单向推送） 改 server.py 中 transport="sse" 再运行 笔记本 6.4.1 代码单元格 跨网络，远程部署
streamable-http HTTP 双工流 改 server.py 中 transport="stream" 再运行 笔记本 6.4.2 代码单元格 实时双向交互

【注解】stdio 模式在 Windows 的 Jupyter Notebook 中不可用（子进程 stdin/stdout 交互不支持）， 因此 stdio 客户端单独写在 mcp_stdio_client.py 文件中， 需要在终端执行 python mcp_stdio_client.py 来运行。

6.4.3 模式三：stdio 模式

stdio 模式通过子进程的 stdin/stdout 通信，由 stdio_client 自动拉起服务端子进程，无需手动启动。

为什么 stdio 客户端要单独放在一个 .py 文件中？
因为 Windows 上的 Jupyter Notebook： 1. 无法通过子进程 stdin/stdout 与 MCP 服务端交互
2. 自带事件循环（asyncio event loop），与 asyncio.run() 冲突
所以 stdio 客户端必须作为独立 .py 文件，在终端中执行：

python mcp_stdio_client.py

import asyncio, json
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

async def call_tool(client_fn, arg, query, tool, params):
    print(f"[请求] {query}")
    print(f"[调用] {tool}({json.dumps(params)})")
    # stdio_client 自动拉起服务端子进程
    async with client_fn(arg) as (r, w):
        async with ClientSession(r, w) as s:
            await s.initialize()
            tools = await s.list_tools()
            print(f"[发现] {len(tools.tools)} 个工具")
            res = await s.call_tool(tool, params)
            print(f"[返回] {res.content[0].text}")

params = StdioServerParameters(command="python", args=["server.py"])
asyncio.run(call_tool(stdio_client, params,
                      "查用户信息", "query_user", {"user_id": 1}))

【注解】StdioServerParameters(command="python", args=["server.py"]) 告诉 stdio_client 要拉起什么进程。它会自动启动 python server.py 子进程，通过 stdin/stdout 通信，无需手动在终端启动服务端。

【注解】三种传输模式的上层 API (ClientSession) 完全一致，区别仅在于 sse_client / streamable_http_client / stdio_client 这三个传输层函数不同。

【注解】asyncio.run() 是 Python 异步代码的入口。MCP 客户端基于异步 (async/await)，需要在 async 函数中调用，然后用 asyncio.run() 启动。

模块总结

LangGraph 核心概念

概念 一句话理解 核心思想
State（状态） 数据包，在节点之间传递 每个节点只修改自己关心的字段，没改的自动保留
Node（节点） 处理函数，每个做一件事 一个函数只干一件事，返回要更新的字段
Edge（边） 固定路线：A 做完直接去 B 适合有明确先后顺序的处理流程
Conditional Edge（条件边） 岔路：根据当前状态决定下一步去哪 可以形成循环（反复执行）或分流（走向不同节点）
Fan-out / Fan-in（扇出/扇入） 一个节点分发给多个 / 多个汇聚到一个 扇出并行执行，扇入等待全部完成再汇总
Human-in-the-Loop（人工干预） 机器做事，人做决策 需要人确认的地方暂停，收到指令再继续

多 Agent 模式

模式 适用场景 实现方式
Sequential（流水线） 有明确先后顺序的任务 节点首尾相连，一个接一个执行
Hierarchical（主管） 需要任务分解 + 专家分工 主管拆任务→专家执行→主管汇总
Collaborative（辩论） 需要多角度评审决策 多 Agent 并行给观点，裁决者综合判断
AutoGPT 完全自主的单 Agent 循环 Think→Act→Observe，自己规划-执行-检查
BabyAGI 任务驱动的多步执行（拆解+优先级+结果驱动） LLM 拆目标→排优先级→执行→根据结果生成新任务→循环

MCP 核心概念

概念 一句话理解
C/S 架构 Server 定义工具（等人调用），Client 主动发起请求
传输与协议分离 传输层（stdio/SSE/HTTP）可变，协议层 API 完全一致
stdio 模式 本地子进程通信，客户端自动拉起服务端，最常用
SSE / Streamable HTTP 网络模式依赖 HTTP，需要手动启动服务端
三步调用 初始化连接 → 发现工具 → 调用工具
