# 第1章：微调原理基础

---

## 1.1 全参数微调（Full Fine-tuning）

全参数微调是微调的最原始形式：**直接更新模型所有参数**。理解它是掌握 PEFT 方法的前置基础。

### 微调核心流程

```
完整微调流程：

预训练模型
    │
    ▼
┌──────────────────────────────┐
│ 1. 加载预训练权重             │  ← from_pretrained
├──────────────────────────────┤
│ 2. 准备下游任务数据           │  ← Dataset / DataLoader
├──────────────────────────────┤
│ 3. 配置训练参数 + 优化器      │  ← TrainingArguments + AdamW
├──────────────────────────────┤
│ 4. 前向传播 → 计算Loss        │
│    → 反向传播 → 更新所有权重   │  ← 关键：所有参数都更新
├──────────────────────────────┤
│ 5. 保存微调后模型 和词表            │
└──────────────────────────────┘
```

【注解】全参数微调的核心在第 4 步：**所有参数的梯度都要计算和更新**。这是它"贵"的根本原因。

以 0.8B 模型（权重 BF16）为例：

模型权重（BF16）：**1.6GB**

梯度（FP32）：**3.2GB**

AdamW 优化器状态（FP32）：

- m（一阶动量）：记录梯度的指数移动平均，主要决定更新的"方向"，抑制梯度噪声 -- **3.2GB**

- v（二阶动量）：记录梯度平方的指数移动平均，用来自适应地缩放每个参数的更新幅度 -- **3.2GB**

仅梯度和优化器就占 **~9.6GB**（权重本身的 6 倍），训练总计 **~11GB**。这也是全参微调显存占用远高于推理，以及 LoRA / QLoRA 能大幅节省资源的关键原因。

### 灾难性遗忘（Catastrophic Forgetting）

全参数微调最大的隐患：**模型学了新任务，忘了旧知识**。

```
灾难性遗忘示意：

微调前： 模型能完成 [翻译, 写作, 问答, 编程, 推理, ...]

微调中： 只喂"客服"数据，反复更新所有参数
         ↓
微调后： 模型只擅长 [客服]，其他能力大幅退化

原因： 所有权重被新数据"冲刷"，原有知识被覆盖
```

**缓解策略**：

| 策略 | 原理 | 效果 |
|---|---|---|
| **混合回放** | 训练数据中混入 10-20% 通用数据 | 较好，实现简单 |
| **学习率衰减** | 使用比预训练小 10-100x 的学习率 | 基础手段 |
| **冻结底层** | 冻结前几层 Transformer，只训后几层 | 有效但不灵活 |
| **EWC（弹性权重巩固）** | 对重要参数施加 L2 惩罚，限制偏离 | 理论好，实现复杂 |
| **切换到 LoRA** | **根本解决：只加少量参数不动原权重** | ★★★★★ 推荐 |

【注解】LoRA/QLoRA 之所以流行，核心原因之一就是**天然规避灾难性遗忘**——原始权重冻结不变，新知识通过低秩矩阵学习，互不干扰。

## 1.2 为什么需要高效微调？

**传统全量微调（Full Fine-tuning）**的问题：

- 需要更新模型 **所有参数** → 0.8B 模型权重 1.6GB（BF16）+ 梯度 3.2GB（FP32）+ 优化器状态 6.4GB（FP32）→ 实际 **~11GB**（约 **7 倍于模型权重本身**）
- 每个任务都要**完整复制**一份模型 → 10 个任务 = 10 × 0.8B 参数的独立模型
- 存储、切换、管理成本极高

**PEFT（Parameter-Efficient Fine-Tuning）**的核心思想：**只更新少量参数**，冻结大部分预训练权重。

【注解】**全量微调 vs PEFT 方法**的类比：

- **全量微调**：为了改一个房间的装修，把整房间拆了重建（更新所有参数）。
- **Adapter**：**在房间里添置家具/装饰**（在 Transformer 层内插入 bottleneck 小网络），房间结构没动，但多了几件可拆装的物件。
- **LoRA**：**改动房间的门口**（在 W₀x 这个"通道"旁并联一个 BA 小通道），房间里的东西丝毫没动，只是进出的路径加了条辅助旁路。
- **Prefix/Prompt Tuning**：**在大楼门口贴张提示牌**（在输入序列前拼接可学习的虚拟 token），房间、门都不动，只是引导人怎么走。

核心区别：Adapter 在"房间内部"加模块，LoRA 在"房间门口"加旁路，Prompt Tuning 只在"大楼入口"加指引。但共同点是整栋楼的结构（预训练权重）完全不动。

### 主流 PEFT 方法对比

|方法|可训练参数|新增参数量（0.8B模型）|显存需求|质量|
|---|---|---|---|---|
|**全量微调**|100%|0.8B（全部更新）|~11GB|最高|
|**LoRA**|0.1-1%|~1-8M|**~2.5GB**|≈ 全量|
|**Adapter**|1-3%|~6-18M|~3GB|≈ 全量|
|**Prefix Tuning**|0.01%|~0.1M|~2GB|≈ 全量（简单任务）|
|**Prompt Tuning**|0.001%|~0.01M|~2GB|略低|

【注解】LoRA 是当前最主流的 PEFT 方法。它的核心优势是：**训练时显存低、推理时零开销**（LoRA 权重可以合并到原权重中，不增加推理延迟）。

## 1.3 LoRA 原理：低秩分解

**LoRA（Low-Rank Adaptation）**：将权重更新量分解为两个低秩矩阵的乘积。

```
数学原理：

  原始: h = W₀x          （W₀: d×d, 冻结）
  LoRA: h = W₀x + BAx    （B: d×r, A: r×d, 训练）
  
  其中 r << d（典型值 r=8, 16, 32）
  
  参数量对比：
  原始 W₀: d × d = 4096 × 4096 = 16.7M
  LoRA BA: d×r + r×d = 2 × 4096 × 8 = 65K（rank=8时）
  
  压缩率: 16.7M / 65K ≈ 257 倍！
```

【注解】微调时不需要在高维空间（d×d）自由移动，只需要在低维子空间（r 维）中搜索。实验证明 r=8 或 16 已经足够，更大的 r 效果提升有限。

### LoRA 三步工作流

三步走完 LoRA 的一轮前向 + 反向：

```
① 识别目标层
   选取 Transformer 中的关键线性层（Q/K/V/FFN）。
   这些层的权重矩阵对下游任务影响最大，
   是注入低秩旁路矩阵的最佳位置。

② 注入旁路矩阵
   为每个目标层添加两个小矩阵 A（向下投影）和 B（向上投影）。

   原始:  h = W₀x                       （W₀ 冻结，不训练）
   LoRA:  h = W₀x + (B · A)x            （只训练 A 和 B）

   其中 A: r×d（降维-- 对x降维），B: d×r（升维-- 对Ax升维），秩 r 极小（通常 4~64）

   压缩比 =  大矩阵参数量       d × d
          ──────────────── = ─────────────
          2个小矩阵参数量      d×r + r×k

   例如 Q 矩阵 (d=k=4096, r=8)：压缩比 = 4096²/(2×4096×8) ≈ 256 倍

③ 梯度更新
   反向传播时：
   - W₀ 的梯度不计算，权重保持冻结
   - 只计算 A 和 B 的梯度并更新
```

【注解】三步法记忆口诀：**找对层 → 加旁路 → 只练 A 和 B**。W 全程冻，A 做降维（信息压缩），B 做升维（信息还原），两者合起来等效于在原通道旁边加了一条低秩辅助通道。

### LoRA 关键超参数

|参数|默认值|说明|调参经验|
|---|---|---|---|
|`r`（rank）|8|低秩维度，控制可训练参数量|r=8 起，r=16 效果好，r=32 收益递减|
|`lora_alpha`|32|缩放因子（输出 = W₀x + α/r *BAx）|通常设为 2×r 或 4×r|
|`target_modules`|q_proj, v_proj|哪些模块应用 LoRA|建议包含所有注意力 + 部分 MLP|
|`lora_dropout`|0.05|防止过拟合（小数据集有用）|数据量大时可设为 0|
|`bias`|"none"|是否训练偏置项|设 "none" 最省|

【注解】`lora_alpha` 的作用：实际计算时 **BAx 会乘以 α/r**。这是因为随机初始化时 BA 的方差是 O(1/r) -- 初始化方差，乘以 α/r 可以稳定训练。如果 α=32, r=8，则缩放因子为 32/8=4。

### Target Modules 详解：哪些层该挂 LoRA？

`target_modules` 决定了 LoRA 适配器挂载在 Transformer 的哪些线性层上。选对层直接影响微调效果和参数量：

```
Qwen3.5 的 Transformer Decoder 层结构：

  输入 → RMS Norm → ┬─ q_proj（查询投影）──┐
                     ├─ k_proj（键投影）────┤
                     ├─ v_proj（值投影）────→ 自注意力 → o_proj（输出投影）→ +
                     │                                              ↑
                     └─ gate_proj（门控）→ SiLU ⊗ up_proj（上投影）→ down_proj（下投影）→ +
                                                      ↑              FFN 部分
                                                   激活值
```

| 模块名 | 全称 | 所在位置 | 作用 | 是否推荐挂 LoRA |
|---|---|---|---|---|
| **q_proj** | Query Projection | 自注意力 | 将输入投影为 Query 向量，决定"我要查什么" | ✅ 必选 |
| **k_proj** | Key Projection | 自注意力 | 将输入投影为 Key 向量，决定"我是什么内容" | ✅ 推荐 |
| **v_proj** | Value Projection | 自注意力 | 将输入投影为 Value 向量，决定"我要输出什么信息" | ✅ 推荐 |
| **o_proj** | Output Projection | 自注意力 | 将多头注意力结果合并回原始维度 | ✅ 推荐 |
| **gate_proj** | Gate Projection | FFN（SwiGLU） | 门控信号，控制 FFN 中哪些信息通过 | △ 可选 |
| **up_proj** | Up Projection | FFN（SwiGLU） | 将输入升维（如 2048→5504），增加表达能力 | △ 可选 |
| **down_proj** | Down Projection | FFN（SwiGLU） | 将升维后的结果降回原始维度（如 5504→2048） | △ 可选 |

**三种常用配置组合**：

| 配置 | modules | 适用场景 | 参数量占比 |
|---|---|---|---|
| 保守（省参数量） | `q_proj, v_proj` | 简单任务、快速实验 | ~0.3% |
| 标准（推荐） | `q_proj, k_proj, v_proj, o_proj` | 大多数下游任务 | ~0.5% |
| 激进（最强） | `q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj` | 复杂任务、追求极致效果 | ~1% |

【注解】**新手建议用"标准"配置**（4 个注意力层）。注意力层对语义理解最关键，挂 LoRA 效果显著；FFN 层（gate/up/down）主要存储知识，挂 LoRA 带来的提升较小但参数翻倍。在 2.4 的 WebUI 和 3.2 的代码示例中，你会看到这两种配置的实际用法。

### 显存分析：为什么 LoRA 比全量微调省显存？

```
全量微调 vs LoRA 的显存组成：

全量微调（以 Qwen3.5-0.8B 为例）:
  模型权重(BF16):   ~1.6GB
  梯度(FP32):       ~3.2GB（FP32 是 BF16 的 2 倍）
  优化器 m+v(FP32): ~6.4GB
  ----------
  合计:            ~11GB

LoRA 微调（rank=8）:
  模型权重:          ~1.6GB（冻结，不计算梯度）
  LoRA 权重:         ~0.003GB（仅这项更新梯度）
  LoRA 梯度:         ~0.003GB
  LoRA 优化器状态:   ~0.006GB
  激活值（中间结果）:  ~0.5-1GB
  ----------
  合计:              ~2.1-2.6GB
```

【注解】这是 LoRA 最大的优势：**不需要存储全量参数的梯度和优化器状态**。LoRA 只需要更新几十兆参数的梯度，而不是几十 GB。

### LoRA 核心优势总结

| 优势 | 具体表现 |
|---|---|
| **参数高效** | 仅微调原模型参数的 **0.1% ~ 1%**，存储从 GB 级降至 MB 级 |
| **内存友好** | 训练显存占用大幅降低（无需存储大权重 W 的梯度） |
| **避免遗忘** | 基础模型 **W 冻结**，原始能力完全保留，天然规避灾难性遗忘 |
| **灵活部署** | 可切换不同任务的适配器，也可合并回原模型实现**零推理开销** |
| **训练快速** | 参数少 → 收敛快，硬件要求低，单卡即可完成 |

【注解】这五大优势环环相扣：因为 W 冻结不训练（「避免遗忘」），所以不需要存储 W 的梯度和优化器状态（「内存友好」），只训极少的 A/B 参数（「参数高效」），因此收敛更快（「训练快速」），最终部署时还能零开销切换（「灵活部署」）。

## 1.4 QLoRA 原理：量化 + LoRA

**QLoRA** = LoRA + 4-bit 量化，在 0.8B 模型上进一步省显存（约 0.4GB），甚至可以在单卡 RTX 4090 上微调 35B 大模型。

|对比项|LoRA|QLoRA|
|---|---|---|
|基础模型精度|BF16/FP16（~1.6GB/0.8B）|INT4/NF4（~0.7GB/0.8B）|
|单卡 6GB 就能跑|Qwen3.5-0.8B ✅|QLoRA 甚至能跑 9B 模型|
|训练质量|≈ 全量微调|≈ LoRA（差距 < 1%）|
|训练速度|基准|略慢（需解量化计算）|

【注解】QLoRA 的关键技术：使用 BitsAndBytes 的 **NF4（Normal Float 4-bit）** 量化基础模型权重，并在计算时通过**双量化（对量化参数本身再量化）**和**分页优化器（显存不足时自动借用 CPU 内存）**进一步压缩显存。训练时只更新 FP16 的 LoRA 参数，极大降低了显存占用。

## 1.5 AdaLoRA 原理：自适应低秩适配

**AdaLoRA（Adaptive Low-Rank Adaptation）** 是 LoRA 的重要升级版本。核心机制：**所有层一开始用相同的初始 rank，训练过程中动态裁剪 Λ 的对角线元素，让不同层自然演化出不同的有效秩**。

```
LoRA： 所有目标模块 rank 固定（如 r=8）
       q_proj: 8  |  k_proj: 8  |  v_proj: 8  |  o_proj: 8
       ← 不管这些模块哪个更重要，一视同仁

AdaLoRA：所有层初始 r 相同（如 r=12），训练中 Λ 的对角元素动态裁剪
       q_proj: 12  |  k_proj: 6  |  v_proj: 10 |  o_proj: 4
       ↑ 训练完成后观察到的结果（并非训练时主动"分配"不同的 r）
       总秩预算不变（如总 r_budget = 32）
```

**数学原理**：

```
LoRA:  ΔW = BA              (B: d×r, A: r×d)
AdaLoRA: ΔW = P·Λ·Q         (P: d×r, Λ: r×r 对角, Q: r×d)

关键差异：
- LoRA 无约束，A 和 B 可能线性相关
- AdaLoRA 强制正交化 P^T·P = Q·Q^T = I
- Λ = diag(σ₁, σ₂, ...)，每个 σᵢ 代表对应秩分量的重要性
- 训练过程中，不重要的 σᵢ 逐渐被裁剪为 0，该秩分量消失
- 最终各层剩余的非零 σᵢ 数量不同 → 表现出"不同层不同有效秩"
```

| 对比维度 | LoRA | AdaLoRA |
|---|---|---|
| rank 机制 | 固定 r，全程不变 | 初始 r 固定，训练中裁剪 Λ 对角元素，有效秩自然分化 |
| 参数效率 | 好 | **更好**（同预算下效果优 1-3%） |
| 训练开销 | 基准 | 略高（需计算 SVD + 正交约束） |
| 实现复杂度 | 简单 | 中等 |

【注解】AdaLoRA 的精髓：初始 rank 所有层相同，但 Λ 的对角元素会在训练中自然分化——重要的分量 σ 被保留，不重要的被裁剪为 0。训练结束后观察，会发现不同层剩下的有效秩不同。比如医疗微调中，模型自然会保留"症状识别相关层"更多秩分量。日常微调**优先用 LoRA/QLoRA**，对效果有极致追求时再考虑 AdaLoRA。

## 1.6 指令微调（SFT）原理

**SFT（Supervised Fine-Tuning）** 是让预训练模型学会"听从指令"的关键步骤。

预训练模型只会做一件事：**预测下一个 token**。它不知道什么是"回答问题"、"写文章"、"翻译"。指令微调就是用一个**高质量的指令-回答数据集**教模型这些技能。

```
SFT 的训练数据格式：

  {
    "instruction": "用Python实现一个二分查找",
    "output": "def binary_search(arr, target):\n    left = 0\n    right = len(arr) - 1\n    ..."
  }

  {
    "instruction": "翻译下面的句子到英文：今天天气真好",
    "output": "The weather is nice today."
  }

  训练时：把 instruction 和 output 拼接成一段文本
  模型的任务：根据 instruction 部分，预测 output 部分的每个 token
```

【注解】SFT 的训练方式仍然是**语言模型的目标（next token prediction）**，只是训练数据从纯文本变成了高质量的指令-回答对。模型通过大量的指令数据学会了"当用户问一个问题时，应该输出一个答案"这个模式。

### 训练参数调优指南

|参数|推荐值|说明|
|---|---|---|
|learning_rate|1e-4 ~ 3e-4|LoRA 用较大学习率（全量微调通常 2e-5）|
|batch_size|4-8|受显存限制，通过 gradient_accumulation 增大|
|epochs|2-5|数据量大时 2-3 轮足够，小数据集 3-5 轮|
|max_seq_length| 1024                     |太长 = 显存不够；太短 = 长回答被截断|
|warmup_ratio|0.03-0.1（约 50-200 步）|让学习率从 0 逐渐增加到目标值，稳定训练。推荐用 ratio 而非绝对步数，避免 total steps 变化时 warmup 占比失调|
|weight_decay|0.01-0.1|L2 正则化，防止过拟合|

【注解】LoRA 的学习率通常比全量微调大 10 倍（2e-4 vs 2e-5）。因为 LoRA 只更新很少的参数，需要更大的更新步长才能充分学习。**建议先用 100 步做小实验，观察 loss 曲线后再决定是否调整学习率**。

## 1.7 多任务微调原理

**多任务微调（Multi-Task Fine-tuning）**：用多个不同类型的任务数据同时训练，让模型在多个领域都有好的表现。

**两种多任务 LoRA 策略**（底层都是基座权重冻结不变）：

| 策略 | 做法 | 优缺点 |
|---|---|---|
| **共享 LoRA（混合训练）** | 所有任务数据混在一起，训练**同一组** LoRA 适配器 | 简单直接，一个 adapter 走天下；任务间可能互相干扰 |
| **独立 LoRA（各自训练）** | 每个任务单独训练**各自的** LoRA 适配器，推理时按需加载/合并 | 互不干扰，但需管理多个 adapter 文件 |

```
单任务微调的问题：
  医疗数据 → 微调 LoRA_A → 擅长医学 → 但翻译/写作退化
  客服数据 → 微调 LoRA_B → 擅长服务 → 但推理/编程退化

共享 LoRA（同一 adapter 学所有任务）：
  医学 + 客服 + 翻译 + 通用 → 微调同一组 LoRA → 多领域均衡提升

独立 LoRA（每个任务各学各的）：
  医学数据 → 微调 LoRA_医疗
  客服数据 → 微调 LoRA_客服
  翻译数据 → 微调 LoRA_翻译
  → 推理时 merge 对应的 adapter，基座权重始终不变
```

> **实际选择**：大部分场景用**共享 LoRA**就够了（数据混一起跑），除非各任务差距极大（如医疗问答 vs 代码生成），才考虑独立 LoRA。

**多任务训练策略**（共享 LoRA 场景下）：

| 策略 | 做法 | 适用场景 |
|---|---|---|
| **简单混合** | 所有数据混在一起 shuffle | 各任务数据量接近 |
| **加权采样** | 对数据少的任务提高采样概率 | 数据量差异大 |
| **任务标识** | instruction 中加入任务类型前缀 | 让模型区分不同任务 |
| **渐进训练** | 先训简单任务，再训复杂任务 | 任务难度差异大 |

### 小结

|知识点|核心内容|
|---|---|
|全参数微调|更新所有参数，显存高（~11GB），易灾难性遗忘|
|灾难性遗忘（全参微调）|学新忘旧，LoRA 是最佳缓解方案|
|PEFT 思想|冻结预训练权重，只更新少量参数|
|LoRA 原理|低秩分解 ΔW = BA，r=8/16 即可|
|QLoRA 原理|4-bit 量化 + LoRA，单卡 4GB 可跑 4B|
|AdaLoRA 原理|初始 rank 固定，训练中裁剪 Λ 对角元素，有效秩自然分化|
|SFT 原理|用指令-回答对训练 next token prediction|
|多任务微调|混合多领域数据，保持模型通用能力|

### 练习题

**练习 1（基础）**：用自己的话解释 LoRA 的数学原理（W₀x + BAx），并说明为什么 r=8 就能达到接近全量微调的效果。

**练习 2（进阶）**：对比 LoRA 和 QLoRA 的显存组成，计算在 0.8B 模型上各需要多少显存（假设 BF16 基础模型 1.6GB，4-bit 量化后 0.4GB，LoRA 参数约 2MB）。

**练习 3（思考）**：同时用医疗和客服数据做多任务微调时，如果医疗数据只有 500 条、客服有 2000 条，你打算怎么避免模型"偏科"？说出至少一种策略。

## 2.1 LLaMA Factory 简介

**LLaMA Factory** 是一个开源的大模型微调全流程管理平台，特点：

- **零代码/低代码**：通过 WebUI 页面点选即可运行完整训练流程

- **开箱即用**：数据预处理(写脚本实现）、训练、评估、合并、导出一条龙

【注解】**LLaMA Factory 的底层是谁？** 它基于 Hugging Face 官方的 **PEFT 库**构建。PEFT 提供了 LoRA/QLoRA 的数学实现（低秩分解、量化、参数注入），LLaMA Factory 在 PEFT 之上封装了 WebUI、数据管理、一键训练等使用者友好的功能。可以把 PEFT 理解为"发动机"，LLaMA Factory 是"整车"——你不用打开引擎盖就能开，但动力来自 PEFT。

【注解】对于初学者，LLaMA Factory 是**上手微调的最佳入口**。你不需要写一行 Python 训练代码，只需要在页面上选参数和准备数据，就能跑通完整的 LoRA 微调流程。先掌握"怎么微调"和"微调步骤"，再深入底层代码。

## 2.3 数据准备

真实场景中，数据往往以 CSV 形式存在。完整工具链：**CSV → JSONL → dataset_info.json 注册 → WebUI 页面点选 → 一键训练**。

### 第一步：CSV 转 JSONL 脚本

假设你有一个开源医学问答数据集 `样例_内科5000-6000.csv`，字段如下：

| 字段 | 含义 | 示例 |
|---|---|---|
| `department` | 科室 | 内科 |
| `title` | 问答标题 | 高血压患者应该注意什么？ |
| `ask` | 患者提问 | 我最近测血压有点高，日常需要注意什么？ |
| `answer` | 医生回答 | 建议您低盐饮食，每天食盐控制在6克以内... |

用下面这段脚本，一键转为 LLaMA Factory 可用的 JSONL：

```python
# csv_to_jsonl.py —— 将 CSV 格式的问答数据转为 LLaMA Factory 的 JSONL 格式
import pandas as pd

CSV_FILE = "data/样例_内科5000-6000.csv"
JSONL_FILE = "data/medical_internal.jsonl"

df = pd.read_csv(CSV_FILE, encoding="utf-8")

# instruction：根据科室动态生成角色设定；input：患者实际提问；output：医生回答
df["instruction"] = df["department"].apply(
    lambda d: f"你是一位经验丰富的{d}医生，请根据以下患者提问，给出专业、准确的回答。"
)
df["input"] = df["ask"]
df["output"] = df["answer"]

# 直接导出 JSONL（orient="records", lines=True）
df[["instruction", "input", "output"]].to_json(
    JSONL_FILE, orient="records", lines=True, force_ascii=False
)

print(f"转换完成：{len(df)} 条 → {JSONL_FILE}")
```

运行后得到 `data/medical_internal.jsonl`，每行一条：

```jsonl
{"instruction": "你是一位经验丰富的内科医生，请根据以下患者提问，给出专业、准确的回答。", "input": "我最近测血压有点高，日常需要注意什么？", "output": "建议您低盐饮食，每天食盐控制在6克以内..."}
{"instruction": "你是一位经验丰富的内科医生，请根据以下患者提问，给出专业、准确的回答。", "input": "我妈妈刚查出糖尿病，饮食上怎么调整？", "output": "糖尿病饮食要做到：1) 控制主食量..."}
```

格式说明：`instruction` 从 `department` 字段动态生成角色设定，`input` 是具体提问，`output` 是标准答案。LLaMA Factory 训练时会自动将 `instruction + input` 拼接为 prompt，`output` 作为 response。

> LLaMA Factory 支持的三种主流数据格式：

| 格式 | 数据长相 | 适用场景 | dataset_info.json 列映射 |
|---|---|---|---|
| **alpaca** | `{"instruction": "角色/任务", "input": "具体输入", "output": "标准答案"}` | 单轮问答（最通用） | `prompt→instruction, query→input, response→output` |
| **sharegpt** | `{"conversations": [{"from": "human", "value": "xxx"}, {"from": "gpt", "value": "yyy"}]}` | 多轮对话、历史上下文 | `messages→conversations`，LLaMA Factory 自动解析 `from/value` |
| **oasst1** | OpenAssistant 树状标注格式（message_tree） | 开源讨论数据（较少用） | 一般不需要手动配置列映射 |

> **选哪个？** 绝大多数场景用 **alpaca** 就够了。如果你是单轮问答（客服、翻译、写作），用 alpaca；如果是多轮对话（助手聊天），用 sharegpt。

### 第二步：注册数据集

编辑 `data/dataset_info.json`，告诉 LLaMA Factory 数据在哪、按什么列映射：

```json
// ⚠️ 注意：等号左边的 prompt/query/response 是 LLaMA Factory 内部字段名（写死的），
//         等号右边的 instruction/input/output 才是你 JSONL 数据的列名（可自定义）
{
  "medical_internal": {
    "file_name": "medical_internal.jsonl",
    "formatting": "alpaca",
    "columns": {
      "prompt": "instruction",     // ← 右边换成你 JSONL 里实际叫啥
      "query": "input",            // ← 同上
      "response": "output"         // ← 同上
    }
  }
}
```

【注解】`columns` 的映射规则：`"LLaMA Factory 内部名字": "你 JSONL 里的列名"`。

| side | 是啥 | 能改吗 |
|---|---|---|
| 左边 `prompt` / `query` / `response` | LLaMA Factory 内部识别的字段名 | ❌ 固定，不能改 |
| 右边 `instruction` / `input` / `output` | 你 JSONL 数据的列名 | ✅ 自定义，和你数据对齐即可 |

所以如果你 JSONL 的列名叫 `question` 和 `answer`，就写成 `"query": "question", "response": "answer"`。

`dataset_info.json` 的核心作用：
1. 数据集名称（`medical_internal`） —— 供 WebUI 数据集选择框引用
2. 文件路径（`file_name`） —— 相对于 `data/` 目录
3. 格式（`alpaca` / `sharegpt` / `oasst1` 等）
4. 列映射 —— LLaMA Factory 按左边名字定位，按右边名字取数据

注册完后，LLaMA Factory 会自动完成 tokenize、拼接模板、构建 training data 等所有预处理。**零 Python 代码**。

## 2.4 LoRA 微调实战（WebUI 页面操作）

LLaMA Factory 提供 Gradio WebUI，所有配置都在前端页面用下拉框和输入框完成，**不需要写 YAML，不需要写 Python**。

### 第一步：启动 WebUI

```bash
# 在 LLaMA-Factory 项目根目录下执行
llamafactory-cli webui

# 输出：
# Running on local URL: http://localhost:7860
```

浏览器打开 `http://localhost:7860`，你会看到 5 个标签页：**模型配置 → 数据配置 → 训练配置 → 评估预测 → 导出**。

### 第二步：在页面上配置参数

按顺序操作，每配置好一个标签页就点"下一步"：

**① 模型配置页**（决定"用哪个模型、怎么训"）

| 参数 | 填/选什么 | 说明 |
|---|---|---|
| 模型名称 | `Qwen/Qwen3.5-0.8B` | 下拉框搜索 qwen，选 Qwen3.5-0.8B。也支持本地路径 |
| 微调方法 | `lora` | 下拉框：`full`（全参）、`lora`（高效）、`freeze`（冻结部分层） |
| 量化等级 | `none` | LoRA 选 none；QLoRA 选 `4-bit` 或 `8-bit`（bitsandbytes）→ 下一步多出一组量化参数 |
| 模型模板 | `qwen3.5` | 自动根据模型名称匹配。决定对话格式（如 `<|im_start|>user`） |
| 精度 | `bf16` | 有 bf16 选 bf16，没有选 fp16。影响显存 + 训练稳定性 |
| 检查点路径 | 留空或填已有 checkpoint | 从断点继续训练时用，填之前的 `checkpoint-xxx` 路径 |
| 语言 | `zh` | 仅影响 tokenizer 的特殊 token 处理，不影响模型本身支持的语言 |

> 🎯 **新手一次性配好口诀**：模型名称 → 微调方法选 lora → 量化 none → 模板自动匹配 → 精度 bf16 → 其余留空，点下一步即可。

**② 数据配置页**

| 参数 | 选什么 | 说明 |
|---|---|---|
| 数据集 | `medical_internal` | 下拉框里会出现 2.3 节注册的数据集名 |
| 最大样本数 | `10000` | 先用 1000 条快速验证，确认没问题再放开 |
| 截断长度 | 1562 | 超过 1562 token 的文本会被截断 |

> 下拉框里出现 `medical_internal`，说明 2.3 节的 `dataset_info.json` 注册成功了。

**③ 训练配置页（重点）**

| 参数 | 填什么 | 说明 |
|---|---|---|
| 输出目录 | `./medical_lora_output` | 模型和日志存哪 |
| 训练轮数 | `3` | 小数据 3 轮，大数据 1-2 轮 |
| 每卡 batch size | `4` | 显存不够就减小 |
| 梯度累积 | `4` | 等效 batch = 4×4 = 16 |
| 学习率 | `2e-4` | LoRA 用 2e-4（全参数微调才用 5e-5） |
| 学习率调度器 | `cosine` | 可选 `cosine`/`linear`/`constant`。cosine 最常用：先增后降 |
| 预热步数 (warmup) | `50` | 学习率从 0 逐渐增到目标值（约 total steps 的 5-10%） |
| 权重衰减 | `0.01` | L2 正则化系数，防止过拟合，LoRA 通常设 0.01 即可 |
| LoRA 秩 (rank) | 8 | 1.3 节讲过，越高表达能力越强 |
| LoRA alpha | 16 | 缩放系数，一般取 rank 的 2 倍 |
| LoRA 随机失活 | `0.1` | dropout 比例，防过拟合。0.8B 小模型建议 0.05-0.1 |
| LoRA 目标模块 | `q_proj,k_proj,v_proj,o_proj` | 1.3 节的标准配置 |
| 保存 checkpoint 步数 | `100` | 每 100 步存一个 checkpoint |
| 最多保存 checkpoint 数 | `3` | 只保留最近 3 个，防止硬盘爆满 |

> 🎯 **常见翻车点**：① 全参微调选了 lora → 记得先把微调方法改 full；② 学习率全参用 5e-5、LoRA 用 2e-4 搞反了 → LoRA 学习率反而要大；③ warmup 填 0 容易初段震荡 → 至少填 30-50 步。

**④ 点击"开始训练"**

页面顶部出现进度条，终端同步输出训练日志：

```

[INFO] LoRA rank=16, alpha=32, target modules: q_proj,k_proj,v_proj,o_proj
[INFO] Trainable parameters: 8.39M / 0.82B (1.02%)
[INFO] ***** Running training *****
[INFO] Step 10/150: loss=2.156, lr=2e-4
[INFO] Step 50/150: loss=0.512, lr=1.2e-4
[INFO] Save model checkpoint to ./medical_lora_output/checkpoint-100
[INFO] Training completed
```

【注解】WebUI 背后自动完成：加载模型 → 预处理数据 → 配置 LoRA → SFTTrainer 训练 → 保存 checkpoint。整个过程**一次代码都不用写**，连 YAML 文件都不用创建。

## 2.5 QLoRA 微调实战

QLoRA 在 WebUI 上就是**多勾一个框**。

回到 **模型配置页**，在"微调方法"选 `lora` 的基础上：

| 参数 | 选什么 | 说明 |
|---|---|---|

| 量化方法 | `bitsandbytes` | 下拉框选 bnb |

| 方法 | 显存占用 |
|---|---|
| LoRA | ~2.5 GB |

| 底层引擎 | PEFT（Hugging Face 官方） | PEFT（Hugging Face 官方） |
| 上手难度 | **极低**（WebUI 页面点选） | 中等（需写 Python 代码） |
| 灵活性 | 中（前端预定义参数） | **高**（可任意定制代码） |
| 训练速度 | 标准 | **快 2x**（内核优化） |
| 显存优化 | 标准（PEFT 默认） | **省 50%**（Unsloth 特有） |
| 多卡/分布式 | ✅ DeepSpeed ZeRO-2/ZeRO-3，多卡开箱即用 | ❌ 开源版不支持多卡，需手动改代码 |
| GPU 兼容性 | ✅ NVIDIA/AMD/MPS 广泛兼容 | ⚠️ 主要针对 NVIDIA，旧卡/A 卡可能报错 |
| 训练方法 | SFT、RM、PPO、DPO、KTO、ORPO 等 | 主要 SFT，RL 支持有限 |
| 适用场景 | 快速实验、批量任务、团队协作、多卡训练 | 性能敏感、生产训练、深度定制 |
| 支持模型 | 100+ | 主流模型（Llama、Qwen、Mistral 等） |

【注解】**两者底层都依赖 PEFT**。PEFT 是 Hugging Face 官方的参数高效微调库，负责 LoRA 低秩分解、量化适配等核心数学逻辑。LLaMA Factory 和 Unsloth 都是 PEFT 的上层封装——前者侧重"零代码易用"，后者侧重"内核优化提速"。二者并非"谁更好"，而是"不同场景用不同工具"。

**选型速记**：单卡 NVIDIA + 追求极致速度 → Unsloth；多卡/AMD 显卡/需要 WebUI/需要 RLHF → LLaMA Factory；小显存单卡玩 7B+ 全参 → 两边都不行，上 QLoRA 或换更大显存。

学习路径建议：先用 LLaMA Factory 跑通 2-3 个微调任务，熟悉"数据准备→配置→训练→评估→合并"的完整流程；再用 Unsloth 深入代码级优化，掌握训练过程的每个细节。

## 2.9 多卡微调：DeepSpeed ZeRO 讲解

当你从单卡升级到多卡训练，核心问题就一个：**4 张卡怎么分活、怎么省显存**。ZeRO 是 DeepSpeed 提供的解决方案，分 4 个等级。

### 一张图理解 ZeRO-0/1/2/3

想象 4 个工人（GPU）各自搬砖（训练 batch），但仓库（显存）有限。训练消耗显存的有三样东西：**模型参数、梯度、优化器状态**。

```
单卡（数据完全放一张卡上）：
  GPU 0: [模型 1.6GB] [梯度 3.2GB] [优化器 6.4GB] = 11GB
  GPU 1: 闲置
  GPU 2: 闲置
  GPU 3: 闲置
```

| 级别 | 切分什么 | 比喻 | 单卡显存变化（0.8B） | 通讯开销 |
|---|---|---|---|---|
| **ZeRO-0** | 啥也不切 | 每人拿完整账本，各算各的 batch | 11GB（和单卡一样） | 最低 |
| **ZeRO-1** | 只切**优化器** | 账本还是自己的，但仓库平分存放 | 11GB → ~9.5GB | 低 |
| **ZeRO-2** | 切优化器 + **梯度** | 账本还是自己，仓库和运输车都对半切 | ~9.5GB → ~8.3GB | 中 |
| **ZeRO-3** | 切优化器 + 梯度 + **模型参数** | 连账本都撕成 4 份，用到哪页找谁借 | ~8.3GB → ~2.8GB | 最高 |

### 通俗解释

**ZeRO-0（纯数据并行）**：每个 GPU 存一份完整的模型。大家分不同 batch 数据独立算，算完梯度一平均。**显存一点没省**，只是算得快了。这是最早的多卡方式，也叫 DDP（DistributedDataParallel）。

**ZeRO-1**：优化器状态（Adam 的 m 和 v）占显存大头（~6.4GB/0.8B）。ZeRO-1 把它切成 N 份，每张卡只存 1/N。**省了优化器的显存**，通讯开销很小，性价比最高。

**ZeRO-2**：在 ZeRO-1 基础上，连梯度也切了。每张卡算完自己的梯度后，只保留自己负责的那部分，其余丢掉。**优化器 + 梯度都省了**。

**ZeRO-3**：最激进——连模型参数都切分。前向传播时，哪层的参数在别的 GPU 上，就从那借过来用完就丢。**所有东西都切了**，几乎线性省显存（4 卡 ≈ 省 4 倍），但 GPU 之间频繁传数据，速度会慢一些。

### DDP vs ZeRO：什么时候选哪个

**DDP（PyTorch 原生 DistributedDataParallel）** 和 ZeRO-0 效果一样——每张卡都存完整模型，只是数据分片。常被误认为"低级"，但实际上：

| | DDP | ZeRO-1/2/3 |
|---|---|---|
| 启动方式 | `torchrun`（PyTorch 原生） | `deepspeed`（需装 DeepSpeed） |
| 配置复杂度 | **零配置**，代码不改 | 需要 `ds_config.json` 配置文件 |
| 单卡能跑下的模型 | **更快**（无切分通讯开销） | 反而慢（多了 ZeRO 框架层） |
| 单卡跑不下的模型 | 直接 OOM | ✅ 救命方案 |
| 小模型（<3B）多卡 | ✅ **最推荐**，简单且快 | 杀鸡用牛刀，得不偿失 |
| 大模型（7B+）多卡 | ❌ 单卡放不下 | ✅ 唯一选择 |

核心结论：**ZeRO 解决的是"放不下"的问题，不是"跑得快"的问题**。如果你的模型单卡显存够（比如 0.8B LoRA ~2.5GB，单卡 24GB 绰绰有余），多卡直接用 DDP 就行——启动简单、没有额外开销、速度快。只有单卡 OOM 了，才需要上 ZeRO。

### 场景速查：我该用哪个

> 以下以 **LoRA/QLoRA（当前主流）** 为主，全参微调另起一行对比。模型以 Qwen3.5 系列为参照（但也适用 Llama 等其他模型，同样参数规模原则相通）。全参微调显存需求远超 LoRA，能用 LoRA 就别上全参。

| 场景 | 方法 | 方案 | 显存估算（单卡） | 理由 |
|---|---|---|---|---|
| 单卡 24GB + 0.8B | LoRA | **单卡** | ~3GB | 跑着玩，多卡纯属浪费 |
| 单卡 24GB + 9B | LoRA | **单卡**，加 `gradient_checkpointing` | ~20GB | 主流场景，刚好能跑。多卡反而慢（通讯 > 加速） |
| 单卡 24GB + 9B | QLoRA (4-bit) | **单卡**，随便跑 | ~8GB | 极致省显存，还能放大 batch |
| 4×24GB + 9B | LoRA | **DDP** | 单卡 ~20GB | 单卡够用就不切，DDP 零配置，跑得快 |
| 4×24GB + 27B | LoRA | **ZeRO-2** | 单卡 ~54GB → 放不下 | 27B bf16 单卡必 OOM，ZeRO-2 刚好 |
| 4×24GB + 35B-A3B | LoRA (MoE) | **ZeRO-3** | 单卡 ~70GB → 放不下 | MoE 总参数量大，必须 ZeRO-3 切开 |
| 4×80GB + 27B | QLoRA (4-bit) | **DDP** 或单卡 | 4-bit 后 ~16GB | 4-bit 后单卡 80GB 绰绰有余，DDP 纯加速 |
| 多卡异构（算力不同） | LoRA | **LLaMA Factory + ZeRO-1** | —— | DeepSpeed 对异构卡容忍度更高 |

> 📎 **全参微调对比**：同一模型全参显存是 LoRA 的 4-5 倍。例如 9B 全参单卡 ~72GB，必须上 ZeRO-2。**结论：能 LoRA 就 LoRA，全参只在 LoRA 效果不够时才考虑。**

> 🎯 **一句话口诀**：模型单卡放得下 → DDP（或干脆单卡）；单卡放不下 → 上 ZeRO。别盲目套 ZeRO，DDP 简单又快，只是不会上新闻。

### LLaMA Factory 里怎么配

> ⚠️ WebUI 的"加速方式"下拉框（`auto/flashattn2/unsloth/liger_kernel`）是**注意力内核优化**，跟 ZeRO/DDP 不是一回事，别搞混。DDP/ZeRO 通过下面的方式配置：

**DDP — 多卡纯加速**

最简单：启动 LLaMA Factory 时指定可见 GPU，PyTorch 自动启用 DDP。

| 方式 | 操作 |
|---|---|
| **WebUI** | 启动前设环境变量：`set CUDA_VISIBLE_DEVICES=0,1,2,3` → 再 `llamafactory-cli webui`，其余参数照常填 |
| **CLI** | `CUDA_VISIBLE_DEVICES=0,1,2,3 llamafactory-cli train ...` |

**ZeRO-2/3 — 单卡放不下，需要切分显存**

ZeRO 通过 DeepSpeed 配置文件指定，LLaMA Factory 内置了现成的配置文件：

```
examples/deepspeed/ds_z2_config.json   # ZeRO-2
examples/deepspeed/ds_z3_config.json   # ZeRO-3
```

两种生效方式：

1. **YAML 训练配置文件中写**：在 `train.yaml` 里加一行 `deepspeed: examples/deepspeed/ds_z2_config.json`，然后用该 YAML 启动训练
2. **CLI 参数传**：`llamafactory-cli train --deepspeed examples/deepspeed/ds_z2_config.json ...`

> ⚠️ WebUI 的"加速方式"下拉框控制的是注意力内核（flashattn2/unsloth 等），选不了 ZeRO。要用 ZeRO，把 deepspeed 参数写在 YAML 配置文件里，然后用 CLI 启动即可。

> 🔑 **一句话要领**：DDP 只需设环境变量，ZeRO 需要指定 `deepspeed` 配置文件路径。前者零配置，后者多一行参数。

## 3.1 Unsloth 中的类与方法速查

### 模块一 → Unsloth 映射

对照模块一的概念，看 Unsloth 中类和方法是怎么用的：

| 类/方法 | 是什么 | 作用 | 生活类比 |
|---|---|---|---|
| `FastLanguageModel` | **类** | Unsloth 的核心类，封装了模型加载、LoRA 配置等所有功能 | "汽车工厂"——既能造车，又能改装车 |
| `FastLanguageModel.from_pretrained()` | **方法** | 从磁盘加载一个预训练模型 + 分词器，返回 `(model, tokenizer)` | 工厂的"出库"功能——从仓库提一辆现成的车 |
| `FastLanguageModel.get_peft_model()` | **方法** | 给模型挂上 LoRA 适配器，返回改装后的模型 | 工厂的"改装"功能——给车加装涡轮增压器 |
| `FastLanguageModel.for_inference()` | **方法** | 把模型切换到推理模式（合并 LoRA 权重，加速推理） | 工厂的"上路"功能——改装完毕，准备驾驶 |
| `SFTTrainer` | **类**（来自 `trl` 库） | 训练器，负责配置训练参数并执行训练循环 | "驾校教练"——安排训练计划并监督练习 |
| `SFTConfig` | **类**（来自 `trl` 库） | 训练参数的配置容器，存学习率、batch size 等 | "训练计划表"——写清楚每天练几小时、怎么练 |
| `tokenizer` | **对象**（由 `from_pretrained` 返回） | 分词器，把文本切成 token 序列，或把 token 还原为文本 | "翻译官"——把人说的话翻译成模型能懂的数字 |
| `model` | **对象**（由 `from_pretrained` 返回） | 大语言模型本身，0.8B 参数的神经网络 | "大脑"——接收 token，输出下一个 token |

## 3.2 Unsloth 简介与安装

**[Unsloth](https://github.com/unslothai/unsloth)** 同样基于 Hugging Face 官方的 **PEFT** 库构建——底层 LoRA 注入逻辑来自 PEFT，Unsloth 在 PEFT 之上做了极致内核优化：

- 训练速度提升 **2x**（手动 CUDA kernel 替换标准算子）
- 显存节省 **50%**（智能梯度检查点 + 内存布局优化）
- 代码极其简洁，与 Transformers API 完全兼容

【注解】和 LLaMA Factory 一样，Unsloth 的 LoRA 实现也来自 **PEFT**。区别在于：LLaMA Factory 在 PEFT 上加了 WebUI，Unsloth 在 PEFT 上做了底层算子优化。如果你在 2.3/2.4 节用 LLaMA Factory 跑过微调，本质上是 PEFT 在替你干活——现在你在 Unsloth 中会**直接用到 PEFT 的 API**（如 `LoraConfig`），能更清楚地看到 PEFT 的完整面貌。

```bash
# 安装 Unsloth（推荐 conda 环境）
pip install unsloth
# 同时安装 trl（SFTTrainer 依赖）
pip install trl
```

【注解】Unsloth 不是替代 PEFT，而是**在 PEFT 基础上做了内核级优化**。它重写了 attention 和 LoRA 的 CUDA 内核，实现了更快的计算和更少的显存占用。使用方式上，你仍然调用 `get_peft_model()`，只是底层实现更快。

## 3.3 Unsloth 主要模块详解

> 进入实战前，先把每个"零件"摸清楚。本节逐一拆解 Unsloth 的核心 API，每个模块讲**它是什么、关键参数、独立小示例**。实战时你只需把这些零件按流程串起来。

```
Unsloth 微调 = 4 个核心模块按顺序拼接：

  ① FastLanguageModel     → 模型的"工厂"：加载、改装、推理
  ② get_peft_model()      → LoRA 的"配置器"：决定在哪些层挂适配器
  ③ SFTTrainer + SFTConfig → 训练的"教练+计划表"：怎么练、练多久
  ④ train_on_responses_only → 损失的"遮罩器"：只对回答部分算梯度

  串联顺序 = ① 加载模型 → ② 挂 LoRA → 准备数据 → ③④ 开始训练
```

### 3.3.1 FastLanguageModel：模型工厂

`FastLanguageModel` 是 Unsloth 的**唯一入口类**。所有微调操作都从它开始——加载模型、挂 LoRA、切换推理模式，全靠它的三个方法。

```
FastLanguageModel 的三个核心方法（类比汽车工厂）：

  from_pretrained()  → "出库"：从仓库提一辆现成的车（加载预训练模型）
  get_peft_model()   → "改装"：给车加装涡轮增压器（挂 LoRA 适配器）
  for_inference()    → "上路"：改装完毕，准备驾驶（合并 LoRA 权重，加速推理）
```

#### 方法①：from_pretrained()——加载模型和分词器

**做什么**：从本地磁盘或 Hugging Face Hub 加载一个预训练大模型 + 对应的分词器。

**返回值**：`(model, tokenizer)` 元组。`model` 是神经网络对象，`tokenizer` 是分词器对象。

```python
from unsloth import FastLanguageModel

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="Qwen/Qwen3.5-0.8B",   # 模型来源：HF Hub 名或本地路径
    max_seq_length=2048,                # 最长 token 序列长度
    dtype=None,                         # 数据精度：None = Unsloth 自动选择
    load_in_4bit=False,                 # 是否 4-bit 量化加载
    trust_remote_code=True,             # 是否信任远程代码（部分模型需要）
)
```

**参数详解**：

- **`model_name`**（推荐 `"Qwen/Qwen3.5-0.8B"` 或本地路径）— 从哪里加载模型。本地路径更快，避免每次下载。
- **`max_seq_length`**（推荐 `2048`）— 模型能处理的最长 token 序列。过长浪费显存，过短截断长对话。LoRA 建议 512~2048。
- **`dtype`**（推荐 `None`）— 数据精度。`None` = 自动判断（Ampere+ → bf16，旧卡 → fp16），也可手动填 `torch.bfloat16` 或 `torch.float16`。
- **`load_in_4bit`**（推荐 `False`）— 是否 4-bit 量化加载。`False` = LoRA（BF16），`True` = QLoRA（4-bit NF4）。**这是 LoRA 和 QLoRA 的唯一代码区别**。
- **`trust_remote_code`**（推荐 `True`）— 是否执行模型仓库中的自定义代码。Qwen 等模型需要设 `True`。

【注解】`from_pretrained()` 返回的 `model` 和 `tokenizer` 是两个**独立对象**，后面所有操作都要用到它们。`model` 是"大脑"，`tokenizer` 是"翻译官"。

#### 方法②：get_peft_model()——挂 LoRA 适配器

**做什么**：在已加载的模型上注入 LoRA 适配器。LoRA 不会修改原始权重，而是在旁边插入一对小矩阵（A 和 B），训练时只更新这两个小矩阵。

**返回值**：改装后的 `model`（同一个对象，但内部已挂上了 LoRA）。

```python
model = FastLanguageModel.get_peft_model(
    model,                              # 上面 from_pretrained() 返回的模型对象
    r=16,                                # LoRA 秩（rank）
    lora_alpha=16,                       # LoRA 缩放系数
    lora_dropout=0,                      # LoRA dropout 率
    target_modules=["q_proj","k_proj","v_proj","o_proj",
                     "gate_proj","up_proj","down_proj"],  # 挂 LoRA 的层
    use_gradient_checkpointing="unsloth", # 梯度检查点模式（见下方详解）
    lora_alpha_linear=False,             # alpha 是否线性缩放
    use_rslora=False,                    # 是否使用 Rank-Stabilized LoRA
    loftq_config=None,                   # LoftQ 初始化配置（高级用法）
)
```

**参数详解**：

- **`r`**（推荐 `16`）— LoRA 秩，决定适配器矩阵的大小（A: d×r, B: r×d）
  - `8` = 轻量，参数量少，适合简单任务
  - `16` = 默认平衡选择
  - `32/64` = 更强表达力，适合复杂任务，但更吃显存

- **`lora_alpha`**（推荐 `16`）— 缩放系数，LoRA 的实际更新量 = `alpha / r × BA`
  - 通常设为 `r` 的值（alpha = r），缩放因子 ≈ 1
  - 设 alpha = 2×r 可以放大更新幅度

- **`lora_dropout`**（推荐 `0`）— LoRA 层的 dropout 率
  - `0` = 不丢弃（推荐，Unsloth 优化后开 dropout 反而拖慢速度）
  - `0.05~0.1` = 小数据集防过拟合

- **`target_modules`**（推荐 `["q_proj","k_proj","v_proj","o_proj","gate_proj","up_proj","down_proj"]`）— 在哪些线性层挂 LoRA
  - Attention 4 层（q/k/v/o）= 基本覆盖
  - 加上 FFN 3 层（gate/up/down）= 更强微调效果，多任务场景推荐

- **`use_gradient_checkpointing`**（推荐 `"unsloth"`）— 梯度检查点模式，控制激活值存储策略（详见下方对比表）
  - `"unsloth"` = 最省显存，速度损失极小（推荐）
  - `True` = 标准省显存
  - `False` = 不省显存

- **`use_rslora`**（推荐 `False`）— 是否用 Rank-Stabilized LoRA
  - `True` = 缩放因子自动稳定，rank ≥ 32 时推荐
  - `False` = rank = 16 时够用

**use_gradient_checkpointing 三种模式对比**：

| 模式 | 激活值存储 | 显存占用 | 训练速度 | 适用场景 | 原理 |
|---|---|---|---|---|---|
| `False` | 全存 GPU | 最大 | 最快 | 显存够时 | 不做特殊处理 |
| `True` | 只存检查点 | 中等（省 50%） | 慢 20~30% | 标准省显存 | 少存 + 重算 |
| `"unsloth"` | 检查点 + 智能卸载 | 最小（再省 30%） | 慢 ~2% | 6GB 小卡 / 长 文本 | 少存 + 重算 + 异步卸载到 CPU RAM |

**"unsloth" 模式的核心原理**：

1. 标准 `gradient_checkpointing`：前向传播时只存少数"检查点"层的激活，反向传播时从检查点重新计算中间层 → 省了 50% GPU 显存
2. **"unsloth" 在此基础上多做一步**：把检查点激活**异步卸载到 CPU 内存** → GPU 上几乎不存激活值，再省 30% 显存；用非阻塞调用，GPU↔CPU 传输与计算重叠 → 速度损失只有 ~2%
3. **智能决策，不是全量卸载**：小张量留在 GPU（卸载开销反而更大），最后一层不卸载（backward 立刻要用），只卸载"值得卸载"的大激活值

> **为什么 `"unsloth"` 比标准 `True` 更好？**
> - `True`：省 50% 激活值显存，但慢 20~30%（重算开销大）
> - `"unsloth"`：省更多显存（再省 30%），反而只慢 ~2%
> - 原因：卸载到 CPU 比在 GPU 上重算更快——CPU 内存带宽足够，而重算要占用 GPU 计算单元
> - **结论：永远用 `"unsloth"`，没有理由用 `True`**

【注解】`get_peft_model()` 不创建新对象，而是**在原模型上原地挂载**。调用后 `model` 内部多了 LoRA 层，但模型对象还是同一个。

#### 方法③：for_inference()——切换推理模式

**做什么**：训练完成后，把模型切换到推理模式。它会合并 LoRA 权重到原始权重中（merge），并禁用 dropout，加速推理。

```python
# 训练完成后，切换到推理模式
FastLanguageModel.for_inference(model)

# 然后就可以正常推理了
inputs = tokenizer("高血压患者应该注意什么？", return_tensors="pt").to("cuda")
outputs = model.generate(**inputs, max_new_tokens=128)
print(tokenizer.decode(outputs[0], skip_special_tokens=True))
```

【注解】`for_inference()` 相当于把"涡轮增压器焊到发动机上"——LoRA 权重不再单独存在，而是合并进了原始权重。推理时不再有 LoRA 的额外计算开销，速度和原始模型一样快。**训练阶段不要调用这个方法**，否则 LoRA 权重被合并后就无法再训练了。

### 3.3.2 SFTTrainer + SFTConfig：训练教练和计划表

`SFTTrainer` 是训练执行器，`SFTConfig` 是训练参数的配置容器。它们来自 `trl`（Transformers Reinforcement Learning）库，不是 Unsloth 独有的，但 Unsloth 对它们做了内核优化。

```
⚠️ 最常见的坑：导入顺序！

  from unsloth import FastLanguageModel   ← 必须在 trl 之前！
  from trl import SFTTrainer, SFTConfig   ← 必须在 unsloth 之后！

  为什么？（一个比喻就懂了）

    trl 自带的 SFTTrainer 像一个「毛坯房」：
      → 不支持传 tokenizer（会报错）
      → 不会自动处理 EOS 标记（你得手动加）
      → 没有 Unsloth 的加速优化

    Unsloth 像一个「装修队」：
      → 它会在 Python 启动时拦截 SFTTrainer
      → 给它加上 tokenizer 支持、EOS 自动处理、内核加速
      → 相当于把毛坯房装修成精装房

    关键规则：Python 的 import 是「先到先得，记死不改」
      → 一个模块第一次被 import，Python 就把它记下来了
      → 后面再 import 同名模块，返回的还是第一次记住的东西

   ❌ 错误顺序（先 trl 后 unsloth）：
     ① from trl import SFTTrainer
           → Python 记下：「SFTTrainer = 毛坯房」
     ② from unsloth import FastLanguageModel
           → Unsloth 想掏出装修工具...
           → Python 说：「SFTTrainer？刚才已经登记了，就是毛坯房」
           → 你拿到的一直是毛坯房 → tokenizer 报错、没加速

   ✅ 正确顺序（先 unsloth 后 trl）：
     ① from unsloth import FastLanguageModel
           → Unsloth 在门口等着，跟 Python 说：
             「等下有人来领 SFTTrainer，先过我这一关」
     ② from trl import SFTTrainer
           → Python 正要记下毛坯房...
           → Unsloth 中途截住：「别急，我装修一下」→ 精装房交付
           → Python 记下：「SFTTrainer = 精装房」
           → 一切正常！

   一句话记忆：「先叫装修队等着，毛坯房送来现装」
             （先 import unsloth，再 import trl）

  正确写法（所有代码块都遵循这个顺序）：
    from unsloth import FastLanguageModel          # ← 第一行
    from trl import SFTTrainer, SFTConfig          # ← 第二行
    from unsloth.chat_templates import train_on_responses_only  # ← 第三行

  错误写法（会踩坑）：
    from trl import SFTTrainer, SFTConfig          # ← 先导入了 trl
    from unsloth import FastLanguageModel           # ← 装修队来晚了！

  一句话记忆：unsloth 必须在 trl 之前导入！
```

```
SFTConfig vs SFTTrainer 的关系：

  SFTConfig  → "训练计划表"：写清楚学习率、batch size、训练多少步等
  SFTTrainer → "教练"：拿着计划表，执行训练循环（前向→反向→更新权重→重复）

  创建流程：
    config = SFTConfig(学习率=..., batch_size=...)    → 先写计划
    trainer = SFTTrainer(model, config, data)          → 教练拿计划和模型
    trainer.train()                                    → 开始训练
```

#### SFTConfig：训练参数一览

```python
# ⚠️ 注意：unsloth 必须在 trl 之前导入！
from unsloth import FastLanguageModel
from trl import SFTConfig

training_args = SFTConfig(
    # === 学习率与优化器 ===
    learning_rate=2e-4,                  # 学习率。LoRA 推荐 1e-4 ~ 5e-4
    optim="adamw_8bit",                  # 优化器。Unsloth 推荐用 8-bit AdamW（省显存）

    # === Batch 与梯度 ===
    per_device_train_batch_size=2,       # 每张卡每次处理几条样本
    gradient_accumulation_steps=4,       # 梯度累积步数。有效 batch = 2×4 = 8
    max_grad_norm=1.0,                   # 梯度裁剪阈值，防止梯度爆炸

    # === 训练时长 ===
    num_train_epochs=1,                  # 训练轮数。小数据集 1~3 轮即可
    # 或用 max_steps 替代：
    # max_steps=100,                     # 训练总步数（和 epochs 二选一）

    # === 显存优化 ===
    bf16=True,                           # BF16 混合精度（A100/H100/40系/50系用 bf16；30系/V100 用 fp16）
    gradient_checkpointing=True,         # 梯度检查点（省 50% 显存，速度略降）
    gradient_checkpointing_kwargs={"use_reentrant": False},  # Unsloth 推荐设 False

    # === 日志与保存 ===
    logging_steps=10,                    # 每 10 步打印一次训练日志
    save_strategy="steps",               # 按步数保存 checkpoint
    save_steps=50,                       # 每 50 步保存一次
    save_total_limit=2,                  # 最多保留 2 个 checkpoint（省磁盘）
    output_dir="./unsloth_lora",         # 输出目录

    # === 数据相关 ===
    dataset_text_field="text",           # 数据集中文本字段的列名
    max_seq_length=2048,                 # 与 from_pretrained 的 max_seq_length 保持一致

    # === 其他 ===
    seed=42,                             # 随机种子（保证可复现）
    report_to="none",                    # 不上报到 wandb 等平台（本地训练用 none）
)
```

**核心参数详解**：

- **`learning_rate`**（推荐 `2e-4`）— 每步更新权重的幅度。LoRA 推荐 1e-4 ~ 5e-4；太大训练爆炸，太小学不动。
- **`optim`**（推荐 `"adamw_8bit"`）— 优化器算法。`adamw_8bit` = 8-bit 量化版 AdamW（来自 bitsandbytes，用 **INT8** 压缩 Adam 状态矩阵，省 50% 优化器显存）；`adamw_torch` = 标准 32-bit 版。
  > **为什么是 INT8 而不是 FP8？** 两者的策略不同：
  > - **INT8**：把数据范围均匀切成 256 个区间，每个区间一样大。数据跨度不大时，256 个格子都很细密 → 精度高。
  > - **FP8（e4m3）**：4 位指数让它的格子分布不均——靠近 0 密集，远离 0 稀疏。好处是能自适应覆盖大跨度数据（所以适合模型权重量化），坏处是稀疏区域的格子很少 → 精度不如 INT8 均匀。
  > 
  > Adam 状态（momentum / variance）被分块处理，2048 个值一组，组内数值跨度很小，INT8 的 256 个均匀格子正好发挥优势。所以这里的 INT8 不是"省事凑合"，而是"刚好适合"。
- **`per_device_train_batch_size`**（推荐 `2` 起步）— 每张卡每次处理的样本数。没有万能值，取决于模型大小 × `max_seq_length` 的总显存开销：小模型（0.5B~1.5B）可以 4~8；中等模型（3B~7B）通常 2；大模型（13B+）往往只能 1。建议从 2 开始，OOM 就降到 1，显存有余就往上加。
- **`gradient_accumulation_steps`**（推荐 `4`）— 梯度累积步数。有效 batch = per_device × accumulation，batch=2×4=8 是常见起点。
- **`num_train_epochs`**（推荐 `1`）— 训练轮数。小数据（<5k 条）1~3 轮；大数据可用 `max_steps` 替代。
- **`bf16`**（推荐 `True`，不支持就换 `fp16=True`）— 混合精度训练。绝大部分近几年的显卡都有 bf16 支持：
  - ✅ 支持 bf16：A100、H100/H800、B200、RTX 40 系及以上（4090/4080…）、RTX 50 系、**RTX 30 系（3090/3080…也能用）**
  - ❌ 不支持 bf16：V100、T4、GTX 系列等较早显卡
  - 不用纠结型号，跑一次试试：设了 `bf16=True` 能跑就用，报错就换 `fp16=True`
- **`gradient_checkpointing`**（推荐 `True`）— 梯度检查点总开关。**必须开**，配合 `use_gradient_checkpointing="unsloth"` 使用。

```
一个关键公式：有效 Batch Size

  有效 batch = per_device_train_batch_size × gradient_accumulation_steps

  为什么需要累积？
    → 显存不够放大 batch（比如 batch=8 要 16GB）
    → 解决办法：每次只算 batch=2（4GB），累积4步后才更新一次
    → 效果等价于 batch=8，但显存只用了4GB

  举例：
    batch=2, accumulate=4 → 有效batch=8（推荐起点）
    batch=4, accumulate=2 → 有效batch=8（显存够时更快）
    batch=1, accumulate=8 → 有效batch=8（显存最省，但最慢）
```

#### SFTTrainer：创建和启动训练

```python
# ⚠️ 注意：unsloth 必须在 trl 之前导入！
from unsloth import FastLanguageModel
from trl import SFTTrainer

trainer = SFTTrainer(
    model=model,                         # 插入了 LoRA 层的完整模型（原始权重 + LoRA 适配器），不只 LoRA
    tokenizer=tokenizer,                 # from_pretrained() 返回的分词器
    train_dataset=train_dataset,        
    eval_dataset=eval_dataset,           
    args=training_args,                  # SFTConfig 配置对象
)

# 启动训练
trainer.train()
```

**SFTTrainer 的参数**：

| 参数 | 填什么 | 含义 |
|---|---|---|
| `model` | `model` | 插入了 LoRA 层的完整模型（原始权重冻结 + LoRA 适配器可训练） |
| `tokenizer` | `tokenizer` | `from_pretrained()` 返回的分词器 |
| `train_dataset` | `train_dataset` | 训练集（HF Dataset 格式） |
| `eval_dataset` | `eval_dataset` | 验证集，用于评估和选择最佳 checkpoint。不传也能训，但保存的 checkpoint 是按 train loss 选的——容易选到过拟合的那个 |
| `args` | `training_args` | SFTConfig 配置对象 |

【注解】`SFTTrainer` 本身很简单——它就是一个"拿模型、拿数据、拿计划，开始训练"的封装。**真正需要你花心思的是 `SFTConfig` 的参数调优**和**数据预处理**（这些在实战步骤中会详细展开）。

### 3.3.3 train_on_responses_only：只训练回答部分

**问题场景**：微调数据格式是这样的：

```
<|im_start|>system
你是内科医生<|im_end|>
<|im_start|>user
高血压注意什么？<|im_end|>
<|im_start|>assistant
低盐饮食，规律运动...<|im_end|>   ← 我们只想训练这一段！
```

默认情况下，SFTTrainer 会把**整条文本**都算损失（包括 system 提示词和 user 问题）。但这不合理——模型不需要学习"怎么提问"，只需要学习"怎么回答"。

**train_on_responses_only 的作用**：给 instruction 部分（system + user）加"遮罩"，让损失函数只计算 assistant 回答部分的梯度。

```python
from unsloth.chat_templates import train_on_responses_only

trainer = train_on_responses_only(
    trainer,                             # 原始 SFTTrainer
    instruction_part="<|im_start|>user\n",   # 不需要计算损失的开始标记（从此处起，到 response_part 之间的 token 不计损失）
    response_part="<|im_start|>assistant\n",  # 不需要计算损失的结束标记（从此处起，恢复计算损失）
)
```

**参数详解**：

- **`trainer`** — 原始 SFTTrainer 对象，会被原地修改。
- **`instruction_part`** — 格式 `"<|im_start|>user\n"`。不需要计算损失的开始标记：从这条标记到 `response_part` 之间的文本，损失 = 0。
- **`response_part`** — 格式 `"<|im_start|>assistant\n"`。不需要计算损失的结束标记：从此处到 `<|im_end|>` 之间的文本，正常计算损失。

```
遮罩效果示意：

  原始 token 序列:  [system内容] [user内容] [assistant回答]
  默认损失计算:      ✓ 计损      ✓ 计损    ✓ 计损  ← 全算，浪费
  遮罩后损失计算:     ✗ 不计      ✗ 不计    ✓ 计损  ← 只训练回答

  为什么这样做更好？
    → 模型已经知道怎么"理解问题"，不需要再学
    → 只学"怎么回答" → 训练更高效，回答质量更高
    → 类比：你不需要教学生"怎么读题"，只需要教"怎么解题"
```

【注解】`instruction_part` 和 `response_part` 的值取决于模型的**对话模板格式**。Qwen 用 `<|im_start|>` 格式（ChatML），Llama 用 `[INST]` 格式。用错标记会导致遮罩失败。如果你用 `tokenizer.apply_chat_template()` 生成数据，格式是自动对的，直接填对应的标记即可。

## 3.4 LoRA 代码实战：分步讲解

完整的 LoRA 微调分 **5 步**。每一步先讲"做什么、为什么"，再给代码。最后把 5 步串起来就是完整脚本。

```
LoRA 微调 5 步流程：

  Step 1: 加载模型和分词器     → from_pretrained()
  Step 2: 配置 LoRA 适配器     → get_peft_model()
  Step 3: 加载并预处理数据     → load_dataset() + map() + apply_chat_template
  Step 4: 配置训练参数并训练   → SFTTrainer + train_on_responses_only + .train()
  Step 5: 保存模型             → save_pretrained()
```

---

### Step 1：加载模型和分词器

**做什么**：从本地路径或 Hugging Face Hub 加载预训练模型 + 分词器。

**为什么先加载**：微调是在已有模型基础上改，必须先拿到"原始模型"。就像改装车，得先有一辆车。

**关键参数解释**：

| 参数 | 填什么 | 含义 |
|---|---|---|
| `model_name` | 本地路径或 HF 模型名 | 从哪里加载模型 |
| `max_seq_length` | 530 | 模型能处理的最长 token 序列长度。超过会被截断 |
| `dtype` | `None` | 数据精度。`None` = Unsloth 自动选（Ampere+ GPU 选 bf16，旧卡选 fp16） |
| `load_in_4bit` | `False` | 是否 4-bit 量化加载。`False` = LoRA（BF16），`True` = QLoRA（4-bit） |

```python
# Step 1: 加载模型和分词器
from unsloth import FastLanguageModel

max_seq_length = 530   # 根据数据长度设定，太长浪费显存，太短截断回答

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="/home/hu/models/qwen3.5-0.8B",  # 本地模型路径（也可用 "Qwen/Qwen3.5-0.8B"）
    max_seq_length=max_seq_length,
    dtype=None,            # Unsloth 自动检测最佳精度
    load_in_4bit=False,    # False = LoRA（BF16 模型）；True = QLoRA（4-bit 模型）
)

# 验证加载成功
print(f"模型类型: {type(model).__name__}")      # 应输出类似 PeftModelForCausalLM
print(f"分词器词表大小: {len(tokenizer)}")       # ~151000 左右
print(f"模型参数量: {model.config.num_hidden_layers} 层")
```

【注解】`from_pretrained()` 返回**两个对象**：`model`（模型）和 `tokenizer`（分词器）。它们是一对搭档——模型吃数字（token ID），分词器负责文本和数字之间的翻译。

---

### Step 2：配置 LoRA 适配器

**做什么**：给模型挂上 LoRA 低秩旁路矩阵（A 和 B），使模型从"全参数更新"变为"只更新少量 LoRA 参数"。

**为什么**：第 1 章讲过，LoRA 的核心是 ΔW = BA。这一步就是在代码中把 A、B 矩阵"挂"到模型的指定层上。

**关键参数解释**：

| 参数 | 填什么 | 含义 | 对应第1章概念 |
|---|---|---|---|
| `r` | 8 | 低秩维度（rank），控制 LoRA 矩阵的大小 | 1.3 节的 r，r=8 → BA 参数量极少 |
| `target_modules` | `["q_proj", "k_proj", "v_proj", "o_proj"]` | 哪些层挂 LoRA | 1.3 节的标准配置（4 个注意力层） |
| `lora_alpha` | 16 | 缩放因子，实际计算 ΔW × α/r | 1.3 节的 α，alpha/r = 16/8 = 2 |
| `lora_dropout` | 0 | LoRA 输出的 dropout 比率 | 防过拟合，小模型建议 0 |
| `bias` | `"none"` | 是否训练偏置项 | 最省参数，设 none |
| `use_gradient_checkpointing` | `"unsloth"` | 梯度检查点策略 | Unsloth 特有优化，省显存 |

```python
# Step 2: 配置 LoRA 适配器
model = FastLanguageModel.get_peft_model(
    model,
    r=8,                                     
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],  # 标准配置：4个注意力层
    lora_alpha=16,                              # 缩放因子，一般取 2×r
    lora_dropout=0,                              # 小模型建议 0，大模型可设 0.05
    bias="none",                                 # 不训练偏置，最省参数
    use_gradient_checkpointing="unsloth",        # Unsloth 特有优化，省 ~50% 显存
)

# 查看可训练参数占比（验证 LoRA 确实只训练极少量参数）
trainable = sum(p.numel() for p in model.parameters() if p.requires_grad)
total = sum(p.numel() for p in model.parameters())
print(f"可训练参数: {trainable/1e6:.2f}M / 总参数: {total/1e9:.2f}B ({trainable/total*100:.2f}%)")
# 应输出类似: 可训练参数: 8.39M / 总参数: 0.82B (1.02%)
```

【注解】`get_peft_model()` 的效果：**把原始模型的指定层冻结，并在这些层旁并联小矩阵 A 和 B**。调用后 `model` 从普通模型变成了"挂了 LoRA 的模型"，只有 A/B 的参数是 `requires_grad=True`（可训练），其余全部冻结。

### Step 3：加载并预处理数据

**做什么**：把 JSONL 格式的原始数据 → 转为模型训练需要的格式化文本。

**为什么需要预处理**：模型不认识 JSON，它只认识 token 序列。我们需要把 `{instruction, input, output}` 转成 ChatML 格式的文本串，再交给分词器切成 token。

**预处理分为 3 个子步骤**：

```
数据预处理 3 步：

  3a. 加载 JSONL → Dataset 对象
  3b. 拆分训练集/验证集
  3c. instruction/input/output → ChatML 格式文本（用 apply_chat_template）
```

#### 3a. 加载 JSONL 数据

```python
# Step 3a: 加载 JSONL 文件为 Dataset 对象
from datasets import load_dataset

# split="train"：直接拿到 Dataset，省去后面写 dataset["train"] 的麻烦。
# 不加的话返回 DatasetDict({"train": Dataset(...)})，也能用，只是每一步都得加 ["train"]。
dataset = load_dataset("json", data_files="样例_内科5000-6000.jsonl", split="train")
print(f"数据集大小: {len(dataset)} 条")
print(f"数据字段: {dataset.column_names}")   # 应包含 instruction, input, output

# 看一条样本
print(dataset[0])
# {'instruction': '你是一位经验丰富的内科医生...', 'input': '我最近测血压...', 'output': '建议您低盐饮食...'}
```



#### 3b. 拆分训练集和验证集

```python
# Step 3b: 90% 训练 + 10% 验证
split = dataset.train_test_split(test_size=0.1, seed=3407)
train_dataset = split["train"]
eval_dataset = split["test"]
print(f"训练集: {len(train_dataset)} 条, 验证集: {len(eval_dataset)} 条")
```

【注解】验证集的作用：训练时定期在验证集上评估 loss，观察模型是否过拟合（训练 loss 下降但验证 loss 上升 = 过拟合信号）。`seed=3407` 确保每次拆分结果一致。

#### 3c. 格式化为 ChatML 文本

**这是最关键的预处理步骤**。把每条数据从 `{instruction, input, output}` 转成模型能直接训练的文本格式。

```python
# Step 3c: 把 instruction/input/output → messages → ChatML 格式文本
def to_text(example):
    # 1. 构建 messages 列表（3 个角色）
    msgs = [
        {"role": "system",    "content": example["instruction"]},  # 系统设定（角色描述）
        {"role": "user",      "content": example["input"]},        # 用户提问
        {"role": "assistant", "content": example["output"]},       # 期望的回答
    ]
    # 2. 用分词器的 apply_chat_template 方法格式化为 ChatML 文本
    #    tokenize=False → 返回文本字符串（不是 token ID）
    #    add_generation_prompt=False → 不追加"<|im_start|>assistant\n"（训练时不需要）
    return {"text": tokenizer.apply_chat_template(msgs, tokenize=False, add_generation_prompt=False)}

# 用 .map() 方法批量转换整个数据集
train_dataset = train_dataset.map(to_text)
eval_dataset = eval_dataset.map(to_text)

# 看一条转换后的文本
print(train_dataset[0]["text"][:300])
# 输出类似：
# <|im_start|>system
# 你是一位经验丰富的内科医生...<|im_end|>
# <|im_start|>user
# 我最近测血压有点高...<|im_end|>
# <|im_start|>assistant
# 建议您低盐饮食...<|im_end|>
```

【注解】`.map(to_text)` 对数据集中每一条样本调用 `to_text` 函数，自动添加 `"text"` 字段。SFTTrainer 训练时会读取 `"text"` 字段，分词后作为模型输入。

```
apply_chat_template() 参数详解 —— 训练预处理中最容易踩坑的地方

  这个方法的作用：把 messages 列表 → 按模型模板 → 拼成一段完整的对话文本

  核心参数（必须搞清楚）：

  ┌─────────────────────────┬──────────────────────────────────────────────────────┐
  │ 参数                     │ 说明                                                  │
  ├─────────────────────────┼──────────────────────────────────────────────────────┤
  │ tokenize=False          │ ✅ 训练时必须设 False                                  │
  │                         │                                                      │
  │   默认值 True 时：       │ → 返回 token ID 列表 [151644, 87281, ...]            │
  │                         │ → 这是一串数字，不是文本字符串                         │
  │                         │ → SFTTrainer 期望的是 "text" 字段（字符串），不是 ID   │
  │                         │ → 传 token ID 给 SFTTrainer 会报错或训练异常           │
  │                         │                                                      │
  │   设 False 时：          │ → 返回完整的文本字符串                                 │
  │                         │ → 如 "<|im_start|>system\n你是内科医生..."             │
  │                         │ → SFTTrainer 会自己负责分词（它内部有 tokenize 步骤）  │
  │                         │ → 这才是正确的训练数据格式                             │
  │                         │                                                      │
  │   一句话：               │ 训练用 False，让 SFTTrainer 自己分词；                 │
  │                         │ 推理时可设 True，直接拿到 token ID 送进 model.generate │
  ├─────────────────────────┼──────────────────────────────────────────────────────┤
  │ add_generation_prompt   │ 训练 vs 推理 用法完全不同！                             │
  │                         │                                                      │
  │   训练时必须设 False：   │                                                      │
  │     原因：训练文本已经包含 assistant 的完整回复                        │
  │           "<|im_start|>assistant\n低盐饮食...<|im_end|>"             │
  │     如果设 True → 会额外追加 "<|im_start|>assistant\n"              │
  │     → 变成两个 <|im_start|>assistant，格式重复出错                   │
  │     → 模型学到了错误的格式，推理时可能出现空回复或重复输出             │
  │                         │                                                      │
  │   推理时必须设 True：    │                                                      │
  │     原因：推理输入只有 user 的提问，没有 assistant 的回复             │
  │           "<|im_start|>user\n高血压注意什么？<|im_end|>"             │
  │     设 True → 在末尾追加 "<|im_start|>assistant\n"                 │
  │     → 告诉模型"现在轮到你说话了，从这里开始生成"                    │
  │     → 模型从这个位置开始输出下一个 token                            │
  │     如果设 False → 文本末尾没有 assistant 起始标记                   │
  │     → 模型不知道该从哪里开始生成，输出会混乱或拒绝回答               │
  │                         │                                                      │
  │   一句话：               │ 训练用 False（回复已在数据里）；                        │
  │                         │ 推理用 True（告诉模型"开始回答"）                      │
  ├─────────────────────────┼──────────────────────────────────────────────────────┤
  │ messages 格式            │ 必须按 role/content 结构                              │
  │                         │                                                      │
  │   顺序：                 │ system → user → assistant                            │
  │   role 必须是：          │ "system" / "user" / "assistant" 之一                 │
  │   content：              │ 实际的文本内容                                        │
  │                         │                                                      │
  │   ⚠️ 常见错误：          │ role 写错（如 "model" 代替 "assistant"）              │
  │                         │ → 不同模型的 role 名称可能不同，                      │
  │                         │   但 Qwen/Llama/ChatGLM 都用 "assistant"              │
  └─────────────────────────┼──────────────────────────────────────────────────────┤

  对比训练 vs 推理的输出差异：

    训练数据（add_generation_prompt=False）：
      <|im_start|>system
      你是内科医生<|im_end|>
      <|im_start|>user
      高血压注意什么？<|im_end|>
      <|im_start|>assistant
      低盐饮食，规律运动...<|im_end|>
      ← 完整对话，包含 assistant 回复，末尾是 <|im_end|>

    推理输入（add_generation_prompt=True）：
      <|im_start|>system
      你是内科医生<|im_end|>
      <|im_start|>user
      高血压注意什么？<|im_end|>
      <|im_start|>assistant
      ← 只有起始标记，没有回复内容，模型从这里开始生成

    如果训练时误设 add_generation_prompt=True：
      <|im_start|>system
      你是内科医生<|im_end|>
      <|im_start|>user
      高血压注意什么？<|im_end|>
      <|im_start|>assistant
      低盐饮食，规律运动...<|im_end|>
      <|im_start|>assistant        ← 多了一行！格式错误！
      ← 模型会学到"回复后面还有一个 assistant 标记"的错误模式
```

### Step 4：配置训练参数并训练

**做什么**：创建训练器（SFTTrainer），配置学习率、batch size 等参数，执行训练。

**关键参数解释**：

| 参数 | 填什么 | 含义 | 对应第1.6节概念 |
|---|---|---|---|
| `per_device_train_batch_size` | 8 | 每张 GPU 每次处理的样本数 | batch_size |
| `gradient_accumulation_steps` | 2 | 梯度累积步数，等效 batch = 8×2 = 16 | 增大等效 batch size |
| `learning_rate` | 2e-4 | 学习率，LoRA 用较大值 | 第1.6节推荐 1e-4~3e-4 |
| `warmup_steps` | 10 | 学习率预热步数 | 从 0 逐渐增加到目标值 |
| `num_train_epochs` | 3 | 训练轮数 | 小数据 3 轮，大数据 1-2 轮 |
| `optim` | `"adamw_8bit"` | 优化器，INT8 量化 AdamW，省 50% 优化器显存 | Unsloth 推荐 |
| `logging_steps` | 10 | 每 10 步输出一次 loss | 监控训练过程 |
| `save_steps` | 500 | 每 500 步保存 checkpoint | 防止训练中断丢失进度 |
| `eval_strategy` / `eval_steps` | `"steps"` / 500 | 每 500 步在验证集上评估 | 观察是否过拟合 |

#### 4a. 创建 SFTTrainer

```python
# Step 4a: 创建训练器
# ⚠️ unsloth 已在 Step 1 中导入（必须在 trl 之前！）
from trl import SFTTrainer, SFTConfig

trainer = SFTTrainer(
    model=model,
    train_dataset=train_dataset,
    eval_dataset=eval_dataset,
    tokenizer=tokenizer,
    args=SFTConfig(
        max_seq_length=max_seq_length,
        per_device_train_batch_size=8,
        gradient_accumulation_steps=2,
        warmup_steps=10,
        num_train_epochs=3,
        learning_rate=2e-4,
        logging_steps=10,
        save_steps=500,
        eval_strategy="steps",
        eval_steps=500,
        output_dir="./unsloth_lora",    # 训练器自动保存 checkpoint 的根目录（与 save_pretrained 无关，见 Step 5）
        optim="adamw_8bit",             # 8-bit 优化器，省显存
        seed=3407,
    ),
)
```

#### 4b. 只对 assistant 回复计算 loss

**为什么**：训练文本包含 system、user、assistant 三部分，但模型**只需要学会生成 assistant 部分**。system 和 user 是已知条件，不应该让模型去"预测"它们。

```python
# Step 4b: 让 loss 只计算 assistant 回复部分（mask掉 system/user）
from unsloth.chat_templates import train_on_responses_only

trainer = train_on_responses_only(
    trainer,
    instruction_part="<|im_start|>user",      # 不需要计算损失的开始标记（user/system 部分 loss 被 mask）
    response_part="<|im_start|>assistant",    # 不需要计算损失的结束标记（assistant 部分开始计算 loss）
)
```

【注解】`train_on_responses_only` 的原理：在 ChatML 格式中，`<|im_start|>user` 到 `<|im_start|>assistant` 之间的 token 被标记为"不需要预测"，loss 设为 0；只有 `<|im_start|>assistant` 之后的 token 才计算 loss。这和 LLaMA Factory WebUI 中自动做的 loss mask 是同一件事，只是在这里用代码显式指定。

#### 4c. 开始训练

```python
# Step 4c: 启动训练
trainer.train()
print("Unsloth LoRA 训练完成！")
```

训练过程中会输出类似日志：

```

[Step 20] train_loss=1.834, lr=1.95e-4
...
[Step 500] train_loss=0.512, eval_loss=0.623  ← 验证集 loss，观察是否过拟合
```

## 3.5 QLoRA 代码实战：分步讲解

QLoRA 和 LoRA 的**唯一区别在 Step 1**：加载模型时 `load_in_4bit=True`。其余步骤完全一样。

```
LoRA vs QLoRA 的代码差异：

  Step 1 加载模型:
    LoRA:  load_in_4bit=False    → 模型以 BF16 加载（~1.6GB / 0.8B）
    QLoRA: load_in_4bit=True     → 模型以 4-bit NF4 加载（~0.4GB / 0.8B）

  Step 2~5: 完全一样，代码不用改
```

### Step 1（QLoRA 版）：加载 4-bit 量化模型

```python
# QLoRA Step 1: 加载 4-bit 量化模型
from unsloth import FastLanguageModel

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="/home/hu/models/qwen3.5-0.8B",
    max_seq_length=2048,
    dtype=None,            # Unsloth 自动检测最佳精度
    load_in_4bit=True,     # ← 唯一区别：4-bit 量化加载
)

print(f"QLoRA 模型加载完成")
print(f"基础模型精度: 4-bit NF4（~0.4GB / 0.8B）")
print(f"LoRA 适配器精度: FP16（~2MB）")
print(f"总显存估算: ~0.6GB")
```

### Steps 2~5：和 LoRA 完全一样

直接复用 3.4 节 Step 2~5 的代码，**不需要任何修改**。唯一可选的调整是 rank 可以稍微大一点（QLoRA 基础模型更小，LoRA 参数多一点不会撑爆显存）：

```python
# QLoRA Step 2（可选调大 rank）
model = FastLanguageModel.get_peft_model(
    model,
    r=16,                                      # QLoRA 可以用 r=16（显存更充裕）
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    lora_alpha=32,                              # alpha = 2×r = 32
    lora_dropout=0,
    bias="none",
    use_gradient_checkpointing="unsloth",
)
```

## 3.6 多任务微调代码实战：分步讲解

本节演示**共享 LoRA** 策略：所有任务数据混在一起，训练同一组 LoRA 适配器。

与单任务微调的区别只有两点：① `target_modules` 可以多挂 FFN 层；② 数据是多个任务的混合 JSONL。其余流程一样。

### Step 1：加载模型（同 3.4）

```python
from unsloth import FastLanguageModel

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="Qwen/Qwen3.5-0.8B",
    max_seq_length=1024,
    dtype=None,
    load_in_4bit=False,
)
```

### Step 2：配置 LoRA（多任务建议多挂层）

```python
# 多任务微调建议挂更多层（激进配置），让模型有更大容量学习不同任务
model = FastLanguageModel.get_peft_model(
    model,
    r=16,
    lora_alpha=32,
    lora_dropout=0,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                    "gate_proj", "up_proj", "down_proj"],  # 激进配置：7层全挂
    bias="none",
    use_gradient_checkpointing="unsloth",
    random_state=3407,
)
```

### Step 3：加载混合数据（和单任务一样，只是 JSONL 包含多种任务）

```python
from datasets import load_dataset

# JSONL 中已包含多种任务的混合数据（instruction 里标记了任务类型）
dataset = load_dataset("json", data_files="multi_task_data.jsonl", split="train")

split = dataset.train_test_split(test_size=0.1, seed=3407)
train_dataset = split["train"]
eval_dataset = split["test"]

def to_text(example):
    msgs = [
        {"role": "system",    "content": example["instruction"]},
        {"role": "user",      "content": example["input"]},
        {"role": "assistant", "content": example["output"]},
    ]
    return {"text": tokenizer.apply_chat_template(msgs, tokenize=False, add_generation_prompt=False)}

train_dataset = train_dataset.map(to_text)
eval_dataset = eval_dataset.map(to_text)
```

### Step 4：训练（学习率可略低于单任务）

```python

from unsloth.chat_templates import train_on_responses_only

    model=model,
    train_dataset=train_dataset,
    eval_dataset=eval_dataset,
    tokenizer=tokenizer,
    args=SFTConfig(
        max_seq_length=1024,
        per_device_train_batch_size=4,
        gradient_accumulation_steps=4,
        warmup_steps=10,
        num_train_epochs=3,
        learning_rate=5e-5,
        logging_steps=20,
        save_steps=200,
        save_total_limit=2,
        eval_strategy="steps",
        eval_steps=200,
        output_dir="./multi_task_lora",
        optim="adamw_8bit",
        seed=3407,
    ),
)

    trainer,
    instruction_part="<|im_start|>user",
    response_part="<|im_start|>assistant",
)

trainer.train()
```


多任务微调（共享 LoRA）的核心技巧：
1. **instruction 中标记任务类型**（如 `#医学问答`、`#代码生成`）——让模型知道当前在做什么任务

## 3.7 AdaLoRA 代码实战

> ⚠️ **注意**：AdaLoRA 目前**仅支持原生 PEFT 库**直接调用，LLaMA Factory 和 Unsloth 均未内置 AdaLoRA 支持。本节用原生 PEFT 演示，帮你理解 AdaLoRA 的配置与训练流程。

### Step 1：用原生 Transformers 加载模型

```python
# AdaLoRA 不走 Unsloth，用原生 transformers + peft
from transformers import AutoModelForCausalLM
import torch

model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen3.5-0.8B",
    device_map="auto",
    torch_dtype=torch.bfloat16,   # Ampere+ 用 bf16 更稳定
)
```

### Step 2：配置 AdaLoRA

**和 LoRA 的关键区别**：

| 参数 | LoRA | AdaLoRA | 含义 |
|---|---|---|---|
| rank | `r=8`（固定） | `init_r=8`（初始）+ `target_r=4`（最终裁剪目标） | AdaLoRA 的 rank 会训练中动态裁剪 |
| 正交约束 | 无 | `orth_reg_weight=0.5` | 1.5 节提过 P^T·P = Q·Q^T = I 这个理想条件，但它不会自动成立——`orth_reg_weight` 就是用来罚"不满足这个条件"的，值越大罚得越狠。0.5 是经验默认值 |
| 总步数 | 不需要 | `total_step=1000` | 重要性裁剪是渐进的（不是一步到位），AdaLoRA 需要知道总共训多少步才能规划"从第几步开始裁、每步裁多少"。写实际总步数，偏差太大会导致裁剪节奏不对 |

```python
from peft import AdaLoraConfig, get_peft_model

ada_config = AdaLoraConfig(
    init_r=8,                      # 初始 rank（AdaLoRA 用 init_r，不是 r）
    lora_alpha=16,
    target_r=4,                    # 最终保留的平均 rank（训练中从 init_r 裁剪到 target_r）
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                    "gate_proj", "up_proj", "down_proj"],
    lora_dropout=0.0,
    bias="none",
    task_type="CAUSAL_LM",
    init_lora_weights=True,
    orth_reg_weight=0.5,           # 正交正则化权重：罚 P^T·P≠I 或 Q·Q^T≠I 的程度。越大 P/Q 越"正交"，Λ 对角元素越能代表真实重要性
    total_step=1000,               # 总训练步数：AdaLoRA 的裁剪不是一步到位，而是渐进式——需要知道总步数来规划"第几步开始裁、每步裁多少"，所以必须填实际值
    beta1=0.85, beta2=0.85,        # 重要性分数的移动平均参数
)

ada_model = get_peft_model(model, ada_config)
```

### Step 3：查看参数

```python
# 查看可训练参数（初始和 LoRA 一样，训练后才裁剪）
total = sum(p.numel() for p in model.parameters())
trainable = sum(p.numel() for p in ada_model.parameters() if p.requires_grad)
print(f"模型总参数: {total/1e9:.2f}B")
print(f"AdaLoRA 可训练参数（初始）: {trainable/1e6:.2f}M")
print(f"训练完成后实际保留: ~{trainable * 4/8 / 1e6:.1f}M（target_r=4 裁剪后）")
```

> **注意**：AdaLoRA 目前依赖 peft 库原生实现。LLaMA Factory 和 Unsloth 均未内置 AdaLoRA 支持（`FastLanguageModel.get_peft_model` 不支持 AdaLoRA）。日常微调**优先用 LoRA/QLoRA**（LLaMA Factory / Unsloth 都支持），对效果有极致追求时再考虑用原生 PEFT 调 AdaLoRA。

### Step 2：切换到推理模式

```python
# Step 3: 用微调后的模型回答问题
```

## 3.9 显存管理与防 OOM 实战指南

> OOM（Out of Memory）是微调实战中最常见的"拦路虎"。本章前面的代码都假设你能跑起来，但现实中 6GB 显存的显卡随时可能崩。本节系统讲清：**显存怎么花出去的、每个参数对显存的影响、遇到 OOM 怎么逐级降配**。

---

### 3.9.1 显存去哪了？——四大开销拆解

训练一个大模型，显存被五部分吃掉。**注意：LoRA 微调和全量微调的开销结构完全不同**——LoRA 只训练 0.8% 的参数，所以梯度+优化器几乎可忽略，大头是模型权重和激活值/KV缓存：

```
LoRA 微调显存开销 = 模型权重 + 激活值/KV缓存 + LoRA适配器 + LoRA优化器 + LoRA梯度

  ┌─────────────────────────────────────────────────────────────┐
  │           LoRA 微调 vs 全量微调的显存结构对比                 │
  │                                                             │
  │  全量微调（所有参数都训练）：                                  │
  │    模型权重    ≈ 25%                                         │
  │    优化器状态  ≈ 50% ← 大头！0.8B参数×8bytes=~6.4GB           │
  │    梯度        ≈ 25% ← 也很大！0.8B参数×4bytes=~3.2GB         │
  │    激活值      ≈ 额外开销                                    │
  │                                                             │
  │  LoRA 微调（只训练 ~0.8% 参数）：                             │
  │    模型权重（冻结的基础模型）  ≈ 35~45% ← 最大头！              │
  │      0.8B × BF16 = 1.6GB，不可压缩除非量化                   │
  │    激活值 + KV 缓存          ≈ 30~40% ← 第二大头！             │
  │      batch × seq_len × hidden × layers → 随batch/seq线性增长 │
  │      gradient_checkpointing 可砍掉50%                        │
  │    LoRA 适配器权重（A/B矩阵） ≈ 0.1% ← 极小！                 │
  │      rank=16 × 7模块 ≈ 1MB                                   │
  │    LoRA 优化器状态           ≈ 0.1% ← 极小！                   │
  │      AdamW 8-bit: ~3.5MB                                    │
  │    LoRA 梯度                 ≈ 0.1% ← 极小！                   │
  │      ~1-2MB                                                  │
  └─────────────────────────────────────────────────────────────┘
```

**各部分详解**：

| 开销 | LoRA 时占比 | 含义 | 谁决定的 | 能不能省 |
|---|---|---|---|---|
| **模型权重（冻结）** | **35~45%** | 基础模型的全部参数。LoRA 时不动这些参数，但它们依然占显存 | 模型大小、精度（BF16 vs 4-bit） | **量化最有效**：BF16→4-bit 砍掉 70% |
| **激活值 + KV 缓存** | **30~40%** | 前向传播的中间结果（含每层 attention 的 Key/Value），反向传播要用 | **batch_size × max_seq_length** | gradient_checkpointing 砍 50%；减 batch/seq 也有效 |
| **LoRA 适配器权重** | **≈0.1%** | 可训练的 A/B 矩阵，实际只有 ~1MB | LoRA rank、target_modules 数量 | 极小，减 rank 省不了多少 |
| **LoRA 优化器状态** | **≈0.1%** | AdamW 对 LoRA 参数存的动量/方差副本 | LoRA rank → 可训练参数量 | 极小，adamw_8bit 省不了多少 |
| **LoRA 梯度** | **≈0.1%** | 反向传播对 LoRA 参数算出的梯度 | LoRA rank → 可训练参数量 | 极小 |

```
关键认知：LoRA 微调的显存大头是模型权重和 KV/激活值

  全量微调时：优化器+梯度是大头（占 75%）→ 减参数就省显存
  LoRA 微调时：优化器+梯度几乎可忽略（占 <1%）→ 大头变成：

    ① 模型权重（1.6GB / BF16）→ 量化（QLoRA）砍到 0.5GB，省最多
    ② 激活值+KV缓存 → 随 batch×seq 线性增长，减batch/seq最有效

  → 不要指望"减 LoRA rank"来省显存！rank=8和rank=32的显存差距只有几MB
  → 真正有效的省显存手段：量化、梯度检查点、减 batch、减 seq_length
```

### 3.9.2 显存估算速查表

以 Qwen3.5-0.8B 为例，不同配置下的预估显存：

```
Qwen3.5-0.8B LoRA 微调显存估算（batch=2, seq_len=2048, rank=16）：

  ┌─────────────┬──────────────┬──────────────┬─────────────┐
  │ 配置         │ 模型占用      │ 训练总占用    │ 推荐显卡     │
  ├─────────────┼──────────────┼──────────────┼─────────────┤
  │ LoRA BF16   │ ~1.6 GB      │ ~5~6 GB      │ 6GB+ (3060) │
  │ LoRA BF16   │ ~1.6 GB      │ ~7~8 GB      │ 8GB+ (3070) │  ← batch=4
  │ QLoRA 4-bit │ ~0.5 GB      │ ~3~4 GB      │ 4GB+ (任意) │
  │ QLoRA 4-bit │ ~0.5 GB      │ ~5~6 GB      │ 6GB+ (3060) │  ← batch=4
  │ LoRA 7B模型 │ ~14 GB       │ ~18~20 GB    │ 24GB(3090)  │
  │ QLoRA 7B    │ ~4 GB        │ ~10~12 GB    │ 12GB(3080)  │
  └─────────────┴──────────────┴──────────────┴─────────────┘

  注意：以上是粗略估算，实际显存还受 gradient_checkpointing、
  数据长度分布等影响。以下公式可用于自行估算：
```

```
自行估算公式（LoRA，0.8B 模型）：

  模型权重（BF16）：
    0.8B × 2 bytes = 1.6 GB

  LoRA 可训练参数（rank=16, 7个target_modules）：
    7 × (4096×16 + 16×4096) × 2 bytes ≈ 1.8 MB（极小，几乎可忽略）

  优化器状态（32-bit AdamW）：
    可训练参数 × 8 bytes ≈ 7.2 MB（极小）

  优化器状态（8-bit AdamW ← Unsloth 推荐）：
    可训练参数 × 2 bytes ≈ 1.8 MB（更小！）

  激活值（大头）：
    batch × seq_len × hidden_dim × layers × 2 bytes
    = 2 × 2048 × 4096 × 24 × 2 ≈ 0.75 GB（单层）
    全部层 ≈ 2~3 GB（开启 gradient_checkpointing 后降至 ~1 GB）

  → 总训练显存 ≈ 1.6(权重) + 0(LoRA极小) + 1(激活) + 框架开销 ≈ 5~6 GB
```

### 3.9.3 六个参数，逐级省显存

遇到 OOM 时，按以下优先级依次调整（效果从大到小排列）：

```
省显存优先级（从最有效到最微弱）：

  ① load_in_4bit=True                          → 砍掉 60~70% 模型权重显存（效果最大）
  ② use_gradient_checkpointing="unsloth"       → 激活值智能卸载到CPU，再省30%（效果很大）
  ③ per_device_train_batch_size                → 直接影响激活值（效果明显）
  ④ max_seq_length                             → 直接影响激活值（效果明显）
  ⑤ optim="adamw_8bit"                         → 砍掉 75% 优化器显存（LoRA时效果一般，因为可训练参数本来就少）
  ⑥ LoRA rank                                  → 影响优化器+梯度（LoRA时效果微弱，因为可训练参数比例极小）
```

#### ① `load_in_4bit=True`：从 LoRA 切 QLoRA（效果最大）

```
切换前后对比：

  LoRA (load_in_4bit=False)：模型以 BF16 加载，0.8B 模型占 ~1.6 GB
  QLoRA (load_in_4bit=True) ：模型以 4-bit NF4 加载，0.8B 模型占 ~0.5 GB

  省了 1.1 GB！这是单个参数能省最多的。

  代价：训练精度略有下降（4-bit量化有信息损失），但实际效果差距很小（<1%）
  代码改动：只需要改一行！其他步骤完全不变
```

```python
# LoRA → QLoRA：只改这一行
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="Qwen/Qwen3.5-0.8B",
    max_seq_length=2048,
    dtype=None,
    load_in_4bit=True,    # ← 把 False 改成 True，就从 LoRA 变成 QLoRA
)
# Step 2~5 完全不变！
```

#### ② `use_gradient_checkpointing="unsloth"`：最省显存（效果很大）

> 原理和三种模式对比已在 3.3.1 的参数表中讲过，这里只讲**实战怎么配**。

```
实战要点（记住这两条就够了）：

  1. 在 get_peft_model() 里设 use_gradient_checkpointing="unsloth"
     → 这是 Unsloth 最省显存的模式，比标准 True 多省 30%，速度损失只有 ~2%

  2. 在 SFTConfig 里也要设 gradient_checkpointing=True + use_reentrant=False
     → 这两行是"开关"，不设的话 get_peft_model 的配置不生效
     → use_reentrant=False 必须配，否则和 Unsloth 内核冲突会报错
```

```python
# Step 2: get_peft_model 中开启 "unsloth" 模式
model = FastLanguageModel.get_peft_model(
    model,
    r=16,
    use_gradient_checkpointing="unsloth",  # ← 关键！最省显存
    # ... 其他参数
)

# SFTConfig 中也要开启（配合开关）
training_args = SFTConfig(
    gradient_checkpointing=True,                    # ← 必开（配合 get_peft_model）
    gradient_checkpointing_kwargs={"use_reentrant": False},  # ← 必配！
    # ... 其他参数
)
```

【注解】这两个地方要**同时设**：`get_peft_model` 里的 `"unsloth"` 决定用什么策略，`SFTConfig` 里的 `gradient_checkpointing=True` 是总开关。只设一个不行——光开 SFTConfig 的开关但没指定 `"unsloth"`，用的就是标准模式（省得少、慢得多）。`use_reentrant=False` 是安全选项，和 Unsloth 内核兼容，**不要漏掉**。

#### ③ `per_device_train_batch_size`：缩小批次（效果明显）

```
batch_size 对显存的影响：

  batch=1 → 激活值最小，但训练不稳定（梯度噪声大）
  batch=2 → 常见起点，6GB卡能跑
  batch=4 → 更稳定，需要 8~12GB 显存
  batch=8 → 效果最好，需要 16~24GB 显存

  有效batch不变的办法：
    batch=2, accumulate=4 → 有效batch=8, 显存占用=batch=2
    比 batch=8, accumulate=1 → 有效batch=8, 显存占用=batch=8 省很多！
```

```python
# OOM 时：缩小 batch，增大 accumulate，保持有效 batch 不变
training_args = SFTConfig(
    per_device_train_batch_size=1,     # ← 从2降到1（省一半激活值）
    gradient_accumulation_steps=8,     # ← 从4升到8（有效batch = 1×8 = 8不变）
    # ... 其他参数
)
```

#### ④ `max_seq_length`：缩短序列长度（效果明显）

```
seq_length 对显存的影响：

  max_seq_length=2048 → 激活值是 512 的 4倍
  max_seq_length=1024 → 激活值是 512 的 2倍
  max_seq_length=512  → 最省显存，但长对话会被截断

  建议：
    先统计你的数据长度分布，选 90% 数据能覆盖的长度
    不要盲目设 2048——如果你的数据平均只有 200 token，512 就够了
```

```python
# Step 1 加载时设定合适的 seq_length
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="Qwen/Qwen3.5-0.8B",
    max_seq_length=512,    # ← 从2048降到512（省75%激活值）
    # ... 其他参数
)

# SFTConfig 也要同步修改
training_args = SFTConfig(
    max_seq_length=512,    # ← 必须和 from_pretrained 保持一致！
    # ... 其他参数
)
```

【注解】`max_seq_length` 在 `from_pretrained()` 和 `SFTConfig` 中**必须保持一致**，否则会报错或浪费显存。两个地方都要改。

#### ⑤ `optim="adamw_8bit"`：8-bit 优化器（LoRA 时效果一般，但精度几乎无损）

```
8-bit AdamW 的原理（通俗版）：

  AdamW 优化器要为每个可训练参数存两个"账本"：
    动量（m）：记录参数的历史更新方向（"之前往哪走"）
    方差（v）：记录更新幅度的波动大小（"走的步子有多大"）

  标准 AdamW（32-bit）：
    每个账本用 float32 存储 → 可训练参数 × 2 × 4 bytes = × 8 bytes

  8-bit AdamW：
    每个账本用 int8 量化存储 → 可训练参数 × 2 × 1 byte = × 2 bytes
    → 省 75% 优化器显存

  为什么精度几乎无损？
    → 量化只压缩"账本"，不压缩模型权重本身
    → 模型权重和梯度仍然是 float32/BF16，计算精度不受影响
    → 动量和方差只是"辅助记录"，用 8-bit 足够保留关键信息
    → 实测：训练 loss 曲线和最终效果与 32-bit 几乎无差异
    → 类比：用粗一点的本子记账，不影响算账结果的准确性

  LoRA 时的实际效果：
    可训练参数极少（~1.8MB）：
      标准：1.8MB × 8 = ~14MB
      8-bit：1.8MB × 2 = ~3.6MB
      → 省了 ~10MB，对总量 5~6GB 微不足道

    全量微调时可训练参数多 → 这招省几GB，很重要
    → 结论：LoRA 时开着就行，反正不亏；但不指望它解决 OOM
```

```python
training_args = SFTConfig(
    optim="adamw_8bit",    # ← 推荐！省优化器显存，精度几乎无损
    # ... 其他参数
)
```

#### ⑥ `r`（LoRA rank）：减小秩（效果微弱）

```
rank 对显存的影响（0.8B 模型，7个 target_modules）：

  rank=16 → 可训练参数 ≈ 0.9MB → 优化器+梯度 ≈ ~7MB
  rank=8  → 可训练参数 ≈ 0.45MB → 优化器+梯度 ≈ ~3.5MB
  rank=32 → 可训练参数 ≈ 1.8MB → 优化器+梯度 ≈ ~14MB

  → 省的量极小（几MB），对 5~6GB 总量几乎无影响
  → 减 rank 主要影响微调效果，不太影响显存
  → 结论：不要为了省显存去减 rank，效果损失远大于显存收益
```

```
六招总结——遇到 OOM 的决策流程：

  Step 1: 开了 use_gradient_checkpointing="unsloth" 和 adamw_8bit 吗？→ 没开就先开
  Step 2: 还 OOM？→ 切 QLoRA (load_in_4bit=True)
  Step 3: 还 OOM？→ 减 batch_size 到 1，增大 gradient_accumulation
  Step 4: 还 OOM？→ 减 max_seq_length（先统计数据长度，选合理的值）
  Step 5: 还 OOM？→ 换更小的模型（0.8B → 没更小的了，该换显卡了）
  Step 6: 以上全做了还 OOM → 显卡不够，至少需要 6GB 显存跑 0.8B QLoRA
```

### 3.9.5 常见 OOM 报错与对应解法

```
OOM 报错类型与解法速查：

  报错1: CUDA out of memory. Tried to allocate XXX MiB
    → 最常见的 OOM，训练 step 时爆了
    → 解法：按 3.9.3 的六招决策流程逐级降配

  报错2: CUDA out of memory during model loading
    → 模型还没训练就爆了，连加载都装不下
    → 解法：切 QLoRA (load_in_4bit=True)，或换更小模型

  报错3: Out of memory during generation/inference
    → 推理时爆了（max_new_tokens 太长）
    → 解法：减小 max_new_tokens，或减少同时推理的 batch

  报错4: RuntimeError: Expected all tensors on the same device
    → 不是 OOM，但常和显存问题一起出现
    → 原因：部分数据在 CPU，部分在 GPU（模型太大自动卸载到 CPU）
    → 解法：确保 load_in_4bit=True 且 device_map="auto" 正确配置

  报错5: Kill (进程被系统杀掉，没有 Python 报错)
    → 不是 GPU OOM，而是 CPU/系统内存 OOM
    → 原因：数据集太大，map() 预处理时把系统内存吃光了
    → 解法：分批处理数据集，或用 streaming 模式加载
```


### 3.9.6 不同显卡的推荐配置

```
常见显卡 × 推荐配置速查：

  ┌────────────┬──────────┬─────────────────────────────────────────────┐
  │ 显卡        │ 显存      │ 推荐配置                                     │
  ├────────────┼──────────┼─────────────────────────────────────────────┤
  │ RTX 3060   │ 6 GB     │ QLoRA, batch=2, seq=512, grad_ckpt=True     │
  │            │          │ 或 LoRA, batch=1, seq=512, grad_ckpt=True   │
  ├────────────┼──────────┼─────────────────────────────────────────────┤
  │ RTX 3070   │ 8 GB     │ QLoRA, batch=4, seq=1024, grad_ckpt=True    │
  │            │          │ 或 LoRA, batch=2, seq=1024, grad_ckpt=True  │
  ├────────────┼──────────┼─────────────────────────────────────────────┤
  │ RTX 3080   │ 12 GB    │ LoRA, batch=4, seq=2048, grad_ckpt=True     │
  │            │          │ QLoRA 7B, batch=2, seq=1024                  │
  ├────────────┼──────────┼─────────────────────────────────────────────┤
  │ RTX 3090   │ 24 GB    │ LoRA 7B, batch=4, seq=2048, grad_ckpt=True  │
  │            │          │ QLoRA 14B, batch=2, seq=1024                 │
  ├────────────┼──────────┼─────────────────────────────────────────────┤
  │ A100       │ 40/80 GB │ 随便配，几乎不会 OOM                           │
  └────────────┴──────────┴─────────────────────────────────────────────┘

  基本原则：
    6GB → 只能跑 0.8B QLoRA，参数要抠
    8GB → 0.8B LoRA 或 QLoRA 舒适，7B 只能 QLoRA+小batch
    12GB+ → 0.8B 很舒适，7B LoRA 可跑
    24GB+ → 7B LoRA 舒适，14B QLoRA 可跑
```

【注解】以上是**0.8B 模型**的参考配置。模型越大，每个档位都要相应降配。核心原则：**先从保守配置开始（小batch、短seq、开ckpt），确认能跑后再逐步加大**——别上来就 batch=4 seq=2048，大概率 OOM。

## 5.4 方案 B：用 Unsloth 微调（推荐性能优化）

```python
# 完整的客服模型微调脚本（Unsloth 官方写法）
# pip install unsloth trl datasets

from unsloth import FastLanguageModel
from trl import SFTTrainer, SFTConfig
from unsloth.chat_templates import train_on_responses_only
from datasets import load_dataset
import torch

# 1. 加载基础模型
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="Qwen/Qwen3.5-0.8B",
    max_seq_length=1024,
    dtype=None,           # Unsloth 自动检测最佳 dtype
    load_in_4bit=False,
)

# 2. 配置 LoRA
model = FastLanguageModel.get_peft_model(
    model,
    r=16,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    lora_alpha=32,
    lora_dropout=0,
    bias="none",
    use_gradient_checkpointing="unsloth",
)

# 3. 加载 JSONL 数据 → messages → text
dataset = load_dataset("json", data_files="客服数据.jsonl", split="train")

# 拆分训练集和验证集（90%训练 + 10%验证）
split = dataset.train_test_split(test_size=0.1, seed=3407)
train_dataset = split["train"]
eval_dataset = split["test"]

def to_text(example):
    msgs = [
        {"role": "system",    "content": example["instruction"]},
        {"role": "user",      "content": example["input"]},
        {"role": "assistant", "content": example["output"]},
    ]
    return {"text": tokenizer.apply_chat_template(msgs, tokenize=False, add_generation_prompt=False)}

train_dataset = train_dataset.map(to_text)
eval_dataset = eval_dataset.map(to_text)

# 4. 训练
trainer = SFTTrainer(
    model=model,
    train_dataset=train_dataset,
    eval_dataset=eval_dataset,
    tokenizer=tokenizer,
    args=SFTConfig(
        max_seq_length=1024,
        per_device_train_batch_size=4,
        gradient_accumulation_steps=4,
        warmup_steps=10,
        num_train_epochs=5,
        learning_rate=2e-4,
        logging_steps=10,
        save_steps=100,
        save_total_limit=2,
        eval_strategy="steps",
        eval_steps=100,
        output_dir="./客服模型_lora",
        optim="adamw_8bit",
        seed=3407,
    ),
)

# loss 只算 assistant 回复，mask 掉 system/user
trainer = train_on_responses_only(
    trainer,
    instruction_part="<|im_start|>user",
    response_part="<|im_start|>assistant",
)

print("开始微调客服模型...")
trainable = sum(p.numel() for p in model.parameters() if p.requires_grad)
print(f"LoRA rank: 16, 可训练参数: {trainable/1e6:.2f}M")

trainer.train()

# 5. 保存
model.save_pretrained("./客服模型_lora/checkpoint-final")
tokenizer.save_pretrained("./客服模型_lora/checkpoint-final")
print("客服模型微调完成！")
```

```python
def generate_answer(model, tokenizer, question):

    text = tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)
    inputs = tokenizer([text], return_tensors="pt").to(model.device)

    return tokenizer.decode(outputs[0], skip_special_tokens=True)
```
