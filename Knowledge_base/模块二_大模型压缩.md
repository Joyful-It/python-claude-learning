## 1.1 为什么需要模型压缩？

在大模型时代，"参数越多能力越强"是共识。但部署一个旗舰大模型的成本极其高昂：


- **Qwen3.5-397B-A17B**：FP16 需要 **794GB** 显存 → 至少 8×H100（¥500万+）
- **DeepSeek-V4-Pro**：1.6T 参数 → 需要 16×H100 集群
- **Qwen3.5-35B-A3B**（社区最爱的 MoE）：FP16 也要 **70GB** 显存 → 一张 H100 才 80GB 刚够


模型压缩就是要回答一个问题：**如何在尽量不损失能力的前提下，把模型塞进更少的显存里？**


【注解】这不只是"省钱"的问题。更小的模型意味着更低的延迟、更高的吞吐、更低的功耗。在移动端、IoT、边缘设备上，不压缩根本跑不动。

## 1.2 数值精度与量化原理

深度学习模型默认使用 **FP32（单精度浮点数，32位）** 训练，推理时会降为 **FP16/BF16**。


**不同精度的存储对比**：


|精度|位数|显存（以 9B 模型为例）|特点|
|---|---|---|---|
|FP32|32 bits|~36 GB|训练标准精度|
|FP16 / BF16|16 bits|~18 GB|推理默认|
|**FP8**|8 bits|**~9 GB**|**生产落地标准（H100 原生支持）**|
|**INT4**|4 bits|**~5 GB**|个人玩家最爱（消费级显卡）|
|**INT3 / INT2**|3 / 2 bits|~3.4 / ~2.25 GB|极端压缩，质量损失大|

【注解】**FP8** 是 2024-2026 年生产落地的标准精度。NVIDIA H100/H200 Tensor Core 原生支持 FP8 计算（速度是 FP16 的 2 倍）。vLLM 等框架的 `--quantization fp8` 就是做这个。


【注解】**INT4** 是个人玩家在 RTX 4090（24GB）上跑大模型的方案。Qwen3.5-35B-A3B 在 INT4 下约 18GB，正好塞进一张 4090。但 INT4 需要 H100 不支持原生 INT4 计算，实际通过 AWQ/GPTQ 实现。


### 量化原理：从数学到实践

#### 1. 量化的基本数学：scale 与 zero_point

量化的本质是**把连续的浮点数范围"切"成离散的整数格子**，核心要算两个参数：

- **scale（缩放因子）**：每个整数格子代表的浮点数"宽度"
- **zero_point（零点）**：哪个浮点数对应整数 0

##### 对称量化 vs 非对称量化

**对称量化**（常用在权重，FP16 → INT4）：
```
FP16 权重范围: [-3.2, 3.2]  （实际模型权重通常在 [-5, 5] 之间）
INT4 范围: [-8, 7]          （4-bit 有符号整数）
映射公式: INT4 = round(FP16 / scale)
其中 scale = max(|FP16|) / 8 = 3.2 / 8 = 0.4
特点: zero_point = 0（关于原点对称），计算快但范围可能浪费
反量化: FP16 ≈ INT4 × scale
```

**非对称量化**（常用在激活值，FP16 → UINT8）：
```
FP16 激活范围: [0.0, 6.3]   （ReLU/GELU 激活后通常非负）
UINT8 范围: [0, 255]
映射公式: UINT8 = round((FP16 - min) / scale)
其中 scale = (max - min) / 255 = 6.3 / 255 ≈ 0.0247
      zero_point = round(-min / scale) = 0
特点: zero_point = 0（这里刚好从 0 开始），范围用满，精度更高
反量化: FP16 ≈ UINT8 × scale + min
```

【注解】为什么用 FP16 举例？现在主流大模型都是 FP16/BF16 训练和存储，几乎没有 FP32 部署的场景。量化就是把 FP16/BF16 压缩到更低位宽（INT4/INT8/FP8）。

#### 2. 量化的三大主流做法（按"什么时候做量化"分）

想象量化就像"压缩文件"——可以压原始文件，也可以边写边压：

| 做法 | 什么时候压 | 类比 | 核心特点 | 代表方案 | 适用场景 |
|------|-----------|------|---------|---------|---------|
| **PTQ（训练后量化）** | 训练完再压 | 先拍照后修图 | 不用重训，几小时搞定 | AWQ、GPTQ、llama.cpp (k-quant)、FP8 静态 | ⭐ 主流 |
| **QAT（量化感知训练）** | 训练时模拟量化 | 边拍照边修图 | 精度高，但要重训 | LLM-QAT | 移动端 NPU、2-bit 极端压缩 |
| **QLoRA 量化加载** | 加载时量化 | 打开时压缩 | 加载省显存，训练时反量化 | bitsandbytes NF4 | 消费级 GPU 微调 |

【注解】为什么 PTQ 是绝对主流？因为训练一个 70B 模型要花几百万美元算力、几个月时间。PTQ 不需要重新训练，只需要几百条校准数据（从训练集随机抽）+ 几小时 GPU 时间，就能把 FP16 模型压到 INT4。

#### 3. 不是所有层都量化：敏感层跳过

量化会损失精度，所以**只有权重占比 95% 的线性层会量化**，其他层保持原精度：

| 不量化的层 | 原因 |
|-----------|------|
| **Embedding / LM Head** | 词表大（10万+），量化掉点严重；查表操作量化反而慢 |
| **Softmax / LayerNorm** | 数值敏感（exp 函数、归一化对精度敏感） |
| **第一/最后 Transformer 层** | 激活分布最不稳定，容易出现 outlier |

#### 4. 不同层用不同量化方案的原因

```
┌─────────────────────────────────────────────────────────┐
│              Transformer 量化策略                       │
├─────────────────────┬───────────────────────────────────┤
│ 层类型              │ 量化方案                          │
├─────────────────────┼───────────────────────────────────┤
│ 线性层权重          │ INT4 group-wise (AWQ/GPTQ)       │
│ 线性层激活          │ FP8 per-token (vLLM/TensorRT)    │
│ KV Cache            │ FP8 / INT8（长上下文省显存）      │
│ LayerNorm           │ 保持 FP16（数值敏感）             │
│ Softmax             │ 保持 FP16（exp 函数）             │
│ Embedding/LM Head   │ 保持 FP16（查表精度要求高）       │
└─────────────────────┴───────────────────────────────────┘
```

**为什么这么分？**
- 权重用 INT4 group-wise：压缩率最高（4×），精度损失可控
- 激活用 FP8 per-token：动态调整范围，应对激活 outlier
- KV Cache 用 FP8/INT8：长上下文（8K+）下显存占用巨大，必须压缩

#### 5. 量化粒度：越细越准，但越慢

以 `[4096, 12288]` 的 QKV 投影矩阵为例：

| 粒度 | 分组方式 | 参数数量 | 精度 |
|------|---------|---------|------|
| **per-tensor** | 整个矩阵共享 1 组 (scale, zp) | 1 组 | 最差 |
| **per-channel** | 每行（输出通道）共享 1 组 | 4096 组 | 中等 |
| **per-group (128)** | 每行内每 128 个权重共享 1 组 | 393,216 组 | 最好 |

【注解】AWQ/GPTQ 默认用 **per-group (128)**：每 128 个连续权重共享一组量化参数，既能捕捉局部权重分布差异，又不会让元数据开销太大（实际 4.19 bit/weight）。


#### 6. 量化损失的本质：小数位丢了就补不回来

理解前面那么多数学之后，回到最直觉的问题——**为什么量化会有损失？**

```
原始 FP16 权重:  0.12345678  (FP16 精度 ~3 位有效小数，只能存成 0.1235)
量化为 INT4:     round(0.12345678 / scale) → 某个整数
反量化回来:      整数 × scale → 0.125    （又丢一位！）
```

**损失的 0.00154322 就是再也补不回来的精度**。INT4 只有 16 个格子（或非对称 16 个），把一大段浮点数"压缩"进这么少的格子里，必然有信息丢失。

```text
       ┌───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┐
FP16:  │ . │ . │ . │ . │ . │ . │ . │ . │ . │ . │ . │ . │ . │ . │ . │ . │  ← 连续值
INT4:  └───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┘
       └─┰─┘└─┰─┘└─┰─┘ ...  → 只有 16 个离散格子，中间的值只能就近"入格"
        0     1     2

         FP16 的小数值落在两个格子之间 → 只能就近选一个 → 误差 = 半格宽度
```

**所有量化算法的本质，都是在想方设法让这个"半格误差"对最终结果的影响尽可能小：**

| 算法 | 它的"减小损失"思路 |
|------|-------------------|
| **对称/非对称量化** | 选好 min/max 让格子范围刚好覆盖数据，不浪费任何格子 |
| **per-group** | 每 128 个权重单独算 scale，格子变"细"了，误差比例变小 |
| **AWQ** | 重要通道的权重乘 scale 放大后再量化，相当于给重要通道用"更细的尺子" |
| **GPTQ** | 量化产生的误差，通过调整其他权重来抵消，直接补偿到输出值 |
| **NF4 (bitsandbytes)** | 研究权重分布的统计规律，把格子放在权重最密集的区域 |

可以这样理解：

```
同一把尺子（per-tensor）量全身 → 误差大 ❌
每个人单独做一把尺子（per-group） → 误差小 ✅
重要的人用放大镜看（AWQ） → 误差更小 ✅✅
量错了从别人身上补回来（GPTQ） → 误差又被抵消一部分 ✅✅✅
```

## 1.3 主流量化方案对比

下面的表格只列出**量化算法/格式本身**，不含推理框架。TensorRT-LLM / vLLM / llama.cpp 等框架放在表后补充说明。

| 方案 | 支持精度/位宽 | 是否需要校准数据 | 原生硬件加速 | 实际定位 |
|---|---|---|---|---|
| **FP8 (E4M3/E5M2)** | 8 位浮点 | 静态需校准 / 动态不需要 | ✅ H100 Tensor Core 原生 | **生产首选**，几乎无损 + 2× 吞吐 |
| **AWQ** | INT4（group-wise，支持 g64/g128） | 是（校准集） | ❌ 无 | INT4 方案推荐首选，精度高推理快 |
| **GPTQ** | 2 / 3 / 4 / 8 bit | 是（校准集） | ❌ 无 | 最早的 INT4 方案，兼容性好 |
| **GGUF k-quant** | Q2_K ~ Q8_K + IQ1~IQ4 | 否（纯权重 PTQ） | ❌ 无 | CPU/本地推理（llama.cpp 原生加载） |
| **BitsAndBytes (NF4/FP4)** | INT8 / INT4 (NF4) / FP4 | 否 | ❌ 无 | QLoRA 微调时省显存（模块三） |

【注解】**校准数据（Calibration Data）**：量化需要一组代表性样本来统计激活值的分布，从而确定最优的量化参数。通常从训练集中随机取 128-1024 条即可。
**是否需要校准数据** 取决于具体方案：AWQ/GPTQ 等**需要**（用前向传播统计权重/激活分布）；FP8 **静态**量化需要校准集，**动态**量化（H100 硬件自动处理）不需要。

【注解】**g64/g128** 指每 64/128 个连续权重共用一组量化参数。分组越细精度越高（C-Eval 提升约 0.1-0.3%），但 scale/zp 数量翻倍 → 推理时 GPU 显存带宽消耗增大（每算一组权重多取一次 scale），导致延迟略增。工业界默认 g128，g64 只在极端看重质量的场景使用。

【注解】**框架对量化方案的支持关系**（框架本身不是量化方案，是承载量化方案的部署工具）：
- **vLLM**：支持 FP8（原生）、AWQ、GPTQ，一行命令启动推理服务
- **TensorRT-LLM**：NVIDIA 官方引擎，支持 FP8（含 KV Cache FP8），需编译流程
- **llama.cpp (GGUF)**：CPU/本地推理方案，内置 k-quant 量化算法

以上框架的量化实操在 （2.3-2.4 节）展开。

```python
# bnb_nf4.py
# 核心原理：加载时直接把 FP16 权重压成 NF4 4-bit，显存减半
# 依赖：bitsandbytes>=0.41.0（0.41+ 开始支持 Windows）

from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig
import torch

MODEL = r"C:\Users\hp\Downloads\models\qwen3.5-0.8B"

# ── NF4 量化配置 ──────────────────────────────────────────────────
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,                  # 启用 4-bit 量化
    bnb_4bit_quant_type="nf4",          # NormalFloat4 格式
    bnb_4bit_compute_dtype=torch.float16,   # 计算精度用 FP16
    bnb_4bit_use_double_quant=True,     # 二级量化进一步压缩
)

print("正在加载模型（NF4 量化）...")
model = AutoModelForCausalLM.from_pretrained(
    MODEL,
    quantization_config=bnb_config,
    device_map="auto",
)
tokenizer = AutoTokenizer.from_pretrained(MODEL)

# ── 推理测试 ───────────────────────────────────────────────────────
prompt = "解释一下什么是大模型量化："
inputs = tokenizer(prompt, return_tensors="pt").to(model.device)
outputs = model.generate(**inputs, max_new_tokens=100)
print(tokenizer.decode(outputs[0], skip_special_tokens=True))

# ── 显存对比 ───────────────────────────────────────────────────────
used = torch.cuda.memory_allocated() / 1024**3
print(f"\nNF4 量化后显存占用: {used:.2f} GB（FP16 原本约 1.6GB）")
```

### 动手实验：查看模型的显存占用


```python
# 【注解】在量化之前，先看看一个模型在不同精度下各占多少显存
# 原理：参数量 × 精度字节数 = 权重显存

def calc_vram(param_b: float, precision: str) -> float:
    """计算模型权重占用的显存"""
    bytes_per_param = {
        'fp32': 4,   # 32位 = 4字节
        'fp16': 2,   # 16位 = 2字节
        'fp8':  1,   # 8位  = 1字节
        'int4': 0.5, # 4位  = 0.5字节
        'int3': 0.375,
        'int2': 0.25,
    }
    return param_b * bytes_per_param[precision]

# 热门模型的参数量（单位：B = Billion）
models = {
    'Qwen3.5-9B':         9,       # Dense
    'Qwen3.5-35B-A3B':   35,       # MoE（总参数）
    'Qwen3.5-397B-A17B': 397,      # MoE旗舰
    'DeepSeek-V4-Flash': 284,      # MoE
    'GLM-4.7-Flash':     30,       # MoE
}

precisions = ['fp16', 'fp8', 'int4']

print(f"{'模型':<25} {'FP16':>8} {'FP8':>8} {'INT4':>8} {'备注'}")
print('-' * 60)
for name, params in models.items():
    vrams = [calc_vram(params, p) for p in precisions]
    note = ''
    if params >= 100:
        note = f'需{params//80+1}×H100'
    elif vrams[1] <= 24:  # FP8 塞得进 4090
        note = f'FP8可跑RTX 4090（{vrams[1]:.0f}GB < 24GB）'
    elif vrams[2] <= 24:  # INT4 塞得进
        note = f'INT4可跑RTX 4090（{vrams[2]:.0f}GB < 24GB）'
    
    print(f"{name:<25} {vrams[0]:>6.0f}GB {vrams[1]:>6.0f}GB {vrams[2]:>6.0f}GB  {note}")
    
total_fp16 = calc_vram(397, 'fp16')
total_fp8 = calc_vram(397, 'fp8')
total_int4 = calc_vram(397, 'int4')
print(f"\nQwen3.5-397B: FP16={total_fp16:.0f}GB → FP8={total_fp8:.0f}GB → INT4={total_int4:.0f}GB")
print(f"压缩率: 从{total_fp16:.0f}GB到{total_int4:.0f}GB，缩小了{total_fp16/total_int4:.1f}倍！")

## 2.1 AWQ：激活值感知量化

**AWQ（Activation-Aware Weight Quantization）** 是目前最流行的 INT4 量化方案，由 MIT 和 NVIDIA 联合提出。


**核心思想**：不是所有权重通道都同等重要。那些**对应大激活值**的权重通道对模型质量影响更大，应该保留更高精度。


```

AWQ 量化流程:

  原始FP16权重 → 计算校准集激活值 → 统计每通道激活值大小
                 ↓
  识别"重要通道"（激活值大的通道）
                 ↓
  重要通道：保留更多精度（per-channel scale）
  非重要通道：正常量化
                 ↓
  输出 INT4 量化模型 + scale 参数

```

【注解】AWQ 的独特之处在于它**不需要反向传播**，而是通过激活值统计来指导量化。因此 AWQ 的量化速度很快（一张 4090 上量化 9B 模型只需几分钟），且质量优于 GPTQ。


【通俗理解】AWQ 不是"增加重要通道的权重值"，而是**量化前先缩放，量化后再反缩放**：

```
校准文本跑一遍前向传播 → 统计每列的平均激活值
找到激活值大的"重要列"
    
对重要列的权重提前乘一个大于 1 的 scale（如 s=2）
此时权重范围被"拉开"了 → 量化时的相对误差变小
    
做完 INT4 量化后，保存 scale 参数
推理时计算完该列结果 ÷ s 校正回来
```

为什么这么设计？激活值大的通道就算权重被 s 放大，推理时除以 s 就回来了，**输出的语义不变**。但权重被放大后，INT4 的量化格子更"精细地"表示了它，损失比例反而小了。

```
打个比方：
  一把尺子量头发丝（权重小）→ 一毫米误差就是大比例 ❌
  换成放大镜看头发丝（乘 s 放大）→ 同样一毫米误差比例小很多 ✅
  看完再把放大镜倍数除以 s 复原 → 测量结果不变
```

## 2.2 GPTQ：基于 Hessian 矩阵的量化

**GPTQ（Post-Training Quantization for GPTs）** 是 2023 年最早流行起来的 INT4 量化方案，使用二阶信息（Hessian 矩阵）指导量化。


**与 AWQ 的核心区别**：


|对比维度|AWQ|GPTQ|
|---|---|---|
|量化依据|激活值统计（一阶）|Hessian 矩阵（二阶）|
|是否需要反向传播|否|是|
|量化速度|快（~5分钟 / 9B）|慢（~20分钟 / 9B）|
|INT4 质量|略好（+0.5%）|好|
|生态支持|AutoAWQ、vLLM|AutoGPTQ、exllama|

【注解】两者最终效果差距不大，在 **vLLM 的生态中 AWQ 和 GPTQ 都支持**。但 AWQ 在 4090 上推理速度略快，建议新项目优先选择 AWQ。


【通俗理解】GPTQ 的核心不是简单 round，而是**量化一个权重后，用 Hessian 矩阵指导去调整同一行中还没量化的权重**，补偿输出误差：

```
校准文本前向传播 → 每层得到输入激活 X → 计算 Hessian = X^T X

对某一行权重：w1, w2, w3, w4  →  输出 = w1·x1 + w2·x2 + w3·x3 + w4·x4

步骤：
  1. 按 Hessian 对角线值排序，从"最不重要"的权重开始量
  2. 把 w1 压成 INT4 → 再反量化 → w1_hat
  3. 输出多了/少了 δ = (w1 - w1_hat) · x1
  4. 从还没量化的 w2, w3, w4 补偿回来！
        补偿量 = δ × Hessian[i, j] / Hessian[j, j]
  5. 重复直到所有权重量化完
```

可以想象成**捏橡皮泥**：

```
一行权重 = 一根橡皮泥条，总长度 = 这行的输出值
  
把 w1 捏成 INT4（切掉一小块）→ 总长度变了 δ
那就从 w2, w3, w4 上挤一点过来补上 δ
  
Hessian 告诉你：从哪块挤、挤多少最合适
```

GPTQ 是**量化后补偿**的思路，和 AWQ 的**量化前保护**正好互补。

## 2.4 TensorRT-LLM 与 FP8 生产量化

FP8 是 2024-2026 年生产落地的**事实标准精度**。它和 INT4/INT8 量化有本质区别——**FP8 不是"压缩"，而是用更少 bit 的浮点数格式**来存储权重。

> **打个比方**：
> - **INT4 量化** = 用**整十厘米的尺子**量身高。你实际 175cm，尺子每 10cm 一个格子，只能记成"170cm"或"180cm"。173cm 和 177cm 的人记成了同一个数字，分不清了。大概知道多高，但精确信息丢了。
> - **FP8 量化** = 用**毫米线模糊了的厘米尺**量身高。你实际 175.3cm，尺子只能读到厘米，记成 175cm。丢了 0.3cm，但绝大部分情况准的，差几毫米没人看得出来。

---

### 2.4.1 FP8 的本质：它不是"量化"，是另一种浮点数

先说清楚一个很多人搞混的点：

| 对比 | INT4 量化 | FP8 |
|------|----------|-----|
| 原始格式 | FP16 / BF16 (浮点数) | FP16 / BF16 (浮点数) |
| 压缩后格式 | **整数** (INT4/INT8) | **浮点数** (FP8) |
| 数学性质 | 需要 scale + zero_point | 自带指数位，天然带范围 |
| 要不要"反量化" | 必须反量化回 FP16 才能计算 | **可以直接计算** |
| 计算方式 | 整数矩阵乘法 → 反量化 → 累加 | FP8 Tensor Core 直接算 |

```text
INT4 的完整链路：

  FP16 权重 → round(FP16 / scale) → INT4 存储
  推理时：INT4 × scale → 反量化回 FP16 → FP16 矩阵乘法 → 累加
                                                      ↑
                                          这里还是用 FP16 单元算的！
                                          所以 INT4 不加速计算只省显存


FP8 的链路：

  FP16 权重 → cast 成 FP8 → FP8 存储
  推理时：FP8 × FP8 → FP8 Tensor Core 直接乘 → FP32 累加
                                            ↑
                                    H100 原生的 FP8 计算单元
                                    算力是 FP16 的 2 倍！
```

**【关键区别】**：
- INT4 只是**省显存**，计算时还是要回到 FP16 单元——所以 INT4 **不加速推理，只省显存**
- FP8 是 **H100 Tensor Core 原生支持的格式**，矩阵乘法直接在 FP8 单元上完成——**既省显存，又提速 2 倍**

这就是为什么 FP8 才是"生产方案"——它不只是存得更小，是算得更快。

---

### 2.4.2 前置知识：FP16 vs BF16，浮点数的"范围与精度"

要理解 FP8 为什么有两种格式，先要搞明白**浮点数是什么**。

#### 浮点数就是"科学记数法"

```text
日常科学记数法（十进制，基数=10）：

  1.2323526 × 10⁴ = 12323.52
    ↑         ↑
  尾数       指数（10⁴ = 放大 1 万倍）


计算机浮点数（二进制，基数=2）：

  sign × mantissa × 2^exponent
   [符号] [尾数]     [指数，2的幂]

结构和日常科学记数法一模一样！
唯一区别：日常十进制基数=10，计算机基数=2。
```

**尾数** = 有效数字（精度），**指数** = 能表示的范围有多大。

---

#### FP16 和 BF16 的区别：一个"精"，一个"远"

大模型推理主要用两种 16 位浮点数：

```text
FP16（也叫 FP16 标准格式）：

  S  E E E E E   M M M M M M M M M M
  1    5 bit        10 bit

  → 指数 5 bit：范围 ±65504（够用，如何计算详见下方注解）
  → 尾数 10 bit：精度高（~3 位有效小数）
  → 适用：权重存放、推理计算


BF16（Brain Float，Google 为深度学习发明的）：

  S  E E E E E E E E   M M M M M M M
  1      8 bit            7 bit

  → 指数 8 bit：范围和 FP32 一样大（±3.4×10⁸）
  → 尾数 7 bit：精度低（~2 位有效小数）
  → 适用：训练（怕梯度爆炸，需要大范围兜底）


**【尾数 bit 数决定"相对精度"】**

尾数 bit 数是关键：**它决定了浮点数在 [1.0, 2.0) 之间有多少个格子**，也就是"相对精度"。

- 尾数 7 bit → [1.0, 2.0) 之间 2⁷ = **128** 个格子 → 相对精度 **1/128 ≈ 0.8%**
- 尾数 10 bit → [1.0, 2.0) 之间 2¹⁰ = **1024** 个格子 → 相对精度 **1/1024 ≈ 0.1%**
- 尾数 23 bit → [1.0, 2.0) 之间 2²³ ≈ **830 万**个格子 → 相对精度 **1/8.3×10⁶ ≈ 0.00001%**

指数只是把这个区间整体放大缩小，格子之间的**相对比例不变**：

​```text
BF16（相对精度 0.8%）：
  数值 ≈ 3    → 相邻两个值的差距 ≈ 3 × 0.8% = 0.024
  数值 ≈ 100  → 相邻两个值的差距 ≈ 100 × 0.8% = 0.8
  数值 ≈ 1500 → 相邻两个值的差距 ≈ 1500 × 0.8% = 12

  精度没变，都是 0.8%。只是同样的百分比用在不同量级上，
  绝对差值不一样而已。

FP16（相对精度 0.1%）：
  数值 ≈ 3    → 相邻两个值的差距 ≈ 3 × 0.1% = 0.003
  数值 ≈ 100  → 相邻两个值的差距 ≈ 100 × 0.1% = 0.1
```

**一句话**：尾数 bit 决定"多精确"（相对精度），指数只决定"量多大"。精度本身只和尾数有关。

**【一个直观的例子】**：存一个数 100.17

在数值 ≈ 100 这个量级，FP16 和 BF16 的差异就很明显了：

```text
FP16（尾数 10 bit）：
  [64, 128) 之间分了 1024 格 → 每格宽 64/1024 = 0.0625
  能存的数：100.0000, 100.0625, 100.1250, 100.1875, ...
  
  要存 100.17 → 最接近的是 100.1875（误差 0.0175）
  → 损失万分之 1.7，几乎看不出 ✅


BF16（尾数 7 bit）：
  [64, 128) 之间分了 128 格 → 每格宽 64/128 = 0.5
  能存的数：100.0, 100.5, 101.0, 101.5, ...
  
  要存 100.17 → 最接近的是 100.0（误差 0.17）
  或 100.5（误差 0.33）
  → 无论哪个都丢了将近 2%！❌


同样的数值，BF16 的格子比 FP16 粗了 8 倍
（因为尾数少了 3 bit → 格子数少了 2³ = 8 倍）。
```

**范围对比**：BF16 的指数 8 bit，最大能到 ±3.4×10³⁸（跟 FP32 一样）；FP16 的指数 5 bit，最大只到 ±65504。这就是"牺牲精度换范围"的代价。

> **【扩展】FP16 的 65504 范围是怎么算的？**（理解指数 bit 的作用）
>
> FP16 的公式：`(-1)^S × 1.mantissa × 2^(E - bias)`
>
> - 指数 E（5 bit）= 1 ~ 30（二进制 00001 ~ 11110）
> - bias（偏移量）= 2^(5-1) - 1 = **15**
> - 实际指数范围 = -14 ~ +15
>
> 最大值计算：
> ```
> 尾数最大 = 1.1111111111（二进制）= 2 - 1/1024 ≈ 1.999
> 最大值 = 1.999 × 2^15 = (2 - 2^(-10)) × 32768 = 65504
> ```
>
> 不同格式的指数 bit 数与范围：
> | 格式 | 指数 bit | bias | 实际指数 | 最大值 |
> |------|---------|------|---------|------|
> | **FP16** | 5 | 15 | -14 ~ +15 | **65504** |
> | **BF16** | 8 | 127 | -126 ~ +127 | **~3.4×10³⁸**（和 FP32 一样大） |
> | **FP8 E4M3** | 4 | 7 | -6 ~ +7 | **±448** |
> | **FP8 E5M2** | 5 | 15 | -14 ~ +15 | **±57344** |
>
> **规律：指数 bit 越多，范围越大。但指数和尾数共用 16 bit，此消彼长。**
```

**【核心直觉】**：指数和尾数就这么多 bit，**分给范围的越多，精度就越差；分给精度的越多，范围就越小。**

​```text
                   ← 精度高 ────  范围大 →
  FP32:  S(1)  E(8)  M(23)      ← 基准（啥都不缺）
  FP16:  S(1)  E(5)  M(10)      ← 精度还行，范围够用
  BF16:  S(1)  E(8)  M(7)       ← 范围超大（跟 FP32 一样），精度差了点
```

**大模型为什么用 BF16 训练，用 FP16 推理？**

- **训练**时梯度可能爆炸/消失，BF16 的大范围能兜住极端值
- **推理**时权重已经收敛稳定，范围不大但精度更重要 → FP16

### 2.4.3 FP8 的两种格式：E4M3 vs E5M2

有了上面的基础，FP8 就很好懂了——**就是 FP16/BF16 的"更小版本"**，指数和尾数各分到的 bit 更少：

| 格式 | 指数位 | 尾数位 | 范围 | 精度 | 类比例子 |
|------|--------|--------|------|------|---------|
| **E4M3** | 4 bit | 3 bit | ±448 | 相对精细 | 像"精简版 FP16"——精度优先 |
| **E5M2** | 5 bit | 2 bit | ±57,344 | 很粗 | 像"精简版 BF16"——范围优先 |

```text
看亲戚关系一目了然：

  FP16 (E5M10)     → 压缩 →  FP8 E4M3（指数少 1，尾数少 7）
  BF16 (E8M7)      → 压缩 →  FP8 E5M2（指数少 3，尾数少 5）

  E4M3 = "我像 FP16，精度为重" → 权重 + 激活值用这个
  E5M2 = "我像 BF16，范围为大" → 梯度、KV Cache 用这个
```

现在再回头看那个 bit 结构：

```text
FP8 E4M3（权重和激活用）：

  S  E E E E   M M M
  1   4 bit     3 bit

  表示的值 = (-1)^S × 2^(E-7) × (1 + M/8)

  [这里的 7 是什么？]
  指数 E（4 bit）只能存 0~15，但我们需要指数能表示负数（存小数）。
  所以引入 bias（偏移量）：bias = 2^(4-1)-1 = 7
  实际指数 = E - 7 → 范围是 -6 ~ +7
  这样既能表示大数（×2⁷），也能表示小数（×2⁻⁶）。

  3 bit 尾数 → 只能表示 1/8 = 0.125 的倍数
  → 3.14 被存成 3.0（丢了 0.14）
  → 但几十亿个权重同时丢一点，误差互相抵消，最终结果几乎不变


FP8 E5M2（梯度、KV Cache 用）：

  S  E E E E E   M M
  1   5 bit      2 bit

  2 bit 尾数 → 只能表示 1/4 = 0.25 的倍数
  → 精度更差，但范围能到 ±57344（比 E4M3 大 128 倍）
  → 梯度训练时数值变化剧烈，范围大比精度更重要
```

**一句话记住：**

> E4M3 = 精度优先（像 FP16），E5M2 = 范围优先（像 BF16）。  
> 两个都是浮点数，**和 FP16/BF16 是同一家人，只是零花钱更少了**。

**【进阶备注】**：这三位数值的"家族谱系"：

```
                  FP32  (E8M23)  ← 祖宗
                 /              \
           FP16 (E5M10)     BF16 (E8M7)
               |                 |
           FP8 E4M3          FP8 E5M2
           (精度像 FP16)     (范围像 BF16)
```

有人会问：**为什么不直接说 FP8 = FP16 砍掉 8 bit 尾数？**  
因为如果全部给尾数（E5M10 → E5M2），精度损失太大。**E4M3 的做法是：从指数借 1 bit 给尾数**，所以指数变少（范围小了），但尾数多了一位（精度好一点）。这就是"范围 vs 精度"的取舍。

### 2.4.4 FP8 量化的两种做法：静态 vs 动态

"FP8 不需要校准数据"这句话不完全准确——它取决于你用哪种方式：

| 方式 | 什么时候算 scale | 需不需要校准 | 精度 | 速度 |
|------|-----------------|------------|------|------|
| **FP8 静态量化** | 推理前，统计整个训练集的激活分布 | **需要**（一次性的） | 更高 | 推理时无额外开销 |
| **FP8 动态量化** | **推理时**，每层每 token 实时算 | **不需要** | 略低 | 推理时有微小开销 |
| **TensorRT-LLM FP8** | 推理时，H100 硬件自动处理 | **不需要** | 高 | **硬件级**，零开销 |

#### 实际场景对照：官方发布的 FP8 模型 vs 自己运行时量化

官方开源的 `Qwen3.6-27B-FP8` 正是理解静态 vs 动态的好例子：

| | **官方 FP8 模型**（静态） | **FP16 模型 + --quantization fp8**（动态） |
|---|---|---|
| **举例** | `Qwen/Qwen3.6-27B-FP8` | `Qwen/Qwen3.6-27B` + vLLM `--quantization fp8` |
| **权重文件** | 已经是 FP8 格式（~27GB） | 还是 FP16 格式（~54GB） |
| **下载大小** | 小（省一半下载时间） | 大（下完再动态转） |
| **推理流程** | 直接读 FP8 权重 → FP8 Tensor Core 计算 | 边加载边把 FP16 转 FP8 → FP8 Tensor Core 计算 |
| **谁做了量化** | **厂商**（阿里巴巴/NVIDIA）用校准集离线算好 | **你本地机器**运行时自动转 |
| **精度** | 通常更高（厂商会微调 scale 优化质量） | 略低（纯数学 cast，不优化） |
| **推荐** | **首选**，有官方 FP8 版就用它 | 没有官方 FP8 版时的备选方案 |

```text
下载量对比（以 Qwen3.6-27B 为例）：

  FP16 版：~54 GB → 千兆宽带也要 7 分钟
  FP8 版： ~27 GB → 3.5 分钟

显存占用对比（推理时）：

  FP16 版（--quantization fp8）：第一次加载慢（要转格式），
                                  但之后显存占用 ≈ FP8 版
  FP8 版：加载即用，省时省力
```

**【一句话结论】**：去 Hugging Face 搜 `-FP8` 后缀的模型，有就直接用，**比自己动态量化更好更快**。只有搜不到官方 FP8 版时，才用 `--quantization fp8` 自己转。

```text
静态量化的流程（TensorRT-LLM / vLLM 的传统做法）：

  离线阶段：
    校准数据集 → 跑一遍前向传播 → 统计每层激活值 min/max
    ↓
    算出每层的 FP8 scale（缩放因子）
    ↓
    把 scale 打包进模型文件

  推理阶段：
    加载模型 + scale 表
    每层计算时：FP16 激活 × scale → cast 成 FP8 → FP8 矩阵乘 → 结果 cast 回 FP16
   

动态量化的流程（H100 原生）：

  每个 token 进来时，硬件自动看当前激活值的范围
  按 token 级别动态决定 FP8 的 scale
  整个过程**不需要任何软件干预**——完全在 Tensor Core 内部完成

  这就意味着：
  ✓ 不需要准备校准数据集
  ✓ 不需要缓存 scale 表
  ✓ 激活值分布变化时自动适应
  ✗ 每个 token 多一步"算范围"的开销（硬件做，几乎可以忽略）
```

**【为什么动态量化也能保证质量？】**

H100 的 FP8 Tensor Core 会在每层计算时**即时统计当前 token 的激活值范围**，然后动态调整 FP8 的 scale。这比静态方案更灵活——因为**不同 token 的激活值分布可能差异很大**。

举个例子：
```text
token "A" 的激活值范围: [-0.5, 0.8]  → scale = 0.8 / 448 = 0.0018
token "我" 的激活值范围: [-3.2, 5.1]  → scale = 5.1 / 448 = 0.0114

静态量化用一个统一的 scale 覆盖所有 token → 对"A"来说格子太粗了 ❌
动态量化为每个 token 单独算 scale → 各自的最优格子 ✅
```

### 2.4.5 TensorRT-LLM 做了什么？

TensorRT-LLM 不是"一个量化工具"，它做三件事：

1. **模型图优化**：把 Transformer 的计算图重新编排，合并 Kernel、消除冗余
2. **自动精度分配**：分析每层对精度的敏感度，自动决定哪些层用 FP8、哪些保持 FP16
3. **KV Cache FP8**：把缓存的历史 K 和 V 也压成 FP8（节省 50% 的 KV Cache 显存）

```text
TensorRT-LLM 的优化流程：

  原始 PyTorch 模型（.pt / .safetensors）
                    ↓
  TRT-LLM 编译：将 PyTorch 图转成 TensorRT 引擎
                    ↓
  精度分析（FP8 Auto Mode）：
    ┌─ 线性层权重  → FP8 E4M3 ✔
    ├─ 激活值      → FP8 E4M3（per-tensor 动态 scale）
    ├─ Attention   → FP8 E4M3 ✔
    ├─ LayerNorm   → 保持 FP16  ✘（数值敏感）
    ├─ Softmax     → 保持 FP16  ✘（exp 函数）
    └─ KV Cache    → FP8 E5M2（范围需要更大）
                    ↓
  TensorRT 引擎（.engine 文件，H100 可直接运行）
```

**TensorRT-LLM vs vLLM 的 FP8 区别**：

| 对比 | TensorRT-LLM | vLLM |
|------|-------------|------|
| 定位 | NVIDIA 官方推理引擎 | 通用推理框架 |
| FP8 精度 | **可能会更好**（自动敏感度分析） | 全量 FP8 |
| KV Cache FP8 | ✅ 原生支持 | `--kv-cache-dtype fp8` |
| 编译时间 | 需要编译引擎（半小时到数小时） | 无需编译，加载即用 |
| 灵活度 | 固定图（改模型要重编译） | 动态图，模型随便换 |
| 使用复杂度 | 高（需要 TRT-LLM 工具链） | 低（一行命令） |

### 2.4.6 FP8 量化的"直觉"

新手最常问的问题：**"FP8 只有 256 个值（2^8），为什么叫量化？INT4 只有 16 个值（2^4）才叫量化啊？"**

这个问题的答案正是 FP8 和 INT4 的核心区别：

```text
FP8 的 256 个值分布（E4M3）：

  ┌────────────────────────────────────────────────────┐
  │   ~0  ≈0.001  ≈0.01  ≈0.1   ≈1    ≈10   ≈100  ~448│
  │   ·    ·       ·      ·      ·     ·     ·     ·   │
  │   ·    ·       ·      ·      ·     ·     ·         │
  │   ·    ·       ·      ·      ·     ·               │
  │   ·    ·       ·      ·      ·                     │
  │   ·    ·       ·      ·                           │
  │   ·    ·       ·                                  │
  │   ·    ·                                          │
  │   ·                                               │
  └────────────────────────────────────────────────────┘

  FP8 的格子不是均匀的！越接近 0 越密，越远越疏。
  这是浮点数的天然优势——指数位自动适配量级。


INT4 的 16 个值分布（对称量化）：

  ┌────────────────────────────────────────────────────┐
  │ -8  -7  -6  -5  -4  -3  -2  -1   0   1   2   3   4│
  │                        ↑                           │
  │              格子等间距，没有"重点区域"               │
  └────────────────────────────────────────────────────┘

  这就是为什么 INT4 量化损失更大——
  所有区域"一视同仁"，但模型权重大部分集中在 0 附近。
  小权重被大格子的误差"淹没"了。
```

**【核心结论】**：
- **FP8 是指数分布**的格子——0 附近的细节保留得很好，大值也能覆盖
- **INT4 是均匀分布**的格子——0 附近和大值区域用同样精度的尺子量，浪费了精度
- 这就是为什么 FP8 ≈ 无损（损失 < 0.5%），而 INT4 损失 ~1-2%

### 2.4.7 实操：vLLM + FP8 实际对比

以下实验可以在 H100 上对比 FP16 和 FP8 的性能差异（有 H100 时可以实操）：

```python
# fp8_benchmark.py
# 对比 vLLM 上 FP16 vs FP8 的吞吐和显存
# 需要 H100 硬件 + vLLM
# pip install vllm

import time
from vllm import LLM, SamplingParams

# 测试 prompt（模拟实际生产负载）
test_prompts = [
    "用 Python 写一个二分查找算法。" * 5,     # 短文本
    "请详细解释 Transformer 架构中自注意力机制的计算过程，"
    "包括 Q/K/V 的生成、注意力分数的计算、softmax、"
    "加权求和以及多头注意力的拼接。请给出具体的数学公式。",
    # 实际测试可以准备 100 条不同的 prompt
]

sampling_params = SamplingParams(
    temperature=0.7,
    max_tokens=256,
)

def test_precision(name, quantization=None):
    """测试指定精度的吞吐"""
    print(f"\n{'='*50}")
    print(f"测试: {name}")
    
    t0 = time.time()
    llm = LLM(
        model="Qwen/Qwen3.5-9B",
        quantization=quantization,           # None=FP16, "fp8"=FP8
        tensor_parallel_size=1,              # 单卡测试
        gpu_memory_utilization=0.90,
        max_model_len=4096,
    )
    load_time = time.time() - t0
    print(f"模型加载时间: {load_time:.1f}s")
    
    # 循环测试
    total_tokens = 0
    t0 = time.time()
    for _ in range(10):  # 跑 10 轮
        outputs = llm.generate(test_prompts, sampling_params)
        for o in outputs:
            total_tokens += len(o.outputs[0].token_ids)
    elapsed = time.time() - t0
    
    throughput = total_tokens / elapsed
    print(f"生成 {total_tokens} tokens, 耗时 {elapsed:.1f}s")
    print(f"吞吐: {throughput:.0f} tokens/s")
    
    del llm

# 对比 FP16 vs FP8
test_precision("FP16 (基线)", quantization=None)
test_precision("FP8 (量化)", quantization="fp8")
```

### 2.4.8 TensorRT-LLM 实操体验（纯代码演示，无需 H100）

对于没有 H100 硬件但想理解 TensorRT-LLM 编译流程的学员，可以用以下代码演示**数据流**（不实际执行，只展示流程结构）：

```python
# 【注解】以下代码演示 TensorRT-LLM 的 FP8 量化流程
# 不需要 H100 即可阅读理解其架构

TensorRT-LLM FP8 量化编译的标准流程：

1. 准备模型（FP16 原始权重）
2. 扫描每层的激活值分布（weight scanning）
3. 自动分配精度策略（auto precision）
4. 编译成 TensorRT engine
5. 推理时加载 engine + FP8 scale 缓存
"""

# 用注释形式展示，不需要实际执行

def trtllm_fp8_pipeline():

    # Step 1: 从 Hugging Face 下载并转换为 TRT-LLM 格式

        --dtype bfloat16

        --max_output_len 2048 \
        --max_num_tokens 8192
    
    # Step 3: 启动推理服务
    python run_triton_server.py \
        --engine_dir ./trtllm_engines/qwen9b_fp8 \
        --model_name qwen_fp8 \
        --tokenizer_dir Qwen/Qwen3.5-9B
    
    What happens inside:
    
    Step 1 → 把 PyTorch 权重转成 TRT-LLM 内部格式
             还会扫描权重的 FP8 scale（静态量化用）
    
    Step 2 → 编译引擎时：
              - 将 Transformer 图重写为 TensorRT IR
              - 自动插入 FP8 cast 节点（权重量化 + 激活量化）
              - 做 kernel 融合（比如 QKV 三个矩阵合在一起算）
              - 优化显存分配
              - 输出 .engine 文件（二进制，模型 + 图 + 优化参数）
    
    Step 3 → 加载 engine 到 H100，启动 Triton Inference Server
    """
    pass
```

**【为什么 TensorRT-LLM 比 vLLM 复杂？】**

vLLM 的 `--quantization fp8` 一句话就搞定了。TensorRT-LLM 要三步。

原因是：

```text
vLLM：      模型加载 → [PyTorch 运行时]                → 推理
            FP8 量化是在 PyTorch 层面做的，走的是 PyTorch 的算子

TensorRT-LLM：模型加载 → [编译优化] → TensorRT 引擎 → 推理
              FP8 量化是编译时确定的，推理时走的是 NVIDIA 最底层的 CUDA C++
              针对你的模型"量体裁衣"做优化
```

类比：
- **vLLM FP8** = 买成品西装，穿上就行（通用，方便，够好）
- **TensorRT-LLM FP8** = 找裁缝量身定做（更合身，但耗时，换人得重做）

**实际生产中怎么选？**
- 快速上线 → vLLM + `--quantization fp8`（Qwen/DeepSeek 官方推荐就是这个）
- 极致性能 → TensorRT-LLM（尤其对延迟敏感的在线服务）
- NVIDIA 官方 Benchmark 显示：TensorRT-LLM FP8 比 vLLM FP8 快约 10-20%，但部署复杂度高得多

---

### 2.4.9 FP8 应用于 KV Cache

这是 FP8 在生产中最"立竿见影"的应用场景——**把注意力机制中的 Key 和 Value 缓存也压成 FP8**。

```text
KV Cache 的大小：
  
  单次请求的 KV Cache = 层数 × 2 × seq_len × hidden_size × 2 bytes（FP16）
  
  Qwen3.5-9B（28 层，hidden=4096）：
    seq_len=4096 → KV Cache ≈ 28 × 2 × 4096 × 4096 × 2 ≈ 1.8 GB
    seq_len=8192 → KV Cache ≈ 3.6 GB（翻倍！）
    seq_len=128K → KV Cache ≈ 56 GB（一张 H100 的显存全占满！）
  
  设为 FP8 后：
    seq_len=128K → KV Cache ≈ 28 GB ← 省了一半，剩下的空间还能放权重
```

这就是为什么长上下文模型（128K+ token）必须量化 KV Cache——不量化的话，**一个用户的会话就占满一张 H100**。

```text
vLLM KV Cache FP8 启用：

  python -m vllm.entrypoints.openai.api_server \
      --model Qwen/Qwen3.5-35B-A3B \
      --quantization fp8 \          # 权重 + 激活 FP8
      --kv-cache-dtype fp8 \        # KV Cache 也用 FP8
      --max-model-len 131072        # 128K 长上下文
```

**【KV Cache FP8 为什么不会降低质量？】**

因为 KV Cache 中的 value 只是一个"缓存"，推理时 KV Cache 的值经过 softmax（0~1 之间），数值范围非常稳定。FP8 的动态范围完全够用。

---

### 2.4.10 搞清概念：为什么 FP8 不算"量化"？

很多教材把 FP8 叫做"量化"，但你学了上面这些应该明白了——**FP8 不是量化，是原生浮点数格式的降级**。

| 概念 | 真正含义 | 例子 |
|------|---------|------|
| **量化（Quantization）** | 浮点数 → **整数**，需要 scale 反量化 | INT4 / INT8 / NF4 |
| **精度降级（Precision Casting）** | 高精度浮点 → 低精度浮点，**无需反量化** | FP16 → FP8 / BF16 |
| **精度提升** | 低精度 → 高精度 | FP16 → FP32（训练梯度） |

**【为什么人们还是叫它"FP8 量化"？】**

习惯和历史原因：
1. 效果上，它和量化一样**节省了显存**（50%）
2. 和量化一样**可能有精度损失**
3. NVIDIA 自己的文档也这么叫
4. 但它和 INT4 量化的底层机制**完全不同**

```text
正确的思维方式：

  FP8 = 把 16 位浮点数"截断"成 8 位浮点数
  INT4 = 把 16 位浮点数"映射"成 4 位整数

  截断（cast）→ 保留浮点数结构，只是位数少了
  映射（quantize）→ 转换成完全不同的格式，需要变换参数
```

我建议记住一个原则：**FP8 是"缩印"（字体不变，大小减半），INT4 才是"翻译"（从浮点转成整数格式）。**

## 3.1 知识蒸馏（Knowledge Distillation）

知识蒸馏的核心思想：**让一个大模型（教师）教一个小模型（学生）**。但工业界实际怎么做蒸馏，已经变了。


### 现在的主流：数据蒸馏（SFT Distillation）

大模型时代，**"蒸馏" = 用教师模型输出高质量数据 → 用 SFT 微调学生模型**。这是工业界最主流的做法。

```
数据蒸馏流程（实际工业做法）：

  教师模型（大）：GPT / Claude / Qwen3.5-397B / DeepSeek-V4-Pro 等更强的模型
                    ↓ 调用 API 生成回答
  构建蒸馏数据集：10 万条 prompt + 教师输出
                    ↓
  学生模型（小）：Qwen3.5-9B / Qwen3.5-0.8B
                    ↓ 用 SFT 微调（和普通微调完全一样）
  输出能力接近教师的小模型
```

**为什么不用 KL 散度？** KL 散度蒸馏要求教师和学生**同时跑前向传播**（teacher forward + student forward），学生模仿教师每层输出的概率分布。这对 LLM 来说：

1. 教师模型太大（397B），一次推理就几十秒，造数据太慢
2. 学生也是 LLM，不是分类模型，KL 散度对长文本生成的效果很差
3. 直接拿教师输出做 SFT 微调，更简单、效果一样好

**所以现在的实际做法：SFT 微调 = 数据蒸馏。** 学生模型学的不是"概率分布的差异"，而是"教师输出的文本本身"。


### 数据蒸馏的实操流程

```python
# 数据蒸馏实操：用教师模型生成数据 → 微调自己的小模型
# 这里用 OpenAI API 演示（实际换成任何更强的模型均可）
# 用你有的最强模型做教师（GPT-5.5 / Claude / Qwen-最新版 等）

import openai
from transformers import (
    AutoModelForCausalLM, AutoTokenizer,
    Trainer, TrainingArguments
)
import json

# 1. 准备 prompt 列表（你的任务场景）
prompts = [
    "请解释 Python 装饰器的原理。",
    "什么是大模型量化？",
    "Transformer 架构的核心组件有哪些？",
    # ... 实际 1000-10000 条
]

# 2. 调用教师模型生成回答（生成蒸馏数据集）
client = openai.OpenAI()
distill_data = []

for prompt in prompts:
    response = client.chat.completions.create(
        model="gpt-5.5",  # 换成你有的最强模型：GPT-5.5 / Claude / Qwen-最新版 等
        messages=[{"role": "user", "content": prompt}],
        temperature=0.7,
    )
    answer = response.choices[0].message.content
    distill_data.append({
        "instruction": prompt,
        "output": answer,
    })

# 保存蒸馏数据集
with open("distill_data.jsonl", "w", encoding="utf-8") as f:
    for item in distill_data:
        f.write(json.dumps(item, ensure_ascii=False) + "\n")

print(f"生成 {len(distill_data)} 条蒸馏数据")
```

```python
# 3. 用 SFT 微调学生模型（和普通微调完全一样）
from datasets import load_dataset
from trl import SFTTrainer

student_model = "Qwen/Qwen3.5-0.8B"  # 小模型

dataset = load_dataset("json", data_files="distill_data.jsonl", split="train")

# 构建训练样本
def format_func(example):
    return f"用户: {example['instruction']}\n助手: {example['output']}"

training_args = TrainingArguments(
    output_dir="./distill_output",
    per_device_train_batch_size=4,
    num_train_epochs=3,
    learning_rate=2e-5,
    logging_steps=10,
)

trainer = SFTTrainer(
    model=student_model,
    tokenizer=AutoTokenizer.from_pretrained(student_model),
    train_dataset=dataset,
    formatting_func=format_func,
    args=training_args,
)

trainer.train()
print("数据蒸馏微调完成！")
```

【注解】**SFT 还是 LoRA？** 这里用 SFTTrainer 只是演示方便。实际工业中更常用 **QLoRA** 做数据蒸馏——因为蒸馏的目的是"吸收教师的知识"，不是挑战模型极限，LoRA 的参数量足够了，而且省显存得多。"蒸馏数据用 LoRA 微调"在模块三会有完整实操。

【注解】这就是工业界的实际做法。DeepSeek-V4-Flash 的蒸馏标注、Qwen3.5-9B 的蒸馏训练，本质上都是**"用更强的模型造数据，然后 SFT 微调自己"。** 和模块三的 LoRA 微调流程完全一样，区别只是数据来源（蒸馏数据 vs 标注数据）。


### 数据蒸馏 vs 传统 KL 蒸馏

| 对比 | 传统 KL 蒸馏（BERT 时代） | 数据蒸馏（LLM 时代） |
|------|------------------------|-------------------|
| 教师怎么做 | 教师和学生**同时前向传播**，输出 softmax | 教师**单独生成文本**，存下来 |
| 学生学什么 | 模仿教师的概率分布（KL 散度） | 模仿教师的输出文本（SFT） |
| 是否需要标签 | 需要真实标签 + 软标签 | 只需要教师输出（无标签） |
| 训练速度 | 慢（teacher + student 同时跑） | 快（先离线造数据，再单独 SFT） |
| 工业应用 | 已淘汰（仅限小模型分类任务） | **最主流** |
| 代表 | DistilBERT、TinyBERT | DeepSeek-V4-Flash、Qwen3.5-9B |


### 传统蒸馏的原理（保留作为"概念遗产"）

在 BERT 时代（2019-2021），KL 散度蒸馏是标准做法。原理简单但已被 LLM 淘汰：

```
教师输出：猫 0.8 / 狗 0.15 / 车 0.05（软标签）
学生输出：猫 0.6 / 狗 0.3  / 车 0.1
目标：让学生的分布逼近教师 → KL 散度最小
```

"软标签"包含教师学到的"猫和狗比猫和车更相似"这类暗知识。小模型通过模仿分布获得教师的"经验"。

```python
# 传统蒸馏的 KL 散度损失（了解即可，LLM 时代已不这么用）
import torch.nn.functional as F

def traditional_distillation_loss(student_logits, teacher_logits, labels, alpha=0.5, temperature=4.0):
    hard_loss = F.cross_entropy(student_logits, labels)  # 真实标签
    soft_student = F.log_softmax(student_logits / temperature, dim=-1)
    soft_teacher = F.softmax(teacher_logits / temperature, dim=-1)
    soft_loss = F.kl_div(soft_student, soft_teacher, reduction='batchmean') * (temperature ** 2)
    return alpha * hard_loss + (1 - alpha) * soft_loss
```

【注解】**Temperature**（温度）是蒸馏中的超参数。温度越高，概率分布越平滑，学生能学到更多暗知识。训练时通常取 2-8，推理时温度=1。但在 LLM 时代，这个概念已融入 Sampling Temperature（生成文本时的多样性控制），和蒸馏本身关系不大了。


### 蒸馏的三种方式（保留，但数据蒸馏属于黑盒）

|方式|说明|典型应用|对应方法|
|---|---|---|---|
|**黑盒蒸馏**|只使用教师模型的输出（API 调用）|用更强模型（如 GPT / Claude）的输出训练小模型|**数据蒸馏（SFT）**|
|**白盒蒸馏**|可以访问教师模型的所有层输出|用 DeepSeek-V4-Pro 训练更小的 MoE|传统 KL 蒸馏（已淘汰）|
|**自蒸馏**|模型自己教自己（同一架构的不同版本）|Qwen3.5-397B → Qwen3.5-9B 同族|数据蒸馏 + 同族数据 |

【注解】2024-2026 年，所有开源小模型的"distillation"标注，本质上都是**数据蒸馏（黑盒）**——调用教师 API 或内部模型生成数据，然后 SFT 微调。这才是工业界的实际做法。

## 3.2 模型剪枝（Model Pruning）

模型剪枝的思路更直接：**删掉不重要的参数或结构**。


### 剪枝类型对比
|剪枝类型|粒度|硬件加速|质量损失|实现难度|
|---|---|---|---|---|
|**权重剪枝**|单个权重|❌ 无法加速|大|低|
|**神经元剪枝**|单个神经元/通道|✅ 可以加速|中|中|
|**层剪枝**|整个 Transformer 层|✅ 显著加速|小（保留关键层）|高|
|**头剪枝**|注意力头|✅ 可以加速|小（多头有冗余）|中|

【注解】在大模型时代，**剪枝的效果不如量化和蒸馏**。主要原因是大模型有"涌现能力"，删掉部分参数可能导致能力断层。目前业界更关注结构化剪枝（如删减 Transformer 层），头剪枝也有一定效果。


### 实操：结构化层剪枝（删除 Transformer 层）

> 这里演示**删除最后几层 Transformer**，效果立竿见影：参数量减少、推理加速。代码在 `structured_pruning.py`。


```python
# structured_pruning.py
# 结构化剪枝：删除 Qwen3.5-0.8B 最后 4 层 Transformer 层

import torch
import time
from transformers import AutoModelForCausalLM, AutoTokenizer

model_path = r"C:\Users\hp\Downloads\models\qwen3.5-0.8B"

# ===== 加载模型 =====
print("加载 Qwen3.5-0.8B ...")
model = AutoModelForCausalLM.from_pretrained(
    model_path, device_map="auto", dtype=torch.float16
)
tokenizer = AutoTokenizer.from_pretrained(model_path)
total_layers = len(model.model.layers)
print(f"原始层数: {total_layers}")

def count_params(model):
    return sum(p.numel() for p in model.parameters()) / 1e6

def generate(model, tokenizer, prompt, max_new=60):
    messages = [{"role": "user", "content": prompt}]
    
    text = tokenizer.apply_chat_template(
        messages,
        tokenize=False,
        add_generation_prompt=True)
    
    inputs = tokenizer([text], return_tensors="pt").to(model.device)
    t0 = time.time()
    with torch.no_grad():
        outputs = model.generate(**inputs, max_new_tokens=max_new)
    return tokenizer.decode(outputs[0], skip_special_tokens=True), time.time() - t0

# ===== 剪枝前 =====
prompt = "用一句话解释什么是机器学习："
params_before = count_params(model)
resp_before, t_before = generate(model, tokenizer, prompt)
print(f"剪枝前: {total_layers} 层, {params_before:.0f}M 参数, {t_before:.2f}s")

# ===== 删除最后 4 层 =====
model.model.layers = model.model.layers[:-4]
model.config.num_hidden_layers = len(model.model.layers)

params_after = count_params(model)
resp_after, t_after = generate(model, tokenizer, prompt)
print(f"剪枝后: {len(model.model.layers)} 层, {params_after:.0f}M 参数, {t_after:.2f}s")
```

## 3.3 总结：选什么压缩方案？

|场景|推荐方案|理由|
|---|---|---|
|H100 生产部署|**FP8 量化**（vLLM）|无损 + 速度翻倍|
|RTX 4090 本地跑|**AWQ INT4** 或 **GPTQ INT4**|省显存，质量损失小|
|想从大模型获得小模型|**数据蒸馏（SFT）** |用大模型输出造数据微调小模型|
|进一步压缩（量化后还想更小）|**量化 + 蒸馏**组合|两者互不冲突，可以叠加|
|剪枝|❌ 不推荐单独使用|质量损失大，建议配合蒸馏|
|没有 GPU 跑模型|**llama.cpp GGUF**（Q4_K_M）|纯 CPU 推理，属于部署工具，本模块不展开|

【注解】实际企业落地时，**量化**是最优先考虑的方案，因为简单（一行命令）、质量损失小、加速明显。如果需要更极致的压缩，再叠加**蒸馏**。剪枝目前主要在研究场景中使用。
