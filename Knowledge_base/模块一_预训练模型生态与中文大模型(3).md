## 1.1 预训练模型是什么

在深入各种架构之前，先理解一个核心概念：**预训练（Pre-training）**。

想象你正在学习用 LangGraph 搭建智能体：

- **预训练** = 先学会 State（状态）、Node（节点）、Edge（边）这些基础概念是什么，但还没系统学如何把它们串联起来做成一个完整 Agent
- **微调（Fine-tuning）** = 再专门学习如何把这些组件串联起来——如何设计状态流转、如何写条件边做路由、如何组装成能处理实际业务的智能体工作流

大语言模型也是这个道理。先在**海量文本**（互联网、书籍、百科）上"通读"一遍，学会语言的通用规律（语法、语义、常识），相当于掌握了 State/Node/Edge 这些"零件"；然后再针对具体任务（对话、翻译、代码）进行微调，相当于学会把这些零件组装成能完成特定任务的 Agent 工作流。

> **核心公式：预训练模型 = 通用语言知识 + 特定任务适配**

【注解】预训练（Pre-training）和微调（Fine-tuning）是 NLP 领域的两个核心阶段。预训练让模型"见过世面"，微调让模型"专精一行"。这种"预训练+微调"的范式，彻底改变了 NLP 的发展轨迹。

## 1.2 Transformer：所有大模型的"老祖宗"

2017 年 Google 发表的 **Attention Is All You Need** 论文，提出了 Transformer 架构，彻底改变了 NLP 领域。

【注解】Transformer 的核心创新是**自注意力机制（Self-Attention）**，它让模型在处理一个词时，能同时"看到"句子中所有其他词，并自动学习它们之间的关联强度。

### Transformer 的两大核心组件

| 组件 | 功能 | 一句话理解 |
|------|------|-----------|
| **Encoder（编码器）** | 双向理解上下文 | 像阅读理解，通读全文后再答题 |
| **Decoder（解码器）** | 单向生成序列 | 像写作文，从左到右逐字生成 |

```
Transformer 架构简图:

    Input → [Embedding] → [Encoder × N] → [Decoder × N] → [Linear] → Output
                              ↑                ↑
                         双向注意力         单向注意力
                         (看全部上下文)     (只能看左边已生成的)
```

【注解】Embedding（嵌入）是将文字转成数字向量的过程。模型只能处理数字，所以要把"猫"变成 [0.2, -0.5, 0.8, ...] 这样的向量。

## 1.3 主流架构对比：BERT vs GPT vs T5 vs BART

基于 Transformer，不同的研究者发展出了不同的"流派"，各有侧重：

### 四大架构核心区别

| 模型 | 架构 | 训练目标 | 擅长任务 | 一句话理解 |
|------|------|---------|---------|-----------|
| **BERT** | Encoder-only | 掩码语言模型（MLM） | 理解类任务 | 完形填空高手，擅长理解 |
| **GPT 系列** | Decoder-only | 因果语言模型（CLM） | 生成类任务 | 续写高手，只能看左边（Decoder-only） |
| **T5** | Encoder-Decoder | 片段掩码重构（遮住一段，还原整段） | 翻译、摘要 | 编码器双向看全局，解码器单向生成 |
| **BART** | Encoder-Decoder | 去噪自编码 | 文本生成、理解 | 全能选手，理解生成兼顾 |

【注解】**掩码语言模型（MLM）**：把句子中的某些词遮住（用 [MASK] 代替），让模型预测被遮住的词是什么。例如："今天 [MASK] 气很好" → 预测 "天"。

【注解】**因果语言模型（CLM）**：只根据前面的词预测下一个词。例如："今天天" → 预测 "气"。这就是为什么 GPT 擅长"续写"。

### BERT：双向理解的王者（2018）

BERT（Bidirectional Encoder Representations from Transformers）由 Google 发布，开启了"预训练+微调"的新时代。

```
BERT 的 MLM 训练方式:

    输入:  今天 [MASK] 气很好，我想去 [MASK] 园玩
           ↓
    预测:  今天 「天」 气很好，我想去 「公」 园玩
           ↑                ↑
        双向上下文         双向上下文
        (看整句话)        (看整句话)
```

**BERT 的核心预训练任务**：

- **MLM（掩码语言模型）**：随机遮住 15% 的词，让模型预测。例如："今天 [MASK] 气很好" → 预测 "天"。

【注解】BERT 的"双向"意味着在处理每个词时，可以同时关注它左边和右边的所有词。这与 GPT 的"单向"（只能看左边）形成鲜明对比。

### GPT 系列：生成能力的进化史

GPT（Generative Pre-trained Transformer）由 OpenAI 发布，走了一条与 BERT 完全不同的路。

| 版本 | 发布时间 | 参数量 | 关键突破 |
|------|---------|--------|---------|
| GPT-1 | 2018 | 1.17亿 | 证明生成式预训练有效 |
| GPT-2 | 2019 | 15亿 | 零样本能力初现 |
| GPT-3 | 2020 | 1750亿 | 涌现能力，上下文学习 |
| GPT-3.5 | 2022 | 未公开 | ChatGPT 诞生，RLHF 引入 |
| GPT-4 | 2023 | 未公开 | 多模态，推理能力飞跃 |
| GPT-5 | 2025 | 未公开 | 统一架构，内置推理，实时路由系统 |
| GPT-5.5 | 2026 | 未公开 | 全新预训练，Agent 能力飞跃，当前最强模型 |

```
GPT 的自回归生成方式:

    第1步: 今天 → 预测 "天"
    第2步: 今天天 → 预测 "气"
    第3步: 今天天气 → 预测 "很"
    第4步: 今天天气很 → 预测 "好"
    ...
    
    只能看左边已生成的，不能偷看右边未来的
```

【注解】**因果掩码（Causal Mask）**：GPT 是纯 Decoder 架构，没有独立编码器。输入文本经过 Embedding 后直接进入 Decoder Block，每层都使用因果掩码——每个 token 只能关注自己及之前位置的 token。例如输入"今天天气很好"，处理"很"时能看到"今天天气"，但看不到"好（未来）"。

【注解】**RLHF（Reinforcement Learning from Human Feedback）**：人类反馈强化学习。让人类标注员对模型的多个回答打分，然后用强化学习训练模型生成更符合人类偏好的回答。这是 ChatGPT"会说话"的关键。

【注解】**涌现能力（Emergent Abilities）**：当模型大到一定程度（约 100B 参数），会突然展现出小模型没有的能力，如上下文学习（In-context Learning）、思维链（Chain-of-Thought）等。

### 为什么主流大模型都用 Decoder-only？

尽管 Encoder-Decoder（T5）和 Encoder-only（BERT）各有优势，但近年的大模型（GPT、LLaMA、Qwen、DeepSeek 等）几乎都采用 Decoder-only 架构，原因如下：

1. **扩展性更好**：单塔结构更简单，参数量一定时计算效率更高，Scaling Law 表现更优
2. **上下文学习（ICL）自然涌现**：因果掩码的"预测下一个词"任务天然适合上下文学习，这是 GPT-3 之后最重要的能力突破
3. **泛用性强**：生成任务可以覆盖理解任务（分类只需看生成结果），但反之不然——一个架构搞定所有任务
4. **推理效率高**：单塔的 KV Cache 更友好，推理速度快；预训练目标简单统一（预测下一个 token）
5. **对话能力天生匹配**：因果语言模型的训练方式与对话/续写天然一致，微调成本低

一句话：**Decoder-only 在"大力出奇迹"的 Scaling Law 下，性价比最高，且涌现出的能力最实用。**

### T5：统一框架的尝试（2019）

T5（Text-to-Text Transfer Transformer）由 Google 提出，核心理念是：**把所有 NLP 任务都转换成"文本到文本"的问题**。

```
T5 的统一范式:

    翻译:     "translate English to German: How are you?" → "Wie geht es dir?"
    摘要:     "summarize: 长篇文章内容..." → "摘要内容"
    分类:     "cola sentence: The movie was good." → "acceptable"
    问答:     "question: 谁发明了电灯? context: 爱迪生..." → "爱迪生"
    
    所有任务都是: 输入文本 → 输出文本
```

【注解】T5 的"Text-to-Text"思想将所有 NLP 任务统一为"输入文本→输出文本"的格式，确实简化了训练和部署。但从架构角度看，**GPT 的 Decoder-only 方案更为简洁通用**——同样的统一范式（输入文本→生成文本），却不需要编码器，架构更简单，扩展性更强。

【注解】T5 采用**编码器-解码器架构**，两者的注意力机制不同：

- **编码器（Encoder）**：使用**双向注意力**（无掩码），输入文本的所有 token 互相可见，类似 BERT
- **解码器（Decoder）**：使用**因果注意力（Causal Mask）**，每个 token 只能关注自己及之前的位置，与 GPT 相同

这是 T5 与 GPT 的关键架构差异：**T5 的编码器能全面理解输入上下文，而 GPT 没有编码器，从头到尾都是单向因果掩码。**

### BART：去噪自编码的融合（2019）

BART（Bidirectional and Auto-Regressive Transformers）由 Facebook 提出，结合了 BERT 的双向编码器和 GPT 的自回归解码器。

**BART 的去噪训练方式**：

- 对输入文本进行各种"破坏"（删词、打乱、掩码）
- 让模型学习如何"还原"原始文本

```
BART 的去噪训练:

    原始:  我今天要去公园玩
    破坏:  我今天 [MASK] 公园 [MASK]
           ↓
    还原:  我今天要去公园玩
           ↑
        编码器双向理解 + 解码器自回归生成
```

【注解】BART 的设计让它在文本生成任务（摘要、翻译）和理解任务（分类、抽取）上都有不错表现，是一个"全能型"架构。

---

### 小结

| 模型 | 核心特点 | 适用场景 | 代表产品 |
|------|---------|---------|---------|
| BERT | 双向理解，完形填空 | 文本分类、NER | 搜索排序、语义匹配 |
| GPT | 单向生成，自回归 | 文本生成、对话、代码 | ChatGPT、Copilot |
| T5 | Text-to-Text 统一 | 翻译、摘要、多任务 | Google 翻译底层 |
| BART | 编码器+解码器融合 | 生成+理解兼顾 | 文本摘要、纠错 |

**核心思想**：没有最好的架构，只有最适合的架构。理解任务需求，选择对应的模型类型。

### 加载 BERT 进行文本分类（Encoder-only 架构代表）

【注解】`AutoTokenizer` 和 `AutoModelForSequenceClassification` 是 Transformers 库的"自动选择"类。它们会根据模型名称自动加载对应的分词器和模型架构。

```python
# 【注解】transformers：Hugging Face 的核心库，提供预训练模型的统一接口
from transformers import AutoTokenizer, AutoModelForSequenceClassification
import torch

# ---------- 第①步：加载分词器和模型 ----------
# 【注解】bert-base-chinese：Google 发布的中文 BERT 基础版，约 1 亿参数
# 虽然是 2018 年的模型，但仍是理解类任务的标杆
model_name = "bert-base-chinese"

# 【注解】Tokenizer 负责将文本转成模型能理解的数字（token IDs）
tokenizer = AutoTokenizer.from_pretrained(model_name)

# 【注解】AutoModelForSequenceClassification 是用于分类任务的模型
# num_labels=2 表示二分类（正面/负面）
model = AutoModelForSequenceClassification.from_pretrained(model_name, num_labels=2)

# ---------- 第②步：准备输入文本 ----------
text = "这家餐厅的服务非常好，食物也很美味！"

# 【注解】return_tensors="pt" 表示返回 PyTorch 张量格式
# padding=True 自动填充到统一长度，truncation=True 超长截断
inputs = tokenizer(text, return_tensors="pt", padding=True, truncation=True)

print("Tokenized input:", inputs)
print("Input IDs shape:", inputs['input_ids'].shape)
```

输出：
```
Tokenized input: {'input_ids': tensor([[ 101,  682,  677,  ... ,  102]]), 'attention_mask': tensor([[1, 1, 1, ... , 1]])}
Input IDs shape: torch.Size([1, 23])
```

【注解】**`attention_mask` 的作用**：用于处理**批次内不同长度的文本**。当一次推理多条文本（batch > 1）时，短的文本需要填充（padding）到与最长文本相同的长度。`attention_mask` 告诉模型：标记为 1 的位置是真实内容，标记为 0 的位置是填充的"空字符"，计算时忽略。因为上面代码 batch_size=1（只有一条文本），所有位置都是 1，没有区别。但**在批量推理或训练时，attention_mask 至关重要**——没有它会将填充位的无效信息混入注意力计算，降低模型效果。

```python
# ---------- 第③步：模型推理 ----------
# 【注解】torch.no_grad() 关闭梯度计算，节省内存，推理时必需
with torch.no_grad():
    # 【注解】字典解包（**inputs）自动传入所有参数
    # 完整写法等价于：model(input_ids=inputs['input_ids'], attention_mask=inputs['attention_mask'])
    outputs = model(**inputs)
    
    # 【注解】logits 是模型输出的原始分数，形状为 [batch_size, num_labels]
    logits = outputs.logits
    
    # 【注解】softmax 将分数转换为概率分布
    probabilities = torch.softmax(logits, dim=-1)
    
    # 【注解】argmax 取概率最大的类别索引
    predicted_class = torch.argmax(probabilities, dim=-1).item()

print(f"Logits: {logits}")
print(f"Probabilities: {probabilities}")
print(f"Predicted class: {predicted_class} ({'正面' if predicted_class == 1 else '负面'})")
```

输出：
```
Logits: tensor([[ 0.5234, -0.1234]])
Probabilities: tensor([[0.6789, 0.3211]])
Predicted class: 0 (负面)

# 注意：这是随机初始化的分类头结果，实际使用需要微调
```

【注解】上面的结果是随机初始化的分类头输出的，不代表真实语义。实际应用中，需要在标注数据上微调（Fine-tune）模型。

### 使用 Qwen3.5-0.8B 演示 GPT 类模型的文本生成（Decoder-only 架构）

【注解】前面 1.3 节讲过，GPT 系列是 Decoder-only 架构，使用因果掩码做自回归生成。这里用 **Qwen3.5-0.8B** 来演示——**Qwen 与 GPT 同属 Decoder-only 架构**，核心都是 `AutoModelForCausalLM` + `generate()`，**代码完全通用**。选择 Qwen 而非 GPT-2 的原因是：中文更好、模型更小（1.6GB vs GPT-2 的 500MB 英文模型）、下载更快。

```python
import torch
from transformers import AutoTokenizer, AutoModelForCausalLM

# ---------- 第①步：加载 Qwen3.5-0.8B ----------
# 【注解】Qwen/Qwen3.5-0.8B：阿里 Qwen3.5 系列最小版本，约 0.8B 参数
# FP16 仅约 1.6GB，4GB 显卡甚至 CPU 均可运行
model_name = r"C:\Users\hp\Downloads\models\qwen3.5-0.8B"

tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    device_map="auto",
    torch_dtype=torch.float16
).eval()

# ---------- 第②步：对话格式处理 ----------
query = "你好，请简单介绍一下你自己。"
messages = [{"role": "user", "content": query}]

# 【注解】apply_chat_template 自动处理对话格式（不同模型格式不同）
# 为什么这里用 apply_chat_template，而前面的 GPT-2 demo 不用？
#   - GPT-2 是基座模型 (base model)→ 训练数据就是原始文本 → 直接 tokenizer(prompt) 即可
#   - Qwen 是指令微调模型 (chat model) → 训练时用的是带角色标记的对话数据
#     例如实际输入 Token 序列是：
#     "<|im_start|>user\n你好，请简单介绍一下你自己。<|im_end|>\n<|im_start|>assistant\n"
#     apply_chat_template 就是自动把 messages 列表拼成这种格式
#   - 如果 Qwen 不加这些标记直接传"你好"，模型会困惑，回答质量很差
text = tokenizer.apply_chat_template(
    messages,
    tokenize=False,
    add_generation_prompt=True
)

inputs = tokenizer([text], return_tensors="pt").to(model.device)

# ---------- 第③步：自回归生成 ----------
with torch.no_grad():
    outputs = model.generate(
        **inputs,
        max_new_tokens=256,
        temperature=0.7,
        top_p=0.9,
        do_sample=True,
    )

response = tokenizer.decode(outputs[0], skip_special_tokens=True)
print(response)
```

输出：
```
user
你好，请简单介绍一下你自己。
assistant
你好！我是 Qwen3.5，一个由阿里云开发的大语言模型。我可以回答问题、提供帮助、进行对话交流。我的知识截止于训练数据的时间点，目前尚不支持多模态功能。有什么我可以帮你的吗？
```

【注解】`generate` 方法参数说明：

**核心参数**
- **`max_new_tokens`**：最多生成的新 token 数。控制回答长度。

**采样参数（控制生成"确定性 vs 创造性"）**
- **`do_sample`**：是否启用随机采样。
  - `False`（默认）→ **贪心解码**：每次选概率最高的词。结果稳定但容易重复、单调。
  - `True` → **随机采样**：按概率分布随机抽词，结果有变化但偶尔漏选好词。此时需配合 temperature/top_p 约束。
- **`temperature`**：**概率分布缩放系数**，控制随机性强度（前提：do_sample=True）。
  - `t→0`（如 0.1）：分布被压尖 → 高概率词几乎必选 → 接近贪心解码
  - `t→1`（如 2.0）：分布被拉平 → 低概率词也有机会 → 更随机、更有创意
  - 推荐范围：0.6~0.9。写作创意用 0.9，事实问答用 0.1~0.3。
- **`top_p`**：**核采样 (Nucleus Sampling)** — 只从累计概率达到 p 的最小候选集里采样（前提：do_sample=True）。
  - `top_p=0.9`：把概率最高的词排好，从小到大累加，直到累计概率≥0.9，只从这批词中采样
  - **直观理解**："不看全部词，只看靠谱的词中排名最靠前的 90%"
  - 作用：自动筛掉长尾的低概率奇怪词（如错别字、无关词），同时保留一定的多样性
  - 与 top_k 的区别：`top_k` 固定取前 K 个（如 top_k=50），`top_p` 动态根据概率分布调整候选数

**典型组合速查**

| 用途 | do_sample | temperature | top_p | 效果 |
|------|-----------|-------------|-------|------|
| 事实问答 / 翻译 | False | 忽略 | 忽略 | 稳定、准确 |
| 日常聊天 | True | 0.7 | 0.9 | 平衡 |
| 创意写作 | True | 0.9 | 0.95 | 丰富多样 |
| 代码补全 | True | 0.2 | 0.8 | 精确但有小变化 |

【注解】以上代码是**所有 Decoder-only 模型（GPT、Qwen、LLaMA、DeepSeek）的统一推理范式**。`bert-base-chinese` 和 `Qwen/Qwen3.5-0.8B` 的代码结构完全一样——这就是 Hugging Face 统一 API 的魅力。**模块三的微调代码同样在这两个模型上通用**。

## 2.1 Hugging Face 是什么

Hugging Face 是当今最大的 AI 开源社区和平台，被称为"AI 界的 GitHub"。

**Hugging Face 的核心产品**：

| 产品 | 功能 | 一句话理解 |
|------|------|-----------|
| **Transformers** | 模型库 | 统一接口，加载各种预训练模型 |
| **Hub** | 模型托管平台 | 下载和分享模型的"应用商店" |
| **Datasets** | 数据集库 | 方便加载和处理训练数据 |
| **Tokenizers** | 分词器库 | 快速高效地将文本转成 token |
| **Accelerate** | 训练加速 | 让模型训练更快、更省显存 |
| **PEFT** | 高效微调 | 用更少资源微调大模型 |

【注解】Hugging Face 的成功在于**标准化**。无论模型内部架构如何不同，都通过统一的 `from_pretrained` 和 `save_pretrained` 接口操作，大大降低了使用门槛。

---

## 2.2 Transformers 库核心 API

Transformers 库的设计遵循"约定优于配置"原则，核心 API 非常简洁。

### 三大核心类

| 类名 | 用途 | 典型用法 |
|------|------|---------|
| `AutoTokenizer` | 自动加载分词器 | `AutoTokenizer.from_pretrained("Qwen/Qwen3.5-0.8B")` |
| `AutoModel` | 自动加载基础模型 | `AutoModel.from_pretrained("Qwen/Qwen3.5-0.8B")` |
| `AutoModelForXXX` | 加载特定任务模型 | `AutoModelForCausalLM`, `AutoModelForSequenceClassification` |

【注解】`Auto` 前缀表示"自动识别"。你不需要告诉它这是 BERT 还是 Qwen，它会根据模型配置文件自动选择正确的类。

### 模型加载的完整流程

```python
# 【注解】transformers 通过 AutoModel / AutoConfig 提供统一的模型加载接口，
# 多数标准属性（vocab_size、hidden_size 等）可直接访问；
# 但对于多模态模型（如 Qwen3.5），文本配置可能嵌套在子对象中，需要具体查看 config 结构。
from transformers import AutoTokenizer, AutoModel, AutoConfig

# ---------- 第①步：查看模型配置 ----------
# 【注解】AutoConfig 可以查看模型的配置信息，无需加载完整模型（省内存）
model_name = r"C:\Users\hp\Downloads\models\qwen3.5-0.8B"
config = AutoConfig.from_pretrained(model_name)

# 【注解】Qwen3.5 是多模态模型，文本配置在 text_config 子对象中
text_cfg = config.text_config
print(f"模型类型: {config.model_type}")
print(f"词表大小: {text_cfg.vocab_size}")
print(f"隐藏层维度: {text_cfg.hidden_size}")
print(f"注意力头数: {text_cfg.num_attention_heads}")
print(f"Transformer 层数: {text_cfg.num_hidden_layers}")
print(f"最大序列长度: {text_cfg.max_position_embeddings}")
```

输出：
```
模型类型: qwen3_5
词表大小: 248320
隐藏层维度: 1024
注意力头数: 8
Transformer 层数: 24
最大序列长度: 262144
```

【注解】**词表大小（vocab_size）**：模型认识多少个字/词。Qwen3.5-0.8B 的词表约 24.8 万，远超 BERT 的 2.1 万，原因是 Qwen 是多语言模型，需要容纳中英文、代码、特殊符号、图像/视频占位 token 等。

【注解】**隐藏层维度（hidden_size）**：模型内部向量的长度。Qwen3.5-0.8B 用1024 维，代表每个 token 被编码成1024 维的向量。维度越高，表达能力越强，但计算量也越大。同一系列的更大模型（如 4B/9B/27B）hidden_size 会更大，所以也越"聪明"、越慢。

```python
# ---------- 第②步：加载分词器 ----------
# 【注解】分词器负责：文本 → Tokens → Input IDs
tokenizer = AutoTokenizer.from_pretrained(model_name)

# 查看分词器的信息
print(f"词表大小: {tokenizer.vocab_size}")
print(f"特殊标记: {tokenizer.special_tokens_map}")

# 测试分词
text = "你好，世界！"
tokens = tokenizer.tokenize(text)
input_ids = tokenizer.convert_tokens_to_ids(tokens)
print(f"\n原始文本: {text}")
print(f"分词结果 (tokens): {tokens}")
print(f"Token IDs: {input_ids}")

# 一次性完成：文本 → Input IDs（Qwen 不自动添加特殊标记，与 BERT 不同）
encoded = tokenizer(text)
print(f"\n完整编码: {encoded}")
print(f"解码还原: {tokenizer.decode(encoded['input_ids'])}")
```

输出：
```
表大小: 248044
特殊标记: {'eos_token': '<|im_end|>', 'pad_token': '<|endoftext|>', 'audio_bos_token': '<|audio_start|>', 'audio_eos_token': '<|audio_end|>', 'audio_token': '<|audio_pad|>', 'image_token': '<|image_pad|>', 'video_token': '<|video_pad|>', 'vision_bos_token': '<|vision_start|>', 'vision_eos_token': '<|vision_end|>'}

原始文本: 你好，世界！
分词结果 (tokens): ['ä½łå¥½', 'ï¼Į', 'ä¸ĸçķĮ', 'ï¼ģ']
Token IDs: [109266, 3709, 96748, 6115]
ID → token: ['ä½łå¥½', 'ï¼Į', 'ä¸ĸçķĮ', 'ï¼ģ']
ID → 还原原文: 你好，世界！

完整编码: {'input_ids': [109266, 3709, 96748, 6115], 'attention_mask': [1, 1, 1, 1]}
解码还原 (带特殊标记): 你好，世界！
```

【注解】Qwen3.5 的分词器与 BERT 有几点关键不同：

1. **分词方式不同**：Qwen 用 BPE（字节对编码），以子词为单位；BERT 用字符级 token，一个字一个 token
2. **`tokenize()` 输出乱码？用 ID 看**：BPE 的 token 存储的是字节级子词，在终端直接打印会显示成乱码（如 `ä½ł`），但 `convert_ids_to_tokens()` 可显示可读形式（如 `▁你`、`好`）
3. **不自动添加特殊标记**：`tokenizer(text)` 直接返回原始 token，没有 [CLS]/[SEP]；而 BERT 会自动在首尾加 [CLS] 和 [SEP]
4. **特殊标记是 ChatML 格式**：
   - `<|im_end|>` — 对话结束标记（Qwen 的 eos_token）
   - `<|endoftext|>` — 段落/PAD 填充标记
   - `<|image_pad|>`、`<|video_pad|>`、`<|audio_pad|>` — 多模态占位符
5. **词表更大（24.8 万 vs BERT 的 2.1 万）**：因为 Qwen 是多语言模型，且 BPE 需要存储大量子词单元

```python
# ---------- 第③步：加载模型并获取隐藏状态 ----------
import torch

# 【注解】AutoModel 加载基础模型，输出隐藏状态（hidden states），不含任务头
model = AutoModel.from_pretrained(model_name)
model.eval()  # 设置为评估模式，关闭 Dropout

# 准备输入
text = "自然语言处理是人工智能的重要方向。"
inputs = tokenizer(text, return_tensors="pt", padding=True)

# 推理
with torch.no_grad():
    outputs = model(**inputs)

# 【注解】last_hidden_state 是最后一层的输出，形状为 [batch_size, seq_len, hidden_size]
last_hidden_state = outputs.last_hidden_state

print(f"输入序列长度: {inputs['input_ids'].shape[1]}")
print(f"隐藏状态形状: {last_hidden_state.shape}")
print(f"每个 token 被编码为 {last_hidden_state.shape[-1]} 维向量")

# [CLS] 标记的向量通常用于表示整个句子
cls_embedding = last_hidden_state[:, 0, :]  # 取第一个位置token的embedding 向量
print(f"\n[CLS] 向量形状: {cls_embedding.shape}")
print(f"[CLS] 向量前5维: {cls_embedding[0, :5]}")
```

输出：
```
输入序列长度: 8
隐藏状态形状: torch.Size([1, 8, 1024])
每个 token 被编码为 1024 维向量

[CLS] 向量形状: torch.Size([1, 1024])
[CLS] 向量前5维: tensor([ 0.6523, -1.4297,  0.0339,  0.5820,  1.9375], dtype=torch.bfloat16)
```

【注解】`last_hidden_state` 是 Transformer 最后一层的输出，包含了每个 token 的上下文表示。形状 `[batch_size, seq_len, hidden_size]` 分别表示：批次大小、序列长度、向量维度

## 2.3 Pipeline：一行代码搞定任务

对于常见任务，Transformers 提供了更高级的 Pipeline API，将分词、推理、后处理封装在一起。

```python

from transformers import pipeline

# ---------- 情感分析任务 ----------
# 【注解】task="sentiment-analysis" 自动选择合适的模型和分词器
# bert-base-chinese 本身没有情感分析能力，必须用微调过的模型才能得到正确且稳定的结果。
classifier = pipeline("sentiment-analysis", model=model_path, device=device)
```

#### GLM 的架构：Prefix-LM（前缀解码器）

> 注意：GLM **既不是** GPT/Qwen 那种纯 Decoder-only，**也不是** T5 那种编码器+解码器双塔架构。
>
> 要理解 GLM，先分清两个概念：
> - **Transformer 架构（原始论文）**：**两套** Transformer Block——一套编码器（N 层），一套解码器（N 层）。T5 和 BART 沿用这种。
> - **Transformer Block**：自注意力 + 前馈网络，一个基本的积木块。GPT/Qwen 只用 **1 套**（Decoder Block × N 层），BERT 也只用 **1 套**（Encoder Block × N 层）。
>
> **GLM 的做法**：和 GPT/Qwen/BERT 一样，只用了 **1 套** Transformer Block（N 层），每层内部通过不同的注意力掩码让不同位置"扮演"不同角色：
> - 输入位置 → 双向掩码（像 BERT 的编码器角色）
> - 生成位置 → 因果掩码（像 GPT 的解码器角色）
>
> 所以 GLM 是**单塔（1 套参数，靠掩码切换角色）**，而 T5 是**双塔（2 套参数，编码器干编码、解码器干解码）**。

**与 T5（双塔）的关键区别**：

| | T5（双塔） | GLM（单塔） |
|---|---|---|
| 用了几个 Transformer Block 堆栈 | **2 套：编码器 Block + 解码器 Block** | **1 套** Transformer Block，靠掩码区分角色 |
| 数据流 | 输入 → 编码器塔 → 解码器塔 → 输出 | 输入 → 同一个塔 → 输出 |
| 通俗类比 | 两个人：一个人负责听（编码器），一个人负责说（解码器） | 一个人：听的时候全局看，说的时候按顺序说 |

**架构定位**：

| 维度 | GPT / Qwen | GLM | BERT |
|------|-----------|-----|------|
| 有几套参数 | **1 套**（Decoder） | **1 套**（Transformer） | 1 套（Encoder） |
| 输入部分注意力 | **单向（因果）** | **双向（互看）** ✅ | 双向 |
| 生成部分注意力 | 单向（因果） | 单向（因果），但能看到全部输入 | 不支持生成 |
| 分类为 | Decoder-only | **Prefix-LM** | Encoder-only |

**【GLM 的 KV Cache 和显存问题】你可能担心：GLM 输入部分是双向的，生成新 token 时是不是要重新双向计算？**

答案是不用。GLM 的工作分两阶段：

- **预填阶段（做一次双向）**：输入"今天天气怎么样" → 所有 token 之间双向互看一次，结果存入 KV Cache。这一步确实比 GPT 的因果预填（三角矩阵）计算量大一些，但**只说一遍**。
- **生成阶段（因果自回归）**：每生成一个新 token（如"今"），它只做因果注意力——看自己和左边已经算好的 KV Cache，**不会触发重新双向计算**。和 GPT 的行为完全一样，KV Cache 大小也完全一样。

简单说：**双向理解只发生在"听懂问题"那一刻，一旦搞清楚了，生成回答就按顺序说，不需要每说一个字就重新理解一遍。** 所以 GLM 在推理时不会比 GPT 多占用 KV Cache。

**核心：2D 分块 Attention Mask**

GLM 的注意力不是简单的全双向或全因果，而是一个 **分块矩阵**（不同区域用不同掩码）：

```text
输入:   「今天 [MASK] 气很好，我想去 [MASK] 园玩」
              ↑                 ↑
           Part A(上下文)    Part B(被掩码的 span1)    Part C(被掩码的 span2)

注意力矩阵（行=Query，列=Key/Value）:

                Part A    Part B    Part C
              ┌─────────────────────────────┐
   Part A     │  双向可见    ❌ 遮挡     ❌ 遮挡  │
              │                             │
   Part B     │  双向可见  因果(自回归)  ❌ 遮挡  │
              │                             │
   Part C     │  双向可见    ❌ 遮挡   因果(自回归) │
              └─────────────────────────────┘
```

**逐区域解读**：

1. **Part A（上下文部分）** — 输入中未被掩码的 token
   - token 之间 **双向可见**（像 BERT），充分理解上下文语义
   - **看不到** Part B/C 被掩码的内容（避免作弊）

2. **Part B / Part C（被掩码的 span）** — 需要预测的连续片段
   - **能看到整段 Part A**（利用全部上下文信息）— 这相当于"cross-attention"
   - **同一 span 内部是因果掩码**（像 GPT，从左到右逐词生成）
   - **不同 span 之间互相遮挡**（Part B 看不到 Part C，反之亦然）

**预训练时的自回归填空工作流**：

```text
第1步：选一段文本，随机挑几个 span 替换成 [MASK]
  原文: 「今天天气很好，我想去公园玩」
         ↓
  输入: 「今天 [MASK] 气很好，我想去 [MASK] 园玩」

第2步：模型同时预测所有被掩码的 span（因为 span 间互不可见，可以并行）
  
  Span 1 推测: "天" → 看到 Part A → "天气" → 看到 Part A + "天" → ...
  Span 2 推测: "公" → 看到 Part A → "公园" → 看到 Part A + "公" → ...
           ↑                      ↑                    ↑
       双向看上下文            利用上下文+已生成内容    自回归生成

第3步：关键区别
  - BERT： "天"独立猜、"气"独立猜 → 可能猜出"天晴"与"气冷"不匹配
  - GLM：  先产生"天"，基于"天"产生"气" → 生成连贯的"天气" ✅
  - GPT：  不能做填空，只能从左到右续写
```

**推理阶段：转为标准生成**

```text
推理时 GLM 自动退化为标准自回归生成：

  用户问：「什么是人工智能？」
  
  模型先用 [gMASK] 标记句首位置，
  然后像 GPT 一样逐 token 生成回答。
  此时整个序列作为一个"生成 span"处理，
  使用因果掩码。
```

【注解】**为什么叫 Prefix-LM？** 输入部分扮演了"前缀"（prefix）角色，使用双向注意力充分理解上下文；生成部分使用因果注意力从左到右生成。一个塔同时扮演了"理解"和"生成"两个角色——这就是 GLM 的精妙之处。

【注解】**[gMASK] vs [MASK]**：[gMASK] 放在句首用于生成类任务（对话、续写）；[MASK] 放在句中用于填空。推理时对话统一用 [gMASK] 开头，模型自动进入生成模式。

【注解】虽然架构不同，但 Hugging Face 对 GLM 也提供了 `AutoModelForCausalLM` 接口（内部是 `GLMModel`），所以代码接口与 GPT/Qwen 一致：`from_pretrained` + `apply_chat_template` + `generate`。只需换 `model_name` 即可。

### Qwen3.5：阿里 2026 开源旗舰家族（本课程核心模型）

Qwen3.5 是阿里巴巴 2026 年 2 月发布的最新大模型家族，共开源 8 款模型（0.8B~397B），全部 Apache 2.0 协议。

#### Qwen3.5 的架构：标准的 Decoder-only 因果语言模型

Qwen3.5 与 GPT 系列同属 **Decoder-only 架构**，其基本单元由以下组件构成：

```text
输入 Embedding
    ↓
┌─────────────────────────────────┐
│      Decoder Block × N 层         │
│  ┌───────────────────────────┐   │
│  │  RMS LayerNorm             │   │
│  │       ↓                    │   │
│  │  Self-Attention（因果掩码）   │   │  ← 核心：每个 token 只看自己和左边
│  │  ┌─────────────────────┐   │   │
│  │  │ Qwen3.5: GDN + Softmax 混合  │   │   │  ← 75% 层 GDN(O(n)) + 25% 层 softmax
│  │  └─────────────────────┘   │   │
│  │       ↓                    │   │
│  │  Residual +                │   │  ← 残差连接
│  │       ↓                    │   │
│  │  RMS LayerNorm             │   │
│  │       ↓                    │   │
│  │  FFN (前馈网络)             │   │  ← SwiGLU 激活函数
│  │       ↓                    │   │
│  │  Residual +                │   │
│  └───────────────────────────┘   │
│           × N 层                  │
└─────────────────────────────────┘
    ↓
输出 Logits → Softmax → 下一个 token
```

【注解】**Decoder-only 的核心是因果掩码（Causal Mask）**：每个 token 只能关注自己及之前位置的 token，不能"偷看"未来的 token。这与 BERT 的双向注意力完全相反。这也是为什么 GPT/Qwen 只能做"续写"（从左到右逐字生成），而 BERT 能做"完形填空"（利用上下文）。

【注解】**RMS LayerNorm** 是简化版的 LayerNorm，去掉了均值中心化步骤，只做方差归一化。计算更快，效果几乎一致。**残差连接（Residual Connection）** 让梯度可以直接流过整个网络，解决了深层网络的梯度消失问题。

#### Qwen3.5 的核心创新：Gated DeltaNet（GDN）混合注意力

> **GDN**（Gated DeltaNet，门控 Delta 网络）是 Qwen3.5 首创引入生产模型的新型注意力机制，此前 Mamba、DeltaNet 等线性注意力方案虽有探索，但首次在大规模生产模型中落地并取得实际加速效果的**是 Qwen3.5**。
>
> 下文还会提到 **GQA**（分组查询注意力），但 GQA 并非 Qwen3.5 的创新——它最早由 Google 在 2023 年提出，Qwen3 和 LLaMA 2 等更早的模型就已经使用了。这里讲解 GQA 是为了完整理解 Qwen3.5 的优化手段。

##### GDN 为什么是 O(n)：从"两两比较"到"递推更新"

要理解 GDN 为什么是 O(n)，先看 Softmax 为什么是 O(n²)。

**Softmax 注意力的计算方式**：每来一个新 token，它要跟**所有已有 token 做一次两两相似度计算**。

```text
例如已有 n=1000 个 token：

  第 1 个 token → 跟 0 个 token 比
  第 2 个 token → 跟 1 个 token 比（第1个）
  第 3 个 token → 跟 2 个 token 比（第1、2个）
  ...
  第 1000 个 token → 跟 999 个 token 比
  
  总计算量 ≈ 0 + 1 + 2 + ... + 999 ≈ 50 万次两两比较 ← O(n²/2)
  其中"两两比较"就是计算 Q·K^T 这一步，这是 Softmax 注意力的核心操作。

扩展到 n=2000：总计算量 ≈ 200 万次两两比较 ← 翻 4 倍
```

直观理解：**每个新来的人都要跟全屋子的人挨个握手**——来第 1000 个人时，他要握 999 次手；来第 2000 个人时，他要握 1999 次手。越往后越慢。

**GDN 的计算方式**：**不做两两比较**，用状态递推替代。

GDN 的核心思路是：不要每次来一个新 token 就重新比较所有旧 token，而是**让旧 token 的信息浓缩成一个"状态"，新 token 只跟这个状态交互**。

```text
旧方案（Softmax）：
  新 token → 跟 token1 比 → 跟 token2 比 → 跟 token3 比 → ... → 跟 tokenN 比
              ↑                                     ↑
        第1000个要跟999个比              O(n²)：n 数越大，每个人的比较次数越多

新方案（GDN）：
  新 token → gate(state_old, new_token) → state_new
              ↑
        每来一个 token 只多做一次递推，不管前面有多少旧 token
        O(n)：n 数越大，递推步数线性增长，但每步的计算量不变
```

**这个"状态"到底是什么？**

状态就是一个向量（比如 2048 维），它代表了到目前为止所有 token 的"摘要"。每来一个新 token，用一个**简单的门控公式**更新这个摘要：

```text
state_new = α × state_old + (1-α) × new_token
  ↑           ↑                     ↑
新摘要     保留多少旧信息          加入多少新内容

  α 是一个 0~1 之间的门控系数，由模型自动学习。
  比如 α=0.9 → 保留 90% 旧摘要，新 token 只贡献 10%
```

整个过程就像**写会议纪要**：

```text
第1个 token：「今」 → 摘要 = "今"
第2个 token：「天」 → 摘要 = 0.9 × "今" + 0.1 × "天" ≈ "今·天"
第3个 token：「天」 → 摘要 = 0.9 × "今天" + 0.1 × "天" ≈ "今天天"
...
第1000个 token：「猫」→ 摘要 = 0.9 × 旧摘要 + 0.1 × "猫"

每步的运算量完全一样：一次加权平均 + 一次矩阵乘法
```

**关键点总结**：

| | Softmax | GDN |
|---|---|---|
| 核心操作 | **两两比较**（Q·K^T） | **递推更新**（门控合并） |
| 第 n 个 token 来了做什么 | 跟前面 n-1 个 token **逐个比较** | 跟一个状态向量**一步合并** |
| 总计算量 | 0+1+2+...+(n-1) ≈ n²/2 → **O(n²)** | 1+1+...+1 = n → **O(n)** |
| 通俗类比 | 新人跟全屋子的人挨个握手 | 新人只跟记录员对个眼神 |
| 代价 | 慢但每个位置的信息都精确保留 | 快但信息有压缩损失（状态是近似摘要） |

这就是为什么 Qwen3.5 采用 **75% GDN + 25% Full Attention** 的混合方案——大多数层用 GDN 做快速递推（O(n)），少数层用 Softmax 保证复杂推理精度（O(n²)），两者互补。

总结 Qwen3.5 的注意力优化策略：

| 策略 | 占比 | 计算复杂度 | 作用 |
|------|------|-----------|------|
| **Gated DeltaNet**（线性注意力） | 75% 层 | O(n) | 长上下文加速，KV Cache 更小 |
| **Full Attention**（标准 Softmax） | 25% 层（每隔 4 层）| O(n²) | 保证复杂推理不降级 |
| **GQA**（分组查询） | 全部层叠加 | - | KV Cache 缩小 4 倍（独立于混合策略） |

**最终效果**：长文本推理速度提升最高 19 倍，短文本质量不减，KV Cache 占用降至 1/4。

##### GQA（分组查询注意力）：省显存的另一关键

> **先通俗理解 KV Cache**：大模型生成文本时，每生成一个新 token，都要重新计算所有已生成 token 的 Key 和 Value 向量。但同一句话里前面的 token 每次算出来都一样，反复重算就是浪费。所以 KV Cache 的做法是——**第一次算完后缓存起来，后面的 token 直接复用，不用再算**。相当于写会议纪要时，前面记下的内容不用每写一句就重新听一遍录音。

除了 GDN，Qwen3.5 还使用了 **GQA（Grouped Query Attention）**——一个同样重要的显存优化手段。

先看标准多头注意力的 KV Cache 问题：

```text
标准多头注意力（MHA）：
  Query: 16 个头 × 每个头独立的 Q
  Key:   16 个头 × 每个头独立的 K  → 全部存 KV Cache
  Value: 16 个头 × 每个头独立的 V  → 全部存 KV Cache
  → 16 套 K、V → KV Cache 很大

GQA（分组查询注意力）：
  Query: 16 个头（每个头独立 Q，不省）
  Key:   4 组 × 每组合 4 个头的 Key  → 共享，只存 4 套
  Value: 4 组 × 每组合 4 个头的 Value → 共享，只存 4 套
  → KV Cache 缩小到原来的 1/4！
```

**通俗理解**：
- **MHA（标准）**：16 个学生各记各的笔记 → 笔记占了 16 份空间
- **MQA（单 KV）**：16 个学生共享同一份笔记 → 省空间但灵活度差
- **GQA（分组）**：16 个学生分成 4 组，每组一份笔记 → **空间省了 4 倍，灵活度还在**

```
**大语言模型（Large Language Model, LLM）** 是人工智能领域中最具影响力的技术之一，它指的是由**深度学习**（特别是 Transformer 架构）训练出来的、能够理解人类语言、进行逻辑推理、生成文本、进行代码编写等任务的智能模型。

简单来说，LLM 就是“会说话”的超级智能体。

以下是对大语言模型的核心解析：

### 1. 核心原理：Transformer 架构
LLM 之所以强大，是因为它采用了**Transformer 模型**。这是一种基于深度学习的架构，其核心思想是：
*   **注意力机制（Attention Mechanism）**：模型会“关注”输入文本中的每一个词，并根据上下文决定当前词应该被强调或忽略。这 使得模型能够理解句子之间的逻辑关系，而不仅仅是简单的字符匹配。
*   **自回归（Self-Consistency）**：模型从“空”开始，逐词生成句子，通过训练让模型学会如何根据前一个词来预测下一个词，从而形成连贯的文本。

### 2. 主要功能
大语言模型具备以下能力：
*   **文本生成与理解**：可以回答各种问题、创作小说、写邮件、写代码，并能理解用户的意图。
*   **多语言支持**：能够处理中文、英文、法语、西班牙语等多种语言。
*   **逻辑推理**：在数学、科学、编程等逻辑领域表现优异。
*   **代码生成**：能够编写功能完整的程序。
*   **对话与角色扮演**：可以扮演医生、律师、程序员等特定角色进行对话。

LLM 不是凭空产生的，而是通过**大规模数据**（如海量文本、代码、图片等）进行**监督学习**（Supervised Learning）训练的。   
*   模型需要被训练成“听话”的机器，它必须学会如何根据输入输出符合人类语言习惯的回复。
*   训练数据通常来自互联网上的公开文本，经过清洗和标注。
```

技术原理：`apply_chat_template` 本质上是一个**模板字符串替换**——它把 messages 列表代入 tokenizer 的 `chat_template` 属性里定义的 Jinja2 模板。你可以直接查看：

```
{% for message in messages %}{{'<|im_start|>' + message['role'] + '\n' + message['content'] + '<|im_end|>\n'}}{% endfor %}{% if add_generation_prompt %}<|im_start|>assistant\n{% endif %}
```

这就是为什么：
- **基座模型 (GPT-2)** → 直接 `tokenizer(prompt)` → 简单
- **对话模型 (Qwen)** → `apply_chat_template(messages) + tokenizer(text)` → 多一步但必要

### DeepSeek-V4：开源旗舰模型（拓展了解）

DeepSeek 是 2024-2026 年开源社区最大的黑马，以 MIT 协议开源全系列模型，API 价格极低，编程能力全球顶尖，是目前生产落地性价比最高的选择之一。

**DeepSeek-V4 系列一览**：

| 模型 | 总参数量 | 激活参数 | 架构 | 核心亮点 |
|------|---------|---------|------|---------|
| **V4-Flash 🔓** | 284B | 13B | MoE | API 仅 $0.14/1M tokens，三种思考模式 |
| **V4-Pro 🔓** | 1.6T | 49B | MoE | SWE-bench 80.6%，1M 上下文，编程能力 #1 |

**核心架构技术**：

| 技术 | 作用 | 带来的效果 |
|------|------|-----------|
| **MoE（混合专家）** | 每层放多个"专家"子网络，每个 token 只激活部分专家（V4-Flash 总284B/激活13B） | 密集模型同等质量下，推理算力只需约 1/10，显存需求也大幅降低 |
| **混合注意力（CSA + HCA）** | 压缩稀疏注意力（CSA）把每 m 个 token 压缩成一条再稀疏选择；重度压缩注意力（HCA）以更高压缩比做全局稠密融合；再加上滑动窗口分支保留局部依赖 | 1M 上下文下，V4-Pro 推理 FLOPs 仅 V3.2 的 27%，KV Cache 仅 7%-10%；V4-Flash 更甚，FLOPs 降至 10% |

通俗理解为啥省了这么多：

- **MoE（专家分工）** 好比一个公司里不全是全才，而是按需叫不同领域的专家。V4-Flash 名义上有 284B 人（总参数），但每次干活只叫 13B 个最对口的专家——其他人在休息，所以不花那么多钱。
- **CSA（先压缩、再稀疏）** 好比看一篇百万字小说，你不逐字读。先把每 4 页折成一叠（压缩），再从所有折好的叠里挑最相关的 512 叠细看（稀疏选择）。剩下的大多数内容只留个模糊印象就行。外加一个"滑动窗口"兜底，保证刚读过的几页细节不丢。
- **HCA（更狠的压缩）** 面对超长上下文时，不用"折成叠"这么细，直接每 128 页捆成一捆，扫一眼每捆的大标题就行——因为压得足够少，所以不用稀疏挑选，全看一遍也不贵。

两个注意力层互补：**CSA 细看关键处，HCA 扫大局，滑动窗口保临近细节。** 三者配合，1M token 上下文才能跑得动。

**核心亮点**：

- **MIT 协议**：完全开源，商用无限制
- **极低 API 价格**：Flash 版 $0.14/1M input，约 ¥1/百万 tokens
- **三种思考模式**：Non-Think（快速响应）/ Think High（深度推理）/ Think Max（极限思考）
- **编程能力顶尖**：Codeforces Rating 3206（全球 #1），SWE-bench Verified 80.6%

## 3.3 中文 Tokenizer 切词特点

中文与英文在分词（Tokenization）上有本质区别，这直接影响模型的中文处理效率和成本。

> **2026 年现状**：现代大模型（Qwen3.5、DeepSeek-V4、GLM-4.7 等）已**全部统一使用 Byte-level BPE（BBPE）**，从 UTF-8 字节开始构建词表，中英文不再区分不同算法。词表规模从 BERT 时代的 2 万扩大到 **24-30 万**，[UNK] 标记在字节级分词下基本消失。

### 中英文分词对比（现代大模型）

| 特点 | 英文 | 中文 |
|------|------|------|
| **分词算法** | BBPE（字节级 BPE） | BBPE（字节级 BPE），与英文完全相同 |
| **词表规模** | 24-30 万 tokens | 24-30 万 tokens（共享同一词表） |
| **最小单位** | 字节（UTF-8），无 OOV | 字节（UTF-8），无 OOV |
| **中文压缩率** | — | 约 1.5-2 tokens/字（Qwen3.5 约 1.6） |
| **生成式 [UNK]** | 不存在 | 不存在（字节级保证任何文本都可编码） |

【注解】**Byte-level BPE（BBPE）** 与普通 BPE 的核心区别：普通 BPE 以字符（character）为最细粒度，遇到词表中没有的字符就会变成 [UNK]；BBPE 以 **UTF-8 字节**为最细粒度，任何字符都能被拆成 1-4 个字节，**不存在"词表中没有"的情况**。代价是序列长度略增——比如"🐶"表情被拆成 4 个字节 tokens。

**BBPE 词表是怎么来的？它不是把每个 UTF-8 字节单独当做一个 token，而是从字节出发，"合并"出常用词。** 过程像搭积木：

1. **起点**：先把所有文本拆成最细的 UTF-8 字节（共 256 种），这是词表的"地基"
2. **统计**：看看哪些字节对（或字节序列）出现频率最高，比如"天"的两个字节经常挨在一起
3. **合并**：把频率最高的字节对合并成一个新 token，加入词表
4. **重复**：继续统计、合并——"天气"的 token 经常挨着"今天"，于是把它们再合并

最终词表里既有单字节 token（处理生僻字符），也有多字节组合 token（覆盖常用字和词）。这就是为什么 Qwen 的 tokenizer 能把"天安门"识别成一个 token——因为在训练数据中这三个字经常一起出现，BPE 算法自动把它们合并了。

【注解】**中文压缩率** 是大模型中文能力的核心指标之一。它表示"一个中文字平均消耗几个 token"。压缩率越低，同样长度的中文文本花费的 token 越少，API 成本越低、推理速度越快。Qwen3.5 的中文压缩率约 1.6 tokens/字，显著优于 GPT-5.5 的 2-3 tokens/字。

```python
}

for name, model_name in models.items():
    print(f"\n{'='*50}")
    print(f"模型: {name}")
    print(f"{'='*50}")
    
    try:
        tokenizer = AutoTokenizer.from_pretrained(model_name)
        
        # 分词
        tokens = tokenizer.tokenize(text)
        input_ids = tokenizer.encode(text)
        
        print(f"原文: {text}")
        print(f"Token 数: {len(tokens)}")
        print(f"Tokens: {tokens[:15]}...")  # 只显示前15个

        print(f"词表大小: {tokenizer.vocab_size}")

        if "Qwen" in name:

            for i, tid in enumerate(input_ids):
```

### 中文分词的挑战

3. **中英混合**："这个API设计得很优雅" → "API"应该作为一个 token 还是拆成字母？
4. **数字和日期**："2024年5月20日"如何切分最合理？

【注解】现代大模型通常使用 **BPE（Byte-Pair Encoding）** 或 **SentencePiece** 算法构建词表，通过统计频率自动学习最优的子词划分，减少人工规则依赖。
